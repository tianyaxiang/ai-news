**The Shift from Capability to Token Efficiency**
**从能力导向转向 Token 效率导向**

We have officially moved past the AI prototyping phase. Building on the concepts in *Escaping the Prototype Mirage* [1], product and engineering teams across every industry are now shipping agentic applications that solve workflows previously dominated by manual grind. Building these autonomous agent prototypes is now a breeze. It is as simple as using key concepts like recursive Agentic Loops (Observe-Think-Act) for execution, setting up headless gateways to connect agents via chat apps, and relying on stored state that persists across reboots (as explained in [1]).
我们已经正式告别了 AI 原型开发阶段。基于《逃离原型幻象》（Escaping the Prototype Mirage）[1] 中的概念，各行各业的产品和工程团队现在正纷纷推出智能体应用，以解决以往依赖人工处理的繁琐工作流。构建这些自主智能体原型现在变得轻而易举。只需使用递归智能体循环（观察-思考-行动）进行执行，设置无头网关通过聊天应用连接智能体，并依赖跨重启持久化的存储状态（如 [1] 中所述），即可轻松实现。

But graduating them to reliable products is another story. The new frontier isn’t proving agents can work, it’s proving they can work profitably. At the same time, internal metrics at enterprises like “token maxing” (unconstrained token use to achieve best results) that were appropriate for the prototyping stage are shifting to measuring the “value-to-token-spent” ratio as agentic products scale. After all, most products need to be profitable and maximize margin as they are moving from leveraging cheap traditional compute (TradCompute) to solve user problems toward using AI intelligence for the same.
但要将它们升级为可靠的产品则是另一回事。新的前沿阵地不再是证明智能体“能工作”，而是证明它们能“盈利”。与此同时，企业内部的指标正在发生变化：在原型阶段适用的“Token 最大化”（即为了获得最佳结果而不受限制地使用 Token）策略，随着智能体产品的规模化，正转向衡量“价值与 Token 消耗比”。毕竟，大多数产品在从利用廉价的传统计算（TradCompute）转向利用 AI 智能来解决用户问题时，都需要实现盈利并最大化利润空间。

But models need reasoning freedom and recent studies have shown that exploratory agentic workflows outperform fixed paths, opening new paths, creating MCP tools, and building infrastructure to solve the problem more efficiently in most cases. This brings the question of balancing the model’s need for agency with the economic reality of inference costs.
然而，模型需要推理的自由度。最近的研究表明，探索性的智能体工作流在大多数情况下优于固定路径，它们能开辟新路径、创建 MCP 工具并构建基础设施，从而更高效地解决问题。这就引出了一个问题：如何在模型对自主性的需求与推理成本的经济现实之间取得平衡。

**Why Constrained Agents Fail to Converge**
**为什么受限的智能体难以收敛**

Agent harnesses store your task context and objectives in markdown (*.md) files, which don’t typically represent tight workflows, but rather outline the intent or the objective you want to accomplish.
智能体框架（Harnesses）通常将任务上下文和目标存储在 Markdown (*.md) 文件中，这些文件通常并不代表严密的工作流，而是勾勒出你想要达成的意图或目标。

**The Paradox of Objective Failure:** In studies on agents solving complex problems, researchers found that providing strict, highly-constrained guidelines where each of the agent’s action takes it closer to the goal, leads to getting stuck in a local optima and suffering an objective failure. An example from Professor Jeff Clune’s research on open-ended agent learning illustrates this perfectly: an agent in a maze, when constantly rewarded solely for seeking the direct path to the exit, will repeatedly bang into walls and get trapped in a local optimum, never reaching the end [2].
**目标失败的悖论：** 在关于智能体解决复杂问题的研究中，研究人员发现，提供严格且高度受限的指导方针（即智能体的每一个动作都必须使其更接近目标）会导致智能体陷入局部最优解，从而导致目标失败。Jeff Clune 教授关于开放式智能体学习的研究提供了一个完美的例子：一个在迷宫中的智能体，如果仅仅因为寻找通往出口的直接路径而不断获得奖励，它会反复撞墙并陷入局部最优，永远无法到达终点 [2]。

**The Power of Unconstrained Harnesses:** Contemporary agent harnesses like Google Antigravity and Anthropic’s Claude Code have been so effective because they allow agents to create, orchestrate, execute complex tasks, and even create their own tools without strict human micro-management. They succeed because they are given the freedom to explore circuitous paths. Consider an edge case in a routine medical intake workflow: if we rigidly constrain a healthcare agent to purely follow a predefined scheduling flow, it breaks in the real world. If a patient mentions chest pain midway through that routine intake, the agent’s Agentic Loop must have the autonomy to instantly recognize the urgency, abandon the scheduling flow, and trigger a safety escalation. It should utilize what we previously defined as a `No-Reply Token` to suppress booking chatter and route the context directly to a human nurse [1]. Rigidly constrained prototypes fail this test spectacularly because they cannot adapt to critical, out-of-bounds context.
**非受限框架的力量：** 像 Google Antigravity 和 Anthropic 的 Claude Code 这样的现代智能体框架之所以如此有效，是因为它们允许智能体在没有严格的人工微观管理的情况下，创建、编排、执行复杂任务，甚至创建自己的工具。它们之所以成功，是因为它们被赋予了探索迂回路径的自由。以常规医疗问诊流程中的一个边缘情况为例：如果我们严格限制医疗智能体仅遵循预定义的预约流程，它在现实世界中就会失效。如果患者在常规问诊过程中提到胸痛，智能体的循环必须具备自主性，能够立即识别紧急情况，放弃预约流程，并触发安全升级。它应该利用我们之前定义的“无回复 Token”（No-Reply Token）来抑制预约对话，并将上下文直接路由给人类护士 [1]。严格受限的原型在这一测试中会惨败，因为它们无法适应关键的、超出预设范围的上下文。

**Infinite Goal Searching is Expensive**
**无限的目标搜索代价高昂**

While providing agency is essential to discover a solution initially, running a full open-ended search for every user workflow request can lead to massive and unsustainable token consumption. At this stage the agent has found a valid path and this approach is inherently allowing it to re-explore or “hallucinate” the workflow structure. While this can be self correcting, such subsequent runs of a similar request destroy enterprise token economics. For example, routing medical intake workflows and even the edge cases that require an escalation can be learnt over a period of time. A clinic or a solution provider’s workflows will graduate to deterministic paths for the most part, leaving some autonomy reserved purely for rare outliers and complex edge cases.
虽然提供自主性对于最初发现解决方案至关重要，但对每个用户工作流请求都进行完全的开放式搜索会导致巨大且不可持续的 Token 消耗。在这一阶段，智能体已经找到了有效的路径，而这种方法本质上允许它重新探索或“幻觉”工作流结构。虽然这可以自我纠正，但对于类似请求的后续运行会破坏企业的 Token 经济性。例如，医疗问诊工作流的路由，甚至是需要升级处理的边缘情况，都可以随着时间的推移被学习。诊所或解决方案提供商的工作流在很大程度上会演变为确定性路径，仅将部分自主权保留给罕见的异常情况和复杂的边缘案例。

**Architectural Solutions Through Early Commitment and Deterministic Replay**
**通过早期承诺与确定性重放实现架构解决方案**

Early Commitment has shown promise in structured problem solving and it can be applied to agentic workflows as well [3]. It involves classifying the problem first, say by structuring the system prompt to require the model to output a specific classification tag. By forcing an agent to classify the problem type and establish constraints before it generates the execution logic, you prevent the agent from hallucinating or exploring dead-end paths. This cuts out noise and focuses the agent purely on execution rather than continuous exploration.
“早期承诺”（Early Commitment）在结构化问题解决中展现出了前景，它同样可以应用于智能体工作流 [3]。它涉及首先对问题进行分类，例如通过构建系统提示词，要求模型输出特定的分类标签。通过强制智能体在生成执行逻辑之前先对问题类型进行分类并建立约束，可以防止智能体产生幻觉或探索死胡同。这消除了噪音，使智能体专注于执行而非持续的探索。

For instance, in a telehealth triage workflow, we can enforce Early Commitment by requiring the agent to definitively classify the encounter as a “routine prescription refill” before taking any action. Once committed to this specific constraint, the agent restricts its tool calls strictly to the pharmacy database, completely bypassing the expensive, open-ended diagnostic reasoning paths it might otherwise wander down trying to diagnose a patient.
