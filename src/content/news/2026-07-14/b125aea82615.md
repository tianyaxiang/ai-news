# Agentic RAG: Let the Agent Search
# Agentic RAG：让智能体去搜索

For many of us, the first LLM application we build is a RAG app. The recipe is simple: chunk, embed, retrieve, then answer. It looks clean on paper. But once you use it on real cases, things get messy very quickly: similarity search finds similar wordings but not necessarily useful chunks, the right evidence never shows up in the retrieved context as it ranks too low, or important context may be split across chunk boundaries. With insufficient context, the LLM has little room to recover.
对于我们许多人来说，构建的第一个大模型（LLM）应用就是 RAG（检索增强生成）。其配方很简单：分块、嵌入、检索，然后回答。这在理论上看起来很完美。但一旦应用到实际案例中，情况很快就会变得一团糟：相似度搜索找到的往往是措辞相似但未必有用的片段；正确的证据因为排名太低而从未出现在检索到的上下文中；或者重要的上下文被分割在不同的块边界之间。在上下文不足的情况下，LLM 几乎没有纠错的空间。

So, how about we make retrieval iterative? What if the model can search, read, decide whether it has enough evidence, and search again when needed? Probably we don’t even need the vector embeddings in the first place. That’s the premise of agentic RAG. In this post, we’ll build a mini agentic RAG workflow with the OpenAI Agents SDK. We’ll examine how the agent iteratively searches, reads, and grounds its answer. At the end, we’ll take a step back and briefly discuss the considerations for building a practical agentic RAG solution.
那么，如果我们让检索过程变得可迭代呢？如果模型能够搜索、阅读、判断证据是否充足，并在需要时再次搜索会怎样？或许我们根本不需要向量嵌入。这就是 Agentic RAG（智能体 RAG）的前提。在这篇文章中，我们将使用 OpenAI Agents SDK 构建一个微型 Agentic RAG 工作流。我们将探讨智能体如何迭代地搜索、阅读并基于证据给出回答。最后，我们将退一步，简要讨论构建实用 Agentic RAG 解决方案的注意事项。

### 1. Case Study: Answering a Policy Question with Agentic RAG
### 1. 案例研究：使用 Agentic RAG 回答政策问题

For our case study, we’ll build a policy RAG agent over a company policy document collection.
在我们的案例研究中，我们将针对公司政策文档集构建一个政策 RAG 智能体。

#### 1.1 Curating The Document Collection
#### 1.1 整理文档集

Here, I created six synthetic company policy docs. They are all markdown files. Each one has a title, an effective date, a short summary, and the policy text. To be realistic, those docs cover 6 common company policy areas:
在这里，我创建了六份模拟的公司政策文档。它们都是 Markdown 文件，每份文档都包含标题、生效日期、简短摘要和政策正文。为了贴近现实，这些文档涵盖了 6 个常见的公司政策领域：

*   `approval_matrix.md`: containing approval levels for common business travel decisions, effective on July 1, 2025.
*   `approval_matrix.md`：包含常见商务差旅决策的审批级别，2025 年 7 月 1 日生效。
*   `conference_guidelines.md`: containing rules for attending external events, effective on May 15, 2025.
*   `conference_guidelines.md`：包含参加外部活动的规则，2025 年 5 月 15 日生效。
*   `faq.md`: containing informal answers to common travel questions, effective on September 1, 2025.
*   `faq.md`：包含常见差旅问题的非正式解答，2025 年 9 月 1 日生效。
*   `policy_updates_2026.md`: containing updates to lodging, conference travel, and approval timing for 2026, effective on January 1, 2026.
*   `policy_updates_2026.md`：包含 2026 年关于住宿、会议差旅和审批时效的更新，2026 年 1 月 1 日生效。
*   `remote_work_policy.md`: containing rules for remote work, effective on February 1, 2026.
*   `remote_work_policy.md`：包含远程办公规则，2026 年 2 月 1 日生效。
*   `travel_policy.md`: containing standard travel booking rules for flights, lodging, meals, and transportation, effective on March 1, 2025.
*   `travel_policy.md`：包含机票、住宿、餐饮和交通的标准差旅预订规则，2025 年 3 月 1 日生效。

We made it intentional that the answer to a policy question may not live in one document. This allows us to see the desired agentic behavior.
我们特意设计了政策问题的答案可能不会存在于单一文档中的情况。这使我们能够观察到预期的智能体行为。

#### 1.2 Defining The Agent
#### 1.2 定义智能体

Next, we configure the agent. For that, we use the OpenAI Agents SDK. At a high level, the agent is just this:
接下来，我们配置智能体。为此，我们使用 OpenAI Agents SDK。从宏观上看，智能体仅仅是这样：

```python
# pip install openai-agents
from agents import Agent

agent = Agent(
    name="Policy research assistant",
    instructions=INSTRUCTIONS,
    model="gpt-5.4",
    tools=[list_docs, search_docs, read_doc],
)
```

Two parts we need to go through: the agent instruction, and the tools it has access to. First, the instruction. This is where we define the desired search behavior:
我们需要了解两个部分：智能体指令及其可访问的工具。首先是指令，我们在其中定义预期的搜索行为：

```python
INSTRUCTIONS = """
[Role] You are a careful internal policy research assistant.
[Research behavior] Answer employee policy questions using the document tools. Find enough relevant evidence to support the answer. Keep conclusions grounded in the policy documents.
[Expected output] Give a direct answer first. Then briefly explain the evidence. Cite the document filenames used for each important claim.
""".strip()
```

For this case study, we require that the agent can only touch the docs via three pre-defined tools.
在本案例研究中，我们要求智能体只能通过三个预定义的工具来访问文档。

#### 1.3 Running One Policy Question
#### 1.3 运行一个政策问题

Now we test the agent with one concrete question: “I am attending a conference in Berlin. The conference organizer lists an official hotel, but the nightly rate is above the normal hotel cap. Can I book that hotel, and what approval do I need before booking?“
现在我们用一个具体问题来测试智能体：“我正在参加柏林的一个会议。会议组织者列出了一家官方酒店，但每晚价格超过了正常的酒店限额。我可以预订那家酒店吗？在预订前我需要什么审批？”

The agent produced the right answer: yes, the employee can book the official conference hotel if there is a practical business reason. It got that information from `conference_guidelines.md`. For the approval part, the agent first identified that approval is needed as the hotel is above the normal cap. Then it gave the corresponding approval conditions. The agent used `travel_policy.md`, `approval_matrix.md`, and `policy_updates_2026.md` to support its answer.
智能体给出了正确的答案：是的，如果有实际的业务理由，员工可以预订官方会议酒店。它从 `conference_guidelines.md` 中获取了该信息。关于审批部分，智能体首先确认由于酒店价格超过了正常限额，因此需要审批。然后，它给出了相应的审批条件。智能体使用了 `travel_policy.md`、`approval_matrix.md` 和 `policy_updates_2026.md` 来支持其回答。
