# Context Rot: Why Claude Code Sessions Decay, and How to Govern Them
# 上下文腐烂：Claude Code 会话为何会衰减，以及如何治理它们

**Sessions rot quietly, long before the token limit. The fix isn't more context, it's governed context.**
**会话在达到 Token 限制之前就会悄然腐烂。解决之道不在于增加上下文，而在于治理上下文。**

Jake Minns | Jul 13, 2026 | 20 min read
Jake Minns | 2026年7月13日 | 20分钟阅读

The context window is a core feature of every frontier model. Measured in tokens, it is typically made up of the system prompt and a growing history of prompts, responses and tool calls stretching back as far as the limit allows. For every token the model generates, it first looks back across the entire context window. This is the only mechanism by which it has memory of a chat session. The model holds no persistent internal state from one turn to the next. And as we will see, the context window is not passive storage. Everything in it has the potential to shape the model’s output, for better or worse.
上下文窗口是每个前沿模型的核心功能。它以 Token 为单位进行衡量，通常由系统提示词以及不断增长的提示词、回复和工具调用历史组成，这些内容会一直延伸到限制允许的范围。模型每生成一个 Token，都会先回顾整个上下文窗口。这是它对聊天会话拥有记忆的唯一机制。模型在不同轮次之间不会保留持久的内部状态。正如我们将要看到的，上下文窗口并非被动的存储空间。其中的每一项内容都有可能影响模型的输出，无论好坏。

This is a common experience: a session that starts strong and slowly loses the thread. The gradual decline in a model’s output quality driven by the contents of the context window is commonly referred to as context rot. We can separate the sources of context rot into two categories. The first is intrinsic rot: a property of how a model applies attention across the context window as it generates output. We’ll go into more detail below, but for now the key takeaway is this: however perfectly relevant the context is, the model’s ability to recall and distinguish information is imperfect.
这是一种常见的体验：一个会话开始时表现出色，但慢慢地失去了重点。由上下文窗口内容驱动的模型输出质量逐渐下降，通常被称为“上下文腐烂”（context rot）。我们可以将上下文腐烂的来源分为两类。第一类是内在腐烂（intrinsic rot）：这是模型在生成输出时如何跨上下文窗口应用注意力的一种属性。我们将在下文详细探讨，但目前的重点是：无论上下文的相关性多么完美，模型回忆和区分信息的能力总是不完美的。

The second category we call content rot. This is the accumulation of stale, wrong and contradictory information across a session. The failed approach the model keeps going back to, or the dozen tangential tool calls that went nowhere. All of it repeatedly reprocessed, slowly shaping the outcome of the session. Fortunately, unlike intrinsic rot, content rot is ours to manage – where learning to control it transforms tools like Claude Code from occasionally frustrating to consistently sharp.
第二类我们称之为内容腐烂（content rot）。这是在整个会话过程中积累的陈旧、错误和矛盾的信息。比如模型不断重复尝试的失败方法，或者十几个毫无进展的旁支工具调用。所有这些内容被反复处理，缓慢地塑造着会话的结果。幸运的是，与内在腐烂不同，内容腐烂是可以由我们管理的——学会控制它，就能让 Claude Code 这类工具从偶尔令人沮丧变得始终高效敏锐。

### Intrinsic rot: the model floor
### 内在腐烂：模型的基准线

Intrinsic rot refers to the performance limitations inherent to the model’s architecture. It is not ours to prompt around or improve; it is the floor everything else rests on. But the mechanism is worth understanding for two reasons: it explains the motivation behind some of the practices described later in this post, and it helps dissolve the illusion that there is a little engineer back there quietly accumulating an understanding of your project.
内在腐烂是指模型架构固有的性能局限性。我们无法通过提示词来绕过或改善它；它是其他一切表现的基准线。但了解其机制有两个原因：它解释了本文后续描述的一些实践背后的动机，并有助于消除一种错觉，即认为模型背后有一个小工程师在默默地积累对你项目的理解。

Each turn, your entire session (prompts, responses, file reads, tool outputs) is flattened into one long sequence of tokens and passed through the model. LLM architecture is complex, but attention heads are the component that matter most for context. Each head answers a single question: how much should each token draw on every token before it? A relevant token contributes heavily to the next prediction; an irrelevant one contributes little, but never nothing. That “never nothing” is the crux.
每一轮对话，你的整个会话（提示词、回复、文件读取、工具输出）都会被扁平化为一个长 Token 序列，并输入模型。大语言模型架构很复杂，但注意力头（attention heads）是影响上下文最关键的组件。每个注意力头回答一个问题：每个 Token 应该在多大程度上参考它之前的每一个 Token？相关的 Token 对下一次预测贡献很大；不相关的 Token 贡献很小，但绝非零贡献。这个“绝非零贡献”就是问题的关键。

Each head’s relevance scores pass through softmax, a function that forces scores to sum to one. This means there is a fixed budget of attention. Because softmax is built on exponentials, no token’s share of the budget can ever reach exactly zero. Every token in your window, however irrelevant, competes for the same fixed budget. This is no implementation accident; softmax is central to attention precisely because it turns competing scores into a smooth, learnable distribution. But, that useful property reflects a tradeoff: irrelevant context is never free.
每个注意力头的相关性分数都会经过 Softmax 函数处理，该函数强制要求所有分数之和为 1。这意味着注意力存在一个固定的预算。由于 Softmax 基于指数运算，没有任何 Token 的预算份额能真正达到零。你窗口中的每一个 Token，无论多么不相关，都在争夺同一个固定预算。这不是实现上的偶然；Softmax 之所以是注意力的核心，正是因为它将竞争性的分数转化为平滑、可学习的分布。但这一有用的特性反映了一个权衡：不相关的上下文永远是有代价的。

The consequence of this is subtle. As context grows, the model may still rank the right tokens highest; the ordering survives. What erodes is the margin: the gap between the token that matters and the diffuse mass of everything else. Even when the model “knows” where to look, the read arrives blurred, mixed with thousands of faint contributions from the surrounding窗口. Think of it as dilution: a fixed budget spread ever thinner, worsening signal-to-noise. The stated context limit is not a performance cliff; the erosion is gradual, and it starts early.
其后果是微妙的。随着上下文的增长，模型可能仍然能将正确的 Token 排在最高位；排序依然有效。被侵蚀的是“边际”：即关键 Token 与其他所有分散信息之间的差距。即使模型“知道”该看哪里，读取的内容也会变得模糊，混杂着来自周围窗口的数千个微弱贡献。可以将其视为稀释：固定的预算被分摊得越来越薄，导致信噪比恶化。标称的上下文限制并非性能断崖；这种侵蚀是渐进的，而且很早就开始了。

Position matters too. Word order is fundamental to language, so models encode where tokens sit. The exact mechanism matters less than the measured consequence: place a fact at varying positions along a long context and retrieval accuracy traces a U, highest at the very start and very end, lowest in the middle. What matters for us is the dip: in an hours-long session, the broad middle is where most of your real content lives, and it is not safe storage.
位置也很重要。词序是语言的基础，因此模型会编码 Token 的位置。确切的机制不如测量出的后果重要：将一个事实放在长上下文的不同位置，检索准确率会呈现“U”型曲线，在开头和结尾最高，在中间最低。对我们而言，重要的是那个低谷：在长达数小时的会话中，中间的大部分区域正是你真正内容所在的地方，而那里并非安全的存储区。
