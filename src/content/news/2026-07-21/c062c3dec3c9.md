# I built a test lab to measure SSG vs SSR vs ISR on real WordPress, here's what I found
# 我搭建了一个测试实验室，在真实的 WordPress 环境中对比 SSG、SSR 和 ISR，以下是我的发现

Most "SSG vs SSR vs ISR" content out there is written from documentation. Someone reads the framework, restates it, and you're left inferring the actual difference in performance and behavior. So I built a lab where you can just run the commands and see it yourself, no table to trust blindly.
市面上大多数关于“SSG vs SSR vs ISR”的内容都是基于文档编写的。有人阅读了框架文档，复述了一遍，而你只能通过推断来理解它们在性能和行为上的实际差异。因此，我搭建了一个实验室，你可以直接运行命令亲自观察，无需盲目相信任何对比表格。

astro-wp-seo-lab builds the same WordPress content four different ways with Astro 7.1.1, then serves all four side by side so you can compare them directly.
`astro-wp-seo-lab` 使用 Astro 7.1.1 以四种不同的方式构建相同的 WordPress 内容，并将它们并排部署，以便你可以直接进行对比。

```bash
git clone https://github.com/nimajafari/astro-wp-seo-lab
npm install
npm run compare
```

That builds each arm into its own directory and serves them all at once.
这会将每个分支构建到各自的目录中，并同时运行它们。

| arm | url | what it is |
| :--- | :--- | :--- |
| ssg-full | http://localhost:4301 | everything prerendered at build time |
| ssr | http://localhost:4302 | rendered per request, no caching |
| ssr-cdn | http://localhost:4303 | per request plus CDN cache headers |
| route-cache | http://localhost:4304 | per request plus Astro 7 route caching |
| islands | http://localhost:4305 | static shell with deferred fragments |

| 分支 (arm) | URL | 说明 |
| :--- | :--- | :--- |
| ssg-full | http://localhost:4301 | 所有内容在构建时预渲染 |
| ssr | http://localhost:4302 | 每次请求时渲染，无缓存 |
| ssr-cdn | http://localhost:4303 | 每次请求渲染，并带有 CDN 缓存头 |
| route-cache | http://localhost:4304 | 每次请求渲染，并使用 Astro 7 路由缓存 |
| islands | http://localhost:4305 | 带有延迟片段的静态外壳 |

Every page has a black bar at the top showing which arm rendered it and when. That timestamp is the instrument for most of what follows. First build takes a few minutes since each arm fetches from WordPress, later builds are faster because the Content Layer loader caches between them.
每个页面顶部都有一个黑色栏，显示渲染该页面的分支及其时间戳。这个时间戳是后续大部分测试的依据。首次构建需要几分钟，因为每个分支都需要从 WordPress 获取数据；后续构建会更快，因为 Content Layer 加载器会在它们之间进行缓存。

It ships pointed at a live WordPress install (oxyplug.com), but it works against any public WordPress site with the REST API exposed.
该项目默认指向一个真实的 WordPress 站点 (oxyplug.com)，但它适用于任何开启了 REST API 的公共 WordPress 站点。

```bash
npm run probe -- https://your-site.com --save mysite
SOURCE=mysite npm run compare
```

`probe` checks what your own install actually exposes, REST API reachability, Yoast presence, permalink structure, then saves it under a name. Use the URLs `npm run compare` prints for your own site instead of the ones below, since those are generated from your own content.
`probe` 会检查你的站点实际暴露了哪些内容、REST API 的可访问性、Yoast 插件的存在情况以及固定链接结构，然后将其保存。请使用 `npm run compare` 为你自己的站点打印出的 URL，而不是下方的 URL，因为后者是基于我自己的内容生成的。

### Build time vs request time
### 构建时间 vs 请求时间

This is the distinction most of the SSG vs SSR debate hinges on, and it takes about 30 seconds to see for yourself. Open these two side by side and reload each a few times.
这是 SSG 与 SSR 争论的核心所在，你只需 30 秒就能亲自验证。将这两个页面并排打开，并分别刷新几次。

* http://localhost:4301/optimization/crl-ocsp-certificate-revocation-methods/ (ssg-full)
* http://localhost:4302/optimization/crl-ocsp-certificate-revocation-methods/ (ssr)

The ssg-full timestamp never changes. It was baked in at build time, and every visitor gets that same file. The ssr timestamp changes on every single reload, milliseconds apart. That page is assembled from scratch for every request.
`ssg-full` 的时间戳永远不会改变。它是在构建时固化的，每个访问者获取的都是同一个文件。而 `ssr` 的时间戳在每次刷新时都会改变，相差仅几毫秒。该页面是为每次请求从零开始组装的。

```bash
for i in 1 2 3; do curl -s http://localhost:4301/... | grep -o 'RENDERED: [^<]*'; done 
# identical three times (三次结果相同)

for i in 1 2 3; do curl -s http://localhost:4302/... | grep -o 'RENDERED: [^<]*'; done 
# three different values (三个不同的值)
```

Why this matters for SEO specifically, freshness and cost trade against each other directly. The SSR page reflects a WordPress edit instantly, the SSG page doesn't until you rebuild. But SSR does that work again for every visitor, and every crawler hit too. Googlebot crawling 10,000 URLs is 10,000 renders, and each one is also a live WordPress API call.
对于 SEO 而言，这之所以重要，是因为内容的新鲜度与成本直接相关。SSR 页面可以即时反映 WordPress 的编辑内容，而 SSG 页面则必须等到重新构建后才能更新。但 SSR 会为每个访问者（包括每个爬虫）重复执行渲染工作。如果 Googlebot 抓取 10,000 个 URL，就意味着 10,000 次渲染，且每一次都是一次实时的 WordPress API 调用。

### What caching actually is
### 缓存的本质

Open route-cache next. http://localhost:4304/optimization/crl-ocsp-certificate-revocation-methods/ Reload it repeatedly. The timestamp freezes, even though this arm renders on demand exactly like ssr does. Wait past its 300 second TTL and reload again, it updates once, then freezes again. That's ISR. That's the whole mechanism. Render on demand, keep the result, serve the kept copy until something invalidates it.
接下来打开 `route-cache` (http://localhost:4304/...)。反复刷新它。你会发现时间戳被冻结了，尽管这个分支和 `ssr` 一样是按需渲染的。等待超过 300 秒的 TTL 后再次刷新，它会更新一次，然后再次冻结。这就是 ISR。这就是它的全部机制：按需渲染，保留结果，并在失效前一直提供该缓存副本。

```bash
curl -sD- -o /dev/null http://localhost:4304/category/optimization/ | grep -i x-astro-cache 
# MISS on the first request, HIT after (首次请求为 MISS，之后为 HIT)
```

Compare the actual cache instructions each arm sends.
对比每个分支发送的实际缓存指令：

```bash
curl -sD- -o /dev/null http://localhost:4302/category/optimization/ | grep -i cache # ssr, nothing
curl -sD- -o /dev/null http://localhost:4303/category/optimization/ | grep -i cache # ssr-cdn
```

`ssr-cdn` sends two different policies on purpose.
`ssr-cdn` 特意发送了两种不同的策略：

* `Cache-Control: public, max-age=0, must-revalidate`
* `CDN-Cache-Control: public, s-maxage=120, stale-while-revalidate=600`

Browsers get told not to cache, the CDN gets told to cache hard. You can purge a CDN, you can't purge a browser. Anything a browser caches stays stale until it expires and there's nothing you can do about it once it's out there, so browsers get the conservative policy and shared caches get the aggressive one.
浏览器被告知不要缓存，而 CDN 被告知进行强缓存。你可以清除 CDN 缓存，但无法清除浏览器缓存。浏览器缓存的内容在过期前会一直保持陈旧，一旦发出就无法撤回，因此浏览器采用保守策略，而共享缓存（CDN）采用激进策略。

To see a real CDN in front of this, spin up Varnish.
要查看真实 CDN 的效果，可以启动 Varnish：

```bash
docker compose -f docker/docker-compose.yml up -d # Varnish on :8081
npm run build:ssr-cdn && npm run serve
curl -sD- -o /dev/null http://localhost:8081/category/optimization/ | grep -iE 'x-cache|age'
```

Request it repeatedly over half a minute and watch Age climb in step with elapsed time. A monotonically increasing Age header is the provider neutral proof that a response came from cache, it works the same way on Varnish, Vercel, Netlify, and Cloudflare.
在半分钟内反复请求，观察 `Age` 头部随时间增长。单调递增的 `Age` 头部是响应来自缓存的通用证明，无论是在 Varnish、Vercel、Netlify 还是 Cloudflare 上，原理都是一样的。

### What a crawler actually sees
### 爬虫实际看到的内容

This is the exercise most worth doing if you only try one thing from this post. Open the islands arm in a browser. http://localhost:4305/optimization/crl-ocsp-certificate-revocation-methods/ Scroll to the bottom. There's a "Related posts" list with five links, looks completely normal. Now View Source, the actual source, Ctrl-U or Cmd-Opt-U, not devtools. Devtools shows the live DOM after JavaScript has run, View Source shows what the server actually sent.
如果你只想尝试本文中的一件事，那么这个练习最值得做。在浏览器中打开 `islands` 分支。滚动到底部，有一个包含五个链接的“相关文章”列表，看起来完全正常。现在查看源代码（Ctrl-U 或 Cmd-Opt-U，不要使用开发者工具）。开发者工具显示的是 JavaScript 运行后的实时 DOM，而“查看源代码”显示的是服务器实际发送的内容。

Search for "Related posts". It's not there. What's there instead is this:
搜索“Related posts”。它不在那里。取而代之的是：

```html
<p data-island-fallback>Loading related posts...</p>
```

```bash
curl -s http://localhost:4305/... | grep -c 'Related posts' # 0
curl -s http://localhost:4301/... | grep -c 'Related posts' # 1, same component, rendered inline
```

Same component, same data, the only difference is `server:defer`. Measure the gap properly.
相同的组件，相同的数据，唯一的区别是 `server:defer`。让我们正确地衡量一下差距：

```bash
node harness/parity-render.mjs --arm islands --base http://localhost:4305
node harness/parity-render.mjs --arm ssg-full --base http://localhost:4301
```

| arm | parity | internal links hidden |
| :--- | :--- | :--- |
| ssg-full | 100% | 0 |
| islands | 87% | 5 |

| 分支 | 完整性 (parity) | 隐藏的内部链接 |
| :--- | :--- | :--- |
| ssg-full | 100% | 0 |
| islands | 87% | 5 |
