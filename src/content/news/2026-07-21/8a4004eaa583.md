# Introducing TypeYAML: A More Expressive, Type-Safe Way to Write YAML
# 介绍 TypeYAML：一种更具表现力、类型安全的 YAML 编写方式

Today, we are releasing TypeYAML, an open-source language that extends YAML with stronger structure, reusable types, validation, and developer-focused tooling. TypeYAML is designed for developers who like YAML’s readability but need more confidence when working with large configuration files, schemas, deployment definitions, automation pipelines, application settings, and structured data.
今天，我们发布了 TypeYAML，这是一种开源语言，它通过更强的结构、可重用的类型、验证机制和以开发者为中心的工具扩展了 YAML。TypeYAML 专为那些喜欢 YAML 的可读性，但在处理大型配置文件、模式、部署定义、自动化流水线、应用程序设置和结构化数据时需要更高确定性的开发者而设计。

The project is now available through:
npm: @magnexis/typeyaml
Visual Studio Code Marketplace: magnificent-language/typeyaml
GitHub: magnexis/typeyaml-lang
TypeYAML is still evolving, but this release establishes the foundation for a language intended to make configuration safer, clearer, and easier to maintain.
该项目现已通过以下渠道提供：
npm: @magnexis/typeyaml
Visual Studio Code 市场: magnificent-language/typeyaml
GitHub: magnexis/typeyaml-lang
TypeYAML 仍在不断演进中，但此次发布为该语言奠定了基础，旨在使配置变得更安全、更清晰且更易于维护。

### Why Build Another YAML-Based Language?
### 为什么要构建另一种基于 YAML 的语言？

YAML is everywhere. It is used for: CI/CD pipelines, Docker Compose files, Kubernetes resources, GitHub Actions, Application configuration, Infrastructure definitions, API specifications, Static-site configuration, Package and project metadata.
YAML 无处不在。它被用于：CI/CD 流水线、Docker Compose 文件、Kubernetes 资源、GitHub Actions、应用程序配置、基础设施定义、API 规范、静态站点配置以及包和项目元数据。

Its popularity makes sense. YAML is compact, readable, and approachable. A configuration file can often be understood without extensive documentation. However, YAML becomes more difficult to manage as projects grow. Large YAML files can contain deeply nested structures, duplicated values, inconsistent field names, invalid data types, undocumented conventions, and subtle indentation errors.
它的流行是有道理的。YAML 简洁、易读且平易近人。通常无需详尽的文档就能理解配置文件。然而，随着项目的增长，YAML 变得越来越难以管理。大型 YAML 文件可能包含深度嵌套的结构、重复的值、不一致的字段名、无效的数据类型、未记录的约定以及细微的缩进错误。

In many cases, the file is only validated after another tool attempts to consume it. That creates a frustrating workflow: Write configuration. Run a command. Receive an unclear validation error. Search through a large YAML file. Fix the issue. Repeat. TypeYAML is an attempt to improve that experience without abandoning what makes YAML useful. The goal is not to replace YAML’s readability. The goal is to build a stronger language around it.
在许多情况下，文件只有在其他工具尝试读取它时才会被验证。这导致了一个令人沮丧的工作流程：编写配置 -> 运行命令 -> 收到不明确的验证错误 -> 在大型 YAML 文件中搜索 -> 修复问题 -> 重复。TypeYAML 旨在改善这种体验，同时不放弃 YAML 的实用性。其目标不是取代 YAML 的可读性，而是围绕它构建一种更强大的语言。

### What Is TypeYAML?
### 什么是 TypeYAML？

TypeYAML is a typed superset of YAML. It introduces a language layer for defining and validating structured configuration while keeping the syntax familiar to developers who already use YAML.
TypeYAML 是 YAML 的类型化超集。它引入了一个用于定义和验证结构化配置的语言层，同时保持了对已使用 YAML 的开发者来说熟悉的语法。

A normal YAML file might look like this:
```yaml
user:
  name: person
  age: 48
  active: true
```
一个普通的 YAML 文件可能如下所示：

This is easy to read, but the file does not independently explain its expected structure. Should age always be a number? Is name required? Can active be omitted? Are additional fields allowed? TypeYAML is designed to make those expectations explicit. A TypeYAML definition could describe the intended structure before data is used by the rest of an application:
这虽然易于阅读，但该文件无法独立解释其预期的结构。age 必须始终是数字吗？name 是必需的吗？active 可以省略吗？允许添加额外的字段吗？TypeYAML 旨在明确这些预期。TypeYAML 定义可以在数据被应用程序的其他部分使用之前描述预期的结构：

```typeyaml
type User:
  name: string
  age: number
  active: boolean
```

The exact syntax and feature set will continue to develop, but the central idea is simple: configuration should be able to describe and verify itself. Instead of treating configuration as loosely structured text, TypeYAML treats it as a language with rules, types, tooling, and predictable behavior.
确切的语法和功能集将继续发展，但核心理念很简单：配置应该能够自我描述和自我验证。TypeYAML 不再将配置视为松散结构的文本，而是将其视为一种具有规则、类型、工具和可预测行为的语言。

### The Main Goals of TypeYAML
### TypeYAML 的主要目标

TypeYAML is being developed around several core principles:
TypeYAML 的开发围绕几个核心原则展开：

1. **Preserve YAML’s Readability**: TypeYAML should remain recognizable to anyone familiar with YAML. A developer should not need to learn an entirely unfamiliar syntax simply to gain validation and stronger structure.
1. **保持 YAML 的可读性**：对于任何熟悉 YAML 的人来说，TypeYAML 应该保持可识别性。开发者不应仅仅为了获得验证和更强的结构而学习一种完全陌生的语法。

2. **Detect Errors Earlier**: Configuration problems should be caught while the file is being written, not only after deployment or execution. TypeYAML aims to support earlier feedback for issues such as incorrect value types, missing required properties, unknown fields, invalid nested structures, duplicate declarations, incorrect references, and malformed language syntax.
2. **更早地检测错误**：配置问题应该在编写文件时就被捕获，而不是在部署或执行之后。TypeYAML 旨在为诸如错误的值类型、缺失的必需属性、未知字段、无效的嵌套结构、重复声明、错误的引用以及格式错误的语法等问题提供更早的反馈。

3. **Make Large Configurations Easier to Maintain**: As configuration grows, repeated structures become difficult to keep consistent. TypeYAML is intended to support reusable definitions so developers can describe a structure once and use it throughout a project.
3. **使大型配置更易于维护**：随着配置的增长，重复的结构很难保持一致。TypeYAML 旨在支持可重用的定义，以便开发者可以定义一次结构并在整个项目中重复使用。

4. **Provide First-Class Developer Tooling**: A language is more than its parser. TypeYAML is being released with tooling intended to support a practical development workflow, including an npm package and a Visual Studio Code extension.
4. **提供一流的开发者工具**：一种语言不仅仅是它的解析器。TypeYAML 发布时附带了旨在支持实际开发工作流程的工具，包括 npm 包和 Visual Studio Code 扩展。

5. **Remain Compatible With Existing Ecosystems**: A major goal is to make it possible to author safer TypeYAML files and transform or compile them into standard YAML or other widely supported formats.
5. **保持与现有生态系统的兼容性**：一个主要目标是能够编写更安全的 TypeYAML 文件，并将其转换或编译为标准 YAML 或其他广泛支持的格式。

### Installing TypeYAML
### 安装 TypeYAML

The TypeYAML package is available on npm under the Magnexis organization. Install it globally for command-line usage:
`npm install -g @magnexis/typeyaml`
Or install it inside a project:
`npm install --save-dev @magnexis/typeyaml`
TypeYAML 包可在 npm 的 Magnexis 组织下获取。全局安装以供命令行使用：
`npm install -g @magnexis/typeyaml`
或者在项目中安装：
`npm install --save-dev @magnexis/typeyaml`

Using it as a project dependency makes it easier to keep builds reproducible because the project can define the exact TypeYAML version it expects.
将其作为项目依赖项可以更轻松地保持构建的可重复性，因为项目可以定义其预期的确切 TypeYAML 版本。

### Using TypeYAML in a Project
### 在项目中使用 TypeYAML

A project using TypeYAML may include dedicated TypeYAML source files alongside its application code. For example:
使用 TypeYAML 的项目可以在其应用程序代码旁边包含专用的 TypeYAML 源文件。例如：

```text
my-project/
├── src/
├── config/
│ ├── application.typeyaml
│ ├── database.typeyaml
│ └── deployment.typeyaml
├── package.json
└── README.md
```

TypeYAML files can act as the source of truth for structured configuration.
TypeYAML 文件可以作为结构化配置的单一事实来源。
