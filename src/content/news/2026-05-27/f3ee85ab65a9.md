# Stop Using LLMs Like Giant Problem Solvers
# 别再把大模型当成万能解题器了

I recently worked on a feature where I had to transform 100 messy compliance pdfs into structured JSON rules. The brute force approach was obvious: give the agent the source text, explain the task, provide examples, and ask it to generate the rules. Since it was the lowest-hanging fruit, I tried it first.
最近我负责开发一个功能，需要将 100 份杂乱的合规性 PDF 文档转换为结构化的 JSON 规则。最直接的“暴力”方法显而易见：将源文本交给 AI Agent，解释任务，提供示例，然后让它生成规则。由于这是最容易实现的方式，我首先尝试了这种方案。

At a glance, the output looked fine. The output JSON was valid and matched what I expected. But as I was manually sampling the results to check for accuracy, the cracks appeared. Some rules were too broad, others were missed. Some rules failed to preserve the nuances of the original text. I tried using another agent to catch and fix the errors but with such a huge corpus, it was impossible to confidently verify the output. That was the frustrating part. The errors were not obvious. This was way too fragile of an implementation to scale.
乍一看，输出结果似乎没问题。生成的 JSON 格式正确，也符合我的预期。但当我手动抽样检查准确性时，问题就暴露了：有些规则过于宽泛，有些则被遗漏了，还有些规则未能保留原文的细微差别。我尝试引入另一个 Agent 来捕捉并修复错误，但面对如此庞大的语料库，根本无法确信输出结果的可靠性。最令人沮丧的是，这些错误并不明显。这种实现方式太脆弱，无法扩展。

Though I cannot share the exact implementation details, what I can share are the architectural lessons I learnt and how I eventually implemented it. Hopefully, these insights will be useful if you’re building AI systems that need to scale, stay reliable, and deal with messy data. And if you have better ways of doing things, do reach out to chat! Okay let’s get to it.
虽然我不能分享具体的实现细节，但我可以分享我学到的架构经验以及我最终的实现方案。希望这些见解对那些正在构建需要扩展、保持可靠性并处理杂乱数据的 AI 系统的人有所帮助。如果你有更好的方法，欢迎随时交流！好了，我们开始吧。

### The problem
### 问题所在

The 100 pdfs I worked with had already been parsed and chunked before they reached me. But the raw content was still messy. There were bullet points, tables, OCR artefacts, translated sections, semi structured headings, footers, headers, inconsistent formatting and document specific quirks. I chose to use an agent because deciding what mattered required semantic judgement. The documents did not follow one consistent pattern, so relevance could not be determined through simple rules alone. You had to understand the surrounding context. None of this was difficult when done on a small chunk of data. The challenge was performing this reliably at scale. These rules were then processed by another downstream system to be evaluated deterministically.
我处理的这 100 份 PDF 在交到我手中之前已经经过了解析和分块，但原始内容依然非常杂乱。其中包含项目符号、表格、OCR 识别错误、翻译片段、半结构化标题、页脚、页眉、不一致的格式以及各种文档特有的怪癖。我选择使用 Agent，是因为判断哪些内容重要需要语义层面的判断力。这些文档没有统一的模式，因此无法仅通过简单的规则来确定相关性，你必须理解上下文。在处理少量数据时，这并不难，但挑战在于如何在大规模场景下可靠地完成任务。这些规则随后会被另一个下游系统进行确定性评估。

### What eventually worked
### 最终有效的方案

After a few experiments, I realised the biggest improvement did not come from a better prompt, a new tool, an MCP server, or a more sophisticated agent harness. It came from changing the shape of the problem. Instead of trying to make the agent smarter, I made the agent’s job smaller.
经过几次实验，我意识到最大的改进并非来自更好的提示词（Prompt）、新工具、MCP 服务器或更复杂的 Agent 框架，而是来自改变问题的形态。我没有试图让 Agent 变得更聪明，而是让 Agent 的工作变得更简单。

The first change was to prepare the source data upfront. Instead of asking the agent to query a database, retrieve records, decide whether it had the right inputs, and then perform the extraction, I gave it a more controlled starting point. In my case, that meant temporarily storing the relevant raw data locally. This may not always be practical. But the underlying principle is to reduce the amount of retrieval uncertainty the agent has to handle. If the agent’s job is to reason over content, do not also make it responsible for figuring out whether it has found the right content. Another option would be to prepare the query upfront. I also used a script to strip away unnecessary metadata and fields before passing the raw content to the agent. Less irrelevant context meant fewer distractions, fewer chances for the agent to latch onto the wrong details and a cleaner reasoning task overall.
第一个改变是预先准备好源数据。我没有让 Agent 去查询数据库、检索记录、判断输入是否正确，然后再进行提取，而是为它提供了一个更可控的起点。就我而言，这意味着将相关的原始数据临时存储在本地。这可能并不总是可行，但其核心原则是减少 Agent 需要处理的检索不确定性。如果 Agent 的任务是分析内容，就不要让它同时负责判断自己是否找到了正确的内容。另一种选择是预先准备好查询。我还编写了一个脚本，在将原始内容传给 Agent 之前，剔除了不必要的元数据和字段。更少的无关上下文意味着更少的干扰，减少了 Agent 抓取错误细节的可能性，使推理任务整体更加清晰。

But the most important change was the unit of work. Instead of processing everything at once, I did things iteratively and processed one document at a time. That made each job smaller, easier to inspect, easier to retry, and easier to audit. I spun up five subagents to process documents in parallel, with each agent logging its progress to a file. If one document failed, I could retry only that document. If one output had formatting issues, I could fix that specific case without rerunning the whole batch. If the pipeline stopped halfway, the cached progress meant it could resume from the last successful checkpoint.
但最重要的改变是工作单元。我没有一次性处理所有内容，而是采用迭代方式，一次处理一份文档。这使得每项任务更小、更易于检查、更易于重试、也更易于审计。我启动了五个子 Agent 并行处理文档，每个 Agent 都会将进度记录到文件中。如果某份文档处理失败，我只需重试该文档即可。如果某个输出存在格式问题，我只需修复该特定案例，而无需重新运行整个批次。如果流水线中途停止，缓存的进度意味着它可以从上一个成功的检查点恢复。

This was also where the separation of responsibilities became clearer. The agent handled the semantic work: understanding the content, identifying the relevant parts and writing the JSON output. The surrounding code handled the mechanical parts: parallelising jobs, enforcing the schema, generating IDs, writing files, caching progress, validating references, and checking whether the output could be traced back to the original source. I also had an orchestrator watch over the progress of the script.
这也是职责分离变得更加清晰的地方。Agent 负责语义工作：理解内容、识别相关部分并编写 JSON 输出。外围代码负责机械性工作：并行化任务、强制执行模式（Schema）、生成 ID、写入文件、缓存进度、验证引用，以及检查输出是否可以追溯到原始来源。我还设置了一个协调器来监控脚本的进度。

### Making the output auditable
### 让输出结果可审计

A useful design decision was adding reference IDs to every generated rule. This meant that each output item pointed back to a specific source. This made the output easier to audit. Instead of asking, “Does this generated rule look right?”, I could ask more precise questions such as: does the referenced source chunk exist? Is the quoted source text actually present in that chunk? I could also get another agent to selectively run audits on larger and more complex documents to ensure that important nuances were preserved.
一个有用的设计决策是为每个生成的规则添加引用 ID。这意味着每个输出项都能追溯到特定的来源，从而使输出更易于审计。我不再问“这个生成的规则看起来对吗？”，而是可以提出更精确的问题，例如：引用的源片段是否存在？引用的源文本是否真的存在于该片段中？我还可以让另一个 Agent 对更大、更复杂的文档进行选择性审计，以确保重要的细微差别得到保留。

On top of that, I did a lightweight version of evals. I ran a small batch of raw documents through the workflow and manually reviewed the results for coverage and accuracy. A full golden dataset was not practical for the scope of this task, but I still needed a way to prove to myself that the workflow was working. My goal was not to build a perfect benchmark but to make the system auditable enough that I could inspect the outputs, catch failures, and iterate toward a higher accuracy bar.
此外，我还进行了一种轻量级的评估。我将一小批原始文档通过工作流进行处理，并手动审查了结果的覆盖率和准确性。对于这项任务的规模来说，构建一个完整的“黄金数据集”并不现实，但我仍然需要一种方法来证明工作流是有效的。我的目标不是建立一个完美的基准测试，而是让系统具备足够的可审计性，以便我能够检查输出、捕捉故障，并不断迭代以提高准确率。

### My biggest takeaway
### 我最大的收获

The pattern that worked was to stop treating the LLM as the whole system. The system became more reliable not because the agent became perfect, but because the workflow made its outputs easier to trace, validate, and recover from.
最有效的模式是：停止将大模型视为整个系统。系统变得更可靠，不是因为 Agent 变得完美了，而是因为工作流使得其输出结果更易于追踪、验证和恢复。
