# What Is a Data Agent?
# 什么是数据代理（Data Agent）？

**Agentic AI: What Is a Data Agent? A simple explanation of what a data agent is and how it works**
**代理式人工智能：什么是数据代理？关于数据代理及其工作原理的简单解释**

Working at Microsoft, I have the opportunity to try new AI-powered analytical tools, including Microsoft Fabric’s data agent. That’s why I want to share what I’ve learned, explain what a data agent is, and highlight the difference between it and a “standard” AI agent. So, without further ado, here is my definition of a data agent: A data agent is a report you can talk to.
在微软工作期间，我有机会尝试各种新型人工智能分析工具，包括 Microsoft Fabric 的数据代理。因此，我想分享我的心得，解释什么是数据代理，并强调它与“标准”人工智能代理之间的区别。废话不多说，以下是我对数据代理的定义：数据代理就是一份你可以与之对话的报告。

For those of us in analytics, this means two long-held wishes might finally become a reality: #1: Analysts spend way less time building visualisations. #2: Self-service insights come closer to business users. Let me elaborate on each of these points a bit more.
对于我们从事分析工作的人来说，这意味着两个长期以来的愿望终于可能成为现实：第一，分析师将花费更少的时间制作可视化图表；第二，自助式洞察将更贴近业务用户。让我详细阐述一下这两点。

### Fewer visualisations, not fewer insights
### 更少的可视化，而非更少的洞察

I really enjoy a good report that can tell me “what’s up” with the metrics I am currently interested in. But being trained in analytics, I know how reports can sometimes cast metrics in the wrong light, leading business users to frequently ask analysts for KPIs interpretation, usually 10 minutes before important meetings. And that’s one of the reasons we often end up in a vicious cycle of having dashboards no one is using, and stakeholders constantly wanting “the number” served ad hoc or via spreadsheets.
我非常喜欢那种能告诉我当前关注指标“进展如何”的优秀报告。但作为受过分析训练的人，我知道报告有时会以错误的方式呈现指标，导致业务用户经常在重要会议前 10 分钟向分析师询问 KPI 的解读。这也是我们陷入恶性循环的原因之一：我们拥有没人使用的仪表板，而利益相关者却不断要求以临时方式或通过电子表格提供“数据”。

On the bright side, visualisations and spreadsheets are not going anywhere, but serving the insights has a new way with a Fabric data agent. Instead of wrapping queries in graphs, you can wrap them in prompts and instructions paired with the consumption-ready governed data estate in Fabric, i.e., in a lakehouse, warehouse, Power BI semantic models, KQL database, or even an ontology.
好消息是，可视化图表和电子表格不会消失，但 Fabric 数据代理为提供洞察力开辟了新途径。你无需将查询封装在图表中，而是可以将它们封装在提示词和指令中，并结合 Fabric 中可供消费的受控数据资产（例如湖仓一体、数据仓库、Power BI 语义模型、KQL 数据库，甚至是本体）。

This implies the underlying data still needs to be prepared and modelled to answer business questions such as “What was the revenue this week compared to last week?” However, from a design perspective, rather than creating a scoped visual report to answer this business question, you now create a scoped data agent to provide this, and other subsets of answers derived from the underlying data model(s).
这意味着底层数据仍需经过准备和建模，以回答诸如“本周收入与上周相比如何？”之类的业务问题。然而，从设计角度来看，你现在不再是创建一个有范围限制的可视化报告来回答这个问题，而是创建一个有范围限制的数据代理，来提供该问题以及从底层数据模型中导出的其他子集答案。

More precisely, the input-output flow goes as follows: (1) a stakeholder asks a question, (2) the agent, powered by Azure OpenAI Assistant API, interprets the question and “decides” which of data sources is most likely to have the answer based on source schemas and agent instructions, (3) generates the appropriate query (SQL, DAX, or KQL depending on the source type), (4) validates it, (5) executes it under the stakeholder’s credentials, and (6) returns the result as a text or a table, not (yet) as a visual.
更准确地说，其输入输出流程如下：（1）利益相关者提出问题；（2）由 Azure OpenAI Assistant API 驱动的代理会解读问题，并根据源架构和代理指令“决定”哪个数据源最有可能包含答案；（3）生成适当的查询（根据源类型选择 SQL、DAX 或 KQL）；（4）验证查询；（5）在利益相关者的凭据下执行查询；（6）以文本或表格形式返回结果，目前尚不支持以可视化形式返回。

In sum, a stakeholder interaction with insights via the data agent is a Q&A session on top of the curated dataset, and drill-down visuals can be replaced with follow-up questions, such as “Can you also break the revenue out by segment?” With that, it is clear how analysts’ work no longer needs to be re-expressed only via dashboards, aka the long-known tangible proof that the work of capturing the business logic within data models was delivered.
总之，利益相关者通过数据代理与洞察力的交互，本质上是在精选数据集之上的问答环节。下钻可视化图表可以被后续问题所取代，例如“你能按细分市场拆分收入吗？”由此可见，分析师的工作不再仅仅需要通过仪表板来呈现——仪表板曾被视为证明业务逻辑已成功捕获并交付到数据模型中的唯一有形证据。

### Self-service insights, closer to where business users “live”
### 自助式洞察，更贴近业务用户的“生活”场景

I mentioned before that reports can sometimes misrepresent metrics, but that’s not the only reason why “If you build it, they will come” rarely works for them or analytics in general. The truth is, the knowledge barrier is often too high to understand the underlying semantic models and how to use BI tools to create visuals on top. Although this points to data literacy, which is a change-management problem, it’s a fact that the targeted business audience, who should be report consumers, often has too much on their plate to bother learning BI tools for self-service analytics.
我之前提到过报告有时会误导指标，但这并不是“只要你构建了，他们就会来”这一理念在报告或分析领域很少奏效的唯一原因。事实是，理解底层语义模型以及如何使用 BI 工具创建可视化图表的知识门槛往往太高。虽然这指向了数据素养问题（这是一个变革管理难题），但事实是，作为报告目标受众的业务人员往往事务繁忙，无暇学习 BI 工具来进行自助式分析。

That’s why it’s important to bring insights closer to where end users “live”, which nowadays points towards AI-powered tools like M365 Copilot. With the possibility to expose insights via data agents outside of Fabric, analysts can now focus on the analytical logic behind self-service data agents, and end users can access insights in the same AI-powered tools that support their other daily tasks, without the complexity of switching to another platform.
因此，将洞察力带到终端用户“生活”的地方至关重要，而如今这指向了像 M365 Copilot 这样的人工智能工具。通过在 Fabric 之外通过数据代理公开洞察力，分析师现在可以专注于自助式数据代理背后的分析逻辑，而终端用户可以在支持其日常任务的同一人工智能工具中获取洞察，无需切换平台的复杂性。

### The difference between data and an AI agent
### 数据代理与人工智能代理的区别

We’ve learned so far that the Fabric data agent is an analytical agent focused on read-only, governed data access, capable of translating natural language prompts into complex database queries that unlock insights, even outside the Fabric tenant. On the other side, an AI agent is defined as a system that allows Large Language Models (LLMs) to do things, not just respond to prompts, on behalf of users or other systems by accessing tools and knowledge.
到目前为止，我们了解到 Fabric 数据代理是一种专注于只读、受控数据访问的分析代理，它能够将自然语言提示转换为复杂的数据库查询，从而解锁洞察力，甚至可以在 Fabric 租户之外使用。另一方面，人工智能代理被定义为一种允许大语言模型（LLM）通过访问工具和知识，代表用户或其他系统执行操作（而不仅仅是响应提示）的系统。

Meaning, the whole magic is in the AI agent setup, where you can use a Fabric data agent as a specialised tool or knowledge source. I’ll illustrate this with one simple example. Imagine an authorised user requests the AI agent to “Draft an email to the team summarising last week’s revenue by segment.” To get this work done, the AI agent would, among other things, need to prepare revenue insights from the enterprise database.
这意味着，其核心魔力在于人工智能代理的设置，你可以将 Fabric 数据代理用作专门的工具或知识源。我用一个简单的例子来说明：假设一位授权用户要求人工智能代理“起草一封给团队的邮件，总结上周各细分市场的收入”。为了完成这项工作，人工智能代理除了其他任务外，还需要从企业数据库中准备收入洞察。

So, in an aim to reduce errors in revenue calculation, the developer would design an agentic workflow to route the input prompt to the Fabric data agent tool, which would handle the heavy lifting of determining the schema, writing the query, executing it, and returning the precise figures. Finally, the AI agent would then use those figures to finish its broader workflow and write the email. What’s the difference between those two, then? It’s that an AI agent acts, while the data agent grounds.
因此，为了减少收入计算中的错误，开发人员会设计一个代理工作流，将输入提示路由到 Fabric 数据代理工具。该工具将负责处理繁重的工作，包括确定架构、编写查询、执行查询并返回精确的数字。最后，人工智能代理将使用这些数字完成其更广泛的工作流并撰写邮件。那么这两者有什么区别呢？区别在于：人工智能代理负责“行动”，而数据代理负责“提供依据（Grounding）”。
