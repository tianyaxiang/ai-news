### First, what is ETL?
### 首先，什么是 ETL？

Before I get into what I built, let me quickly explain what ETL actually means because I had to look this up myself not too long ago. ETL stands for Extract, Transform, Load. It’s one of the most fundamental concepts in data engineering.
在我介绍我构建的内容之前，先快速解释一下 ETL 的实际含义，因为不久前我自己也查过。ETL 代表提取（Extract）、转换（Transform）、加载（Load）。它是数据工程中最基础的概念之一。

*   **Extract** means going somewhere to get data. An API, a database, a website, a file. You’re pulling raw information from a source.
*   **Transform** means cleaning and shaping that data. Removing bad rows, adding new columns, restructuring it so it’s actually useful.
*   **Load** means saving the cleaned data somewhere. A database, a data warehouse, a simple CSV file.
*   **提取**意味着去某个地方获取数据。无论是 API、数据库、网站还是文件，你都是在从源头拉取原始信息。
*   **转换**意味着清洗和整理这些数据。删除无效行、添加新列、重构数据，使其真正有用。
*   **加载**意味着将清洗后的数据保存到某个地方，比如数据库、数据仓库或简单的 CSV 文件中。

That’s it. Those three steps, done in sequence, are what a data pipeline is. Everything else in data engineering, Airflow, Spark, Databricks, is just more sophisticated ways of doing those same three things at scale. I’m at the beginning of my roadmap, so I kept it simple. Pure Python, no orchestration tools yet. But the shape of the problem is the same.
就是这样。这三个步骤按顺序执行，就构成了数据流水线。数据工程中的其他一切——Airflow、Spark、Databricks——都只是在大规模环境下执行这三件事的更复杂方式。我正处于路线图的起点，所以我保持简单：纯 Python，暂不使用编排工具。但问题的本质是一样的。

### Step 1: Extract
### 第一步：提取

The first thing I had to do was figure out how to talk to the GitHub API. An API is basically a door that a company or platform opens so that developers can request data from it programmatically, without having to manually copy and paste anything. GitHub has a free, public API. No account or paid plan needed for basic searches.
我首先要做的是弄清楚如何与 GitHub API 通信。API 本质上是公司或平台打开的一扇门，以便开发人员可以通过编程方式请求数据，而无需手动复制和粘贴任何内容。GitHub 提供免费的公共 API，进行基本搜索无需账户或付费计划。

Here’s the code I wrote to extract the data:
以下是我编写的提取数据的代码：

```python
import requests

url = "https://api.github.com/search/repositories"
params = {
    "q": "language:python created:>2025-04-22",
    "sort": "stars",
    "order": "desc",
    "per_page": 30
}

response = requests.get(url, params=params)
data = response.json()

print(response.status_code)
print(data.keys())
```

I’ll be honest. This block confused me at first. The requests library was new to me. The params dictionary with that q syntax felt alien. I didn’t immediately know what .json() was doing or why I needed it.
老实说，这段代码起初让我很困惑。`requests` 库对我来说是新的，带有 `q` 语法的 `params` 字典感觉很陌生。我一开始并不知道 `.json()` 在做什么，或者为什么需要它。

Let me break it down simply. `requests.get()` is how you knock on GitHub’s door and ask for something. The `url` is the address of what you’re asking for. The `params` dictionary is the specific question you’re asking. In this case: “give me Python repos, sorted by stars, created after April 22, show me 30 results.” `.json()` converts GitHub’s response from raw text into a Python dictionary that you can actually work with.
让我简单拆解一下：`requests.get()` 是你敲开 GitHub 大门并提出请求的方式。`url` 是你请求的目标地址，`params` 字典是你提出的具体问题。在本例中，意思是：“给我 Python 仓库，按星标排序，创建于 4 月 22 日之后，显示 30 个结果。” `.json()` 将 GitHub 的响应从原始文本转换为你可以实际操作的 Python 字典。

When I ran it, I got this:
当我运行它时，我得到了：

```text
200
dict_keys(['total_count', 'incomplete_results', 'items'])
```

The 200 means success. That’s the internet’s way of saying “your request worked.” If you see 403 or 404, something went wrong. The dictionary has three keys. `total_count` tells you how many repos matched the search. `incomplete_results` tells you if GitHub had to cut anything short. And `items` is where the actual data lives.
200 表示成功。这是互联网告诉你“请求已成功”的方式。如果你看到 403 或 404，说明出错了。这个字典有三个键：`total_count` 告诉你匹配搜索的仓库数量；`incomplete_results` 告诉你 GitHub 是否截断了结果；而 `items` 则是实际数据所在的地方。

### Step 2: Transform
### 第二步：转换

Now I had 30 repos sitting in a Python list, each one a nested dictionary with dozens of fields. Most of which I didn’t need. The Transform step is where you take that raw, messy data and shape it into something clean and purposeful. First I pulled out only the fields I cared about and loaded them into a Pandas dataframe:
现在我有 30 个仓库存储在一个 Python 列表中，每个仓库都是一个包含数十个字段的嵌套字典。其中大部分我都不需要。转换步骤就是将这些原始、杂乱的数据整理成干净且有目的性的格式。首先，我只提取了我关心的字段，并将它们加载到 Pandas 数据框中：

```python
import pandas as pd

repos = []
for repo in data['items']:
    repos.append({
        "name": repo['name'],
        "owner": repo['owner']['login'],
        "stars": repo['stargazers_count'],
        "forks": repo['forks_count'],
        "language": repo['language'],
        "description": repo['description'],
        "url": repo['html_url'],
        "created_at": repo['created_at']
    })

df = pd.DataFrame(repos)
df.head()
```

Seeing that dataframe appear was a proper “wow” moment. I went from a wall of JSON to a clean, readable table.
看到数据框出现的那一刻，我真的感到“哇”。我从一堆 JSON 变成了清晰、可读的表格。
