# Introducing the Agent Toolkit for Amazon Web Services
# 介绍 Amazon Web Services (AWS) 代理工具包 (Agent Toolkit)

It’s like having your own personal expert AWS solutions architect and data engineer rolled into one.
这就像拥有了一位集 AWS 解决方案架构师和数据工程师于一身的私人专家。

### What Is Agent Toolkit for AWS?
### 什么是 AWS 代理工具包？

Agent Toolkit for AWS is an open-source project developed by AWS that helps AI coding agents work with AWS more reliably. With the recent addition of the newly released MCP server as part of the Toolkit, coding agents using the Toolkit can now access the AWS-specific context, workflows, guardrails, and tools they need to build, deploy, debug, and operate cloud systems without relying solely on general model knowledge, which is often out of date.
AWS 代理工具包是由 AWS 开发的一个开源项目，旨在帮助 AI 编程代理更可靠地与 AWS 协同工作。随着工具包中新发布的 MCP 服务器的加入，使用该工具包的编程代理现在可以访问构建、部署、调试和运维云系统所需的 AWS 特定上下文、工作流、护栏和工具，而无需仅仅依赖往往已经过时的通用模型知识。

Instead of asking a coding agent to improvise from memory, the Toolkit gives it curated, task-specific instructions. These are packaged as skills, plugins, rules, and an MCP server configuration.
工具包不再要求编程代理凭记忆即兴发挥，而是为其提供经过整理的、特定于任务的指令。这些指令被封装为技能 (skills)、插件 (plugins)、规则 (rules) 和 MCP 服务器配置。

Skills are focused instruction packs. They guide the agent through specific AWS tasks, such as creating an S3 Tables lakehouse table, deploying a serverless app, debugging Lambda timeouts, connecting AWS Glue to a database, or adding memory to an AgentCore agent.
“技能”是专注的指令包。它们引导代理完成特定的 AWS 任务，例如创建 S3 Tables 湖仓表、部署无服务器应用、调试 Lambda 超时、将 AWS Glue 连接到数据库，或为 AgentCore 代理添加内存。

Plugins group related skills together. For example, aws-core covers general AWS development, aws-agents covers Bedrock AgentCore workflows, and aws-data-analytics covers S3 Tables, Glue, Athena, data discovery, and vector storage.
“插件”将相关的技能组合在一起。例如，aws-core 涵盖通用 AWS 开发，aws-agents 涵盖 Bedrock AgentCore 工作流，而 aws-data-analytics 则涵盖 S3 Tables、Glue、Athena、数据发现和向量存储。

Rules files set the agent’s default AWS behaviour. They can tell the agent to prefer infrastructure-as-code, check AWS documentation when unsure, and use AWS MCP tools when available.
“规则”文件用于设置代理的默认 AWS 行为。它们可以指示代理优先使用基础设施即代码 (IaC)、在不确定时查阅 AWS 文档，并在可用时使用 AWS MCP 工具。

AWS MCP Server integration provides agents with access to live AWS documentation, AWS APIs, sandboxed script execution, and auditing via AWS-native controls. The result should be better systems, with more resilience.
AWS MCP 服务器集成使代理能够访问实时 AWS 文档、AWS API、沙盒脚本执行，并通过 AWS 原生控件进行审计。其结果将是构建出更优质、更具韧性的系统。

### Why It Matters
### 为什么它很重要

Modern coding agents can write plausible AWS CLI commands, Terraform, CDK, Lambda handlers, Glue jobs, or IAM policies. Often, these will be correct and usable straight away, but there is a potential problem, and it’s the same issue that ALL coding agents struggle with: Knowledge cut-off.
现代编程代理可以编写出看似合理的 AWS CLI 命令、Terraform、CDK、Lambda 处理程序、Glue 作业或 IAM 策略。通常，这些代码可以直接使用，但存在一个潜在问题，这也是所有编程代理都面临的困境：知识截止日期 (Knowledge cut-off)。

When agents are trained, they are exposed to the latest information available at that time, but when the models are released, that information is typically many months out of date. For example, OpenAI’s latest model at the time of writing is GPT 5.5. It was released towards the end of April 2026, but its knowledge cut-off data was the 1st December 2025. And during the intervening period, new services are introduced, and existing systems, API calls, documentation, etc., are updated.
当代理接受训练时，它们接触的是当时最新的信息，但当模型发布时，这些信息通常已经过时了数月。例如，在撰写本文时，OpenAI 最新的模型是 GPT 5.5。它于 2026 年 4 月底发布，但其知识截止日期为 2025 年 12 月 1 日。在此期间，新的服务不断推出，现有的系统、API 调用、文档等也都在更新。

Cloud development is full of details that may seem small but can break real systems. For example, when creating an analytics table with Amazon S3 Tables, a generic agent might generate an Athena DDL statement with a LOCATION clause because this pattern is common for external tables. But with S3 Tables, that is wrong: the service manages table storage. The correct pattern is to keep the SQL clean and pass the S3 Tables catalogue through Athena’s query execution context.
云开发充满了看似微小但可能导致系统崩溃的细节。例如，在使用 Amazon S3 Tables 创建分析表时，通用代理可能会生成带有 LOCATION 子句的 Athena DDL 语句，因为这种模式在外部表中很常见。但对于 S3 Tables 来说，这是错误的：该服务会自动管理表存储。正确的模式是保持 SQL 简洁，并通过 Athena 的查询执行上下文传递 S3 Tables 目录。

The Agent Toolkit for AWS helps avoid that kind of mistake. Its skills guide the agent to:
AWS 代理工具包有助于避免此类错误。其技能可以引导代理：
* Check what already exists before creating new resources (在创建新资源前检查现有资源)
* Use the correct AWS APIs (使用正确的 AWS API)
* Avoid patterns that AWS does not support (避免使用 AWS 不支持的模式)
* Verify assumptions against current AWS documentation (根据最新的 AWS 文档验证假设)
* Produce tighter IAM policies (生成更严谨的 IAM 策略)
* Run checks after making changes (在更改后运行检查)
* Follow the right troubleshooting path when something breaks (在出现故障时遵循正确的排查路径)

That matters most in AWS work, where the hard part is not writing code. It is writing code that fits the specific AWS service, permissions model, and operating environment.
这在 AWS 工作中至关重要，因为难点不在于编写代码，而在于编写符合特定 AWS 服务、权限模型和运行环境的代码。

### Installing Agent Toolkit in your coding agent
### 在编程代理中安装代理工具包

The agent toolkit is available for most modern coding agents, such as Claude Code, Cursor, Kiro, and VS Code. Sample installation instructions for each agent are in the AWS Toolkit repo, which I’ll link to at the end of the article.
该代理工具包适用于大多数现代编程代理，如 Claude Code、Cursor、Kiro 和 VS Code。每个代理的示例安装说明都在 AWS Toolkit 仓库中，我会在文章末尾提供链接。

My favourite coding agent to use right now is Codex, so that’s what I’ll use in my example. Install that first if you want to follow along. To install the Toolkit for Codex, type the following into a terminal window:
我目前最喜欢的编程代理是 Codex，因此我将在示例中使用它。如果你想跟随操作，请先安装它。要为 Codex 安装工具包，请在终端窗口中输入以下内容：

`$ codex plugin marketplace add aws/agent-toolkit-for-aws`

Next, open the Codex app and type the following:
接下来，打开 Codex 应用并输入：

`/plugins`

Depending on what you may have previously installed, you should see something like this:
根据你之前安装的内容，你应该会看到类似以下的内容：

* AI Agents on AWS
* AWS Core
* AWS Data Analytics
* Browser
* Documents
* Presentations
* Spreadsheets

That means all the required AWS-related plugins are available for your agent to use. Remember, if you run into any issues, just ask your coding agent to fix them.
这意味着所有必需的 AWS 相关插件都已可供你的代理使用。请记住，如果遇到任何问题，只需让你的编程代理修复它们即可。

### Using the Toolkit with your coding agent
### 在编程代理中使用工具包

This is the easy part, as you literally just tell Codex in English what you want to achieve. For my example, I wanted to:
这部分很简单，你只需用英语告诉 Codex 你想实现的目标。在我的示例中，我想：

1. Create an Iceberg orders table using Amazon S3 Tables (使用 Amazon S3 Tables 创建一个 Iceberg 订单表)
2. Ingest order data from a JDBC source with AWS Glue (使用 AWS Glue 从 JDBC 源摄取订单数据)
3. Validate and query the iceberg table with Athena (使用 Athena 验证并查询该 Iceberg 表)

This might seem like a fairly simple request, but when you break it down, it’s a lot more involved than you think. For starters, I don’t have an existing JDBC data source, so I also had to ask Codex to create an RDS database first and populate it with some dummy data. That alone spawns a bunch of other requirements, since my RDS database table is private, I needed a VPC, security groups, IAM permissions, etc. You get the point, and this will be a well-known issue with anyone reading this who uses AWS in anger.
这看起来可能是一个相当简单的请求，但当你拆解它时，会发现它比你想象的要复杂得多。首先，我没有现成的 JDBC 数据源，所以我还得让 Codex 先创建一个 RDS 数据库并填充一些虚拟数据。仅此一项就衍生出了一堆其他需求，因为我的 RDS 数据库表是私有的，我需要 VPC、安全组、IAM 权限等。你明白我的意思，任何经常使用 AWS 的读者都会非常熟悉这个问题。

Even a fairly simple AWS system usually requires a complex setup, as you have to account for security, authorisation, and permissions. But as you will see, the AWS Toolkit does all the heavy lifting for us.
即使是一个相当简单的 AWS 系统，通常也需要复杂的设置，因为你必须考虑安全性、授权和权限。但正如你将看到的，AWS 工具包为我们完成了所有繁重的工作。

Note: Agent Toolkit for AWS runs inside your coding-agent environment. When it needs to inspect, create or change AWS resources, it uses the AWS credentials configured in that environment. For local development, that usually means AWS CLI credentials, SSO, or environment variables, so make sure one or other of these methods is set up before you start.
注意：AWS 代理工具包在你的编程代理环境中运行。当它需要检查、创建或更改 AWS 资源时，它会使用在该环境中配置的 AWS 凭证。对于本地开发，这通常意味着 AWS CLI 凭证、SSO 或环境变量，因此在开始之前，请确保已设置其中一种方法。
