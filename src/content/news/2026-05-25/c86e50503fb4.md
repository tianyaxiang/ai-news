### 2. What is API
### 2. 什么是 API

An API (Application Programming Interface) comprises a set of methods by which different programs communicate with each other and exchange data. Essentially, it is an intermediary that allows applications, devices, servers, and other systems to exchange information, while hiding the processes within each system from each other.
API（应用程序编程接口）包含了一组方法，通过这些方法，不同的程序可以相互通信并交换数据。本质上，它是一个中介，允许应用程序、设备、服务器和其他系统交换信息，同时对彼此隐藏各自系统内部的处理过程。

Imagine a library with a large collection of books, and the librarian who knows where to find the exact book a certain reader needs. Here we can referred the librarian as an API that simplifies the process of accessing information, saving readers (our “frontend”) from wasting time searching through the entire book catalog (our “backend”), allowing to focus only on their specific request. Furthermore, if readers need another books, they can repeat the process of sending request to API again.
想象一下一个藏书丰富的图书馆，图书管理员知道在哪里可以找到读者需要的特定书籍。在这里，我们可以将图书管理员比作 API，它简化了获取信息的过程，使读者（我们的“前端”）不必浪费时间在整个图书目录（我们的“后端”）中搜索，从而能够专注于他们的特定需求。此外，如果读者需要其他书籍，他们可以重复向 API 发送请求的过程。

This analogy highlights the role of the API as an intermediary between the user and the data source, providing convenient and efficient access to information. A special case of API is a REST API which follows the concepts of the REST (REpresentational State Transfer) architecture. REST APIs are named the industry standard because they are lightweight, flexible, and use common data formats like JSON or XML.
这个类比突显了 API 作为用户与数据源之间中介的作用，提供了便捷高效的信息访问方式。API 的一种特殊形式是 REST API，它遵循 REST（表述性状态转移）架构的概念。REST API 被称为行业标准，因为它们轻量、灵活，并使用 JSON 或 XML 等通用数据格式。

### 3. Components of REST API
### 3. REST API 的组成部分

Each of the REST API components below plays a vital role in organizing client-server interactions.
以下每个 REST API 组件在组织客户端与服务器交互方面都发挥着至关重要的作用。

#### 3.1. Resources
#### 3.1. 资源 (Resources)

A resource is any entity that can be accessed through the API. Each resource has a unique identifier (URI), for example: `https://api.thecatapi.com/v1/images/search?size=med`
资源是指可以通过 API 访问的任何实体。每个资源都有一个唯一标识符（URI），例如：`https://api.thecatapi.com/v1/images/search?size=med`

Here, `images` is a collection of cats’ images from The Cat API web page [1], and `search?size=med` is the filter to view only medium-sized images.
在这里，`images` 是来自 The Cat API 网页 [1] 的猫咪图片集合，而 `search?size=med` 是仅查看中等尺寸图片的过滤器。

#### 3.2. HTTP Methods
#### 3.2. HTTP 方法

HTTP methods are used to interact with resources:
HTTP 方法用于与资源进行交互：

* **GET** — retrieve data about a resource;
* **GET** — 获取资源数据；
* **POST** — create a new resource;
* **POST** — 创建新资源；
* **PUT** — update a resource;
* **PUT** — 更新资源；
* **PATCH** — partial update a resource;
* **PATCH** — 部分更新资源；
* **DELETE** — delete a resource.
* **DELETE** — 删除资源。

#### 3.3. Requests and Responses
#### 3.3. 请求与响应

Data is exchanged between the client and server via HTTP requests and responses. In most cases, the JSON format is used because it is easy to read and supported by the vast majority of programming languages.
数据通过 HTTP 请求和响应在客户端与服务器之间交换。在大多数情况下，使用 JSON 格式，因为它易于阅读且受到绝大多数编程语言的支持。

#### 3.4. HTTP Headers
#### 3.4. HTTP 头部 (Headers)

Headers are used to convey additional information, such as the content type (Content-Type) or authentication parameters (Authorization).
头部用于传达额外信息，例如内容类型（Content-Type）或身份验证参数（Authorization）。

#### 3.5. HTTP Response Codes
#### 3.5. HTTP 响应代码

Each HTTP request receives a response with a specific status code:
每个 HTTP 请求都会收到一个带有特定状态代码的响应：

* **200 OK** — successful request;
* **200 OK** — 请求成功；
* **201 Created** — resource successfully created;
* **201 Created** — 资源创建成功；
* **400 Bad Request** — client request error;
* **400 Bad Request** — 客户端请求错误；
* **401 Unauthorized** — lack of access rights;
* **401 Unauthorized** — 缺乏访问权限；
* **404 Not Found** — resource not found;
* **404 Not Found** — 未找到资源；
* **500 Internal Server Error** — server-side error.
* **500 Internal Server Error** — 服务器端错误。

### 4. API Clients
### 4. API 客户端

API clients like Postman or Bruno [2] simplify API interaction by providing a dedicated workspace for sending requests and managing responses. Instead of using command-line tools or writing code as we did in Case 6.1, these agents offer visual interfaces and automation features that speed up workflows. Thus, in Case 6.2, we will consider using Bruno to interact with the JokeAPI web page [3]. Using Bruno simplifies the complex process of interaction between different software systems. Without Bruno and other API clients, developers would have to manually construct each HTTP request and process each raw response from scratch.
像 Postman 或 Bruno [2] 这样的 API 客户端通过提供专门的工作区来发送请求和管理响应，从而简化了 API 交互。与我们在案例 6.1 中使用命令行工具或编写代码不同，这些代理提供了可视化界面和自动化功能，加快了工作流程。因此，在案例 6.2 中，我们将考虑使用 Bruno 与 JokeAPI 网页 [3] 进行交互。使用 Bruno 简化了不同软件系统之间复杂的交互过程。如果没有 Bruno 和其他 API 客户端，开发人员将不得不手动构建每个 HTTP 请求并从头处理每个原始响应。

### 5. Tips on Creating Good API Documentation
### 5. 创建优秀 API 文档的建议

Creating effective API documentation is crucial for ensuring that users can easily understand and utilize your API. Here are some key tips to keep in mind:
创建有效的 API 文档对于确保用户能够轻松理解和使用您的 API 至关重要。以下是一些需要牢记的关键建议：

#### 5.1. Prioritize Simplicity, Clarity, and Consistency
#### 5.1. 优先考虑简洁、清晰和一致性

Avoid technical jargon and inconsistent terminology. Instead, use straightforward and simple enough language that is easy to follow. If necessary, establish a style guide to maintain uniformity throughout your documentation.
避免使用技术术语和不一致的术语。相反，使用直观且简单的语言，使其易于理解。如有必要，建立风格指南以保持整个文档的统一性。
