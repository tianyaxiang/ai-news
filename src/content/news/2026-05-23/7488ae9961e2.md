### Introduction
### 引言

I tried to build an agentic AI network for my company that advises manufacturing plants on how to mature their operations. The system was designed to be data-driven, allowing users to upload assessment data directly through the chat interface. The first working prototype was finished surprisingly quickly, and at first glance the results looked promising.
我曾尝试为公司构建一个代理式人工智能网络，旨在为制造工厂提供运营成熟度方面的建议。该系统被设计为数据驱动型，允许用户直接通过聊天界面上传评估数据。第一个工作原型完成得异常迅速，乍看之下，结果似乎很有前景。

There was only one problem: Most of the results were wrong! Even worse, the AI quickly learned which numerical ranges looked plausible and began generating convincing — but fabricated — outputs. Combined with the eloquent language generation of the LLM, these results could easily be mistaken for truth. And this behavior was not limited to a single model. Similar patterns appeared across all tested systems: ChatGPT, Gemini Enterprise, DIA Brain, and Microsoft Copilot.
但有一个问题：大部分结果都是错的！更糟糕的是，人工智能很快学会了哪些数值范围看起来“合理”，并开始生成令人信服但却是捏造的输出。结合大语言模型（LLM）流畅的语言生成能力，这些结果很容易被误认为是事实。这种行为并非仅限于单一模型，在所有测试过的系统中都出现了类似的模式：ChatGPT、Gemini Enterprise、DIA Brain 和 Microsoft Copilot。

But, plausible data is not enough, Enterprise AI systems require reliable data! Further investigation revealed recurring failure modes. Even with “Code Interpreter” enabled, the systems: skipped rows or columns, applied incorrect filters, returned identical results for different inputs, silently mixed parts of the dataset, or simply collapsed under more complex analytical tasks. This led to a crucial realization: Probabilistic reasoning is extremely powerful for interpretation and interaction — but foundational data analysis requires deterministic execution.
然而，看似合理的数据是不够的，企业级人工智能系统需要的是可靠的数据！进一步调查揭示了反复出现的故障模式。即使启用了“代码解释器”（Code Interpreter），这些系统依然会：跳过行或列、应用错误的筛选器、对不同的输入返回相同的结果、悄无声息地混淆数据集的部分内容，或者在处理更复杂的分析任务时直接崩溃。这引出了一个关键的认识：概率推理在解释和交互方面极其强大，但基础数据分析需要确定性的执行。

### 1 The Use Case
### 1 应用场景

Although the specific use case is of secondary importance, it is briefly outlined here to support the practical understanding of the underlying architectural challenge. The primary task of our agent is to advise manufacturing plants and value streams on how to improve their operational maturity: optimizing processes, improving productivity, reducing inventory levels, and ultimately lowering operational costs.
尽管具体的应用场景属于次要信息，但在此简要概述，以帮助读者从实践层面理解底层架构所面临的挑战。我们代理的主要任务是为制造工厂和价值流提供如何提高运营成熟度的建议：包括优化流程、提高生产力、降低库存水平，并最终降低运营成本。

To achieve this, the consultation agent operates in two modes: It provides generic recommendations for improving specific operational topics based on the retrieval of specialized “how-to” documentation and assessment questionnaires. The agent is intended to analyze the current situation of a plant or value stream based on assessment results and assessors’ written recommendations. Based on this analysis, it is expected to provide highly specific recommendations for the next improvement steps.
为实现这一目标，咨询代理以两种模式运行：它通过检索专业的“操作指南”文档和评估问卷，为改进特定运营主题提供通用建议。此外，代理旨在根据评估结果和评估人员的书面建议，分析工厂或价值流的现状。基于此分析，它应能为下一步的改进措施提供高度具体的建议。

In both modes — as with most LLM-based AI models — the user can interactively discuss ideas and recommendations with the agent in order to derive the most suitable action plan. For the second operation mode, it is essential that the agent can reliably process and analyze assessment data. In our case, this data is provided as an Excel export from a central database. Ideally, the agent should be able to process the file without any prior manual preparation.
在这两种模式下——正如大多数基于大语言模型的人工智能模型一样——用户可以与代理进行交互式讨论，以得出最合适的行动计划。对于第二种操作模式，代理必须能够可靠地处理和分析评估数据，这一点至关重要。在我们的案例中，这些数据以 Excel 导出文件的形式从中央数据库提供。理想情况下，代理应该能够在无需任何预先手动准备的情况下处理该文件。

The structure of the file, however, is challenging. Since all assessment results, intermediate calculations, metadata, and detailed assessment questions are stored in separate columns, the worksheet contains more than 800 columns. The number of rows corresponds to the number of assessments in the database and can range from one to several hundreds (Fig. 1). Assessment ratings are represented as integers from 0 to 4. In addition, the file contains more than 160 free-text fields with qualitative observations, strengths, weaknesses, and recommendations from the assessors.
然而，该文件的结构极具挑战性。由于所有的评估结果、中间计算、元数据和详细的评估问题都存储在不同的列中，工作表包含超过 800 列。行数对应于数据库中的评估数量，范围从一行到几百行不等（图 1）。评估等级以 0 到 4 的整数表示。此外，该文件还包含超过 160 个自由文本字段，用于记录评估人员的定性观察、优势、劣势和建议。

### 2 The Hybrid Architecture
### 2 混合架构

The core idea for overcoming the analytical challenge was to clearly separate deterministic data analysis from LLM-based reasoning and interpretation. Fig. 2 shows the selected system architecture after several improvement iterations. The system was implemented in Microsoft Copilot Studio because the platform allows deterministic workflow elements, such as topics and flows, to be combined with LLM-based reasoning components.
克服这一分析挑战的核心思路是：将确定性数据分析与基于大语言模型的推理和解释明确分开。图 2 展示了经过多次改进迭代后所选定的系统架构。该系统是在 Microsoft Copilot Studio 中实现的，因为该平台允许将确定性的工作流元素（如主题和流程）与基于大语言模型的推理组件相结合。

The parent agent handles all communication with the user. It orchestrates the sub agents and the analytics module, delegates tasks to them, receives their responses, and composes the final answer. The sub agents are specialized LLM-based modules with access to specific knowledge sources. These include descriptions of maturity-level expectations for the value streams, questionnaires with detailed assessment questions, and more general guidelines for operational excellence.
父代理负责处理与用户的所有沟通。它协调子代理和分析模块，将任务委派给它们，接收它们的响应，并汇总最终答案。子代理是专门的、基于大语言模型的模块，可以访问特定的知识源。这些知识源包括价值流成熟度期望的描述、包含详细评估问题的问卷，以及更通用的卓越运营指南。

The analytics module is the main focus of this article. It performs the deterministic data analysis and is designed to provide reproducible and reliable analytical results. It receives an analysis instruction in natural language from the parent agent, referred to as Parent_Instruction. The analytics module itself consists of topics, flows, and AI modules, which are called “prompts” in Copilot Studio. The topic T_receive_Excel_File handles the upload and storage of assessment files.
分析模块是本文的重点。它执行确定性数据分析，旨在提供可复现且可靠的分析结果。它从父代理接收自然语言的分析指令（称为 Parent_Instruction）。分析模块本身由主题、流程和人工智能模块组成，在 Copilot Studio 中被称为“提示词”（Prompts）。主题 T_receive_Excel_File 负责处理评估文件的上传和存储。
