The previous part turned a PDF into `line_df`, one row per line of text on the page. This one covers the rest of the model: the full set of tables a parser should emit, what each one holds, and how they link together, so the table on page 14 keeps its columns and the renewal fee stays attached to its label. The other three bricks, and the highlighted answer at the end, all read these tables, never the raw PDF.
上一篇将 PDF 转换为了 `line_df`（页面上每一行文本对应一行数据）。本文将涵盖模型的其余部分：解析器应输出的完整表格集、每个表格的内容及其关联方式，从而确保第 14 页的表格能保留其列结构，且续费金额能与其标签保持对应。其余三个模块以及最后的突出显示答案，读取的都是这些表格，而非原始 PDF。

RAG tutorials start the same way: `text = extract_text(pdf)`. That single line is where the PDF problems begin. You build a RAG pipeline. It works on a few clean documents. Then a customer sends you a real contract: 30 pages, with a Schedule of Charges table on page 14. The user asks “what’s the renewal fee?” and the model returns the wrong number. The team says: “the model can’t read tables.” The model reads tables fine. The problem is upstream.
RAG 教程通常以 `text = extract_text(pdf)` 开头。正是这一行代码导致了 PDF 处理问题的开端。你构建了一个 RAG 流水线，它在处理少量干净文档时表现良好。但当客户发来一份真实的合同（30 页，第 14 页包含一份费用明细表）时，用户询问“续费金额是多少？”，模型却返回了错误的数字。团队会说：“模型读不懂表格。”其实模型读表格没问题，问题出在前端处理上。

Your parser walked the table cell by cell and joined them into one long string. The column structure is gone. The link between a label and its amount is gone. Your model is asked to guess which number is the renewal fee. Sometimes it guesses right. Often it doesn’t.
你的解析器逐个单元格遍历表格，并将它们拼接成一个长字符串。列结构消失了，标签与金额之间的关联也消失了。你的模型被迫去猜测哪个数字是续费金额。有时它猜对了，但更多时候是错的。

The parser did not fail. It gave you what you asked for. You asked for the wrong thing. A good PDF parser does not extract text. It models the document as a relational set of tables. One PDF in, one table per kind of thing out (seven or eight today, and more as new needs show up).
解析器并没有失败，它只是给了你所要求的东西。是你要求的内容不对。一个优秀的 PDF 解析器不会仅仅提取文本，而是将文档建模为一组关系型表格。输入一个 PDF，输出每种类型对应的表格（目前有七八种，随着新需求出现还会增加）。

*   `toc_df`: the sections, like the author wrote them.
*   `page_df` and `line_df`: the body. Every page. Every line.
*   `image_df`: every figure on every page.
*   `span_df`: bold, italic, color, font size. Every span of every line.
*   `object_registry`: every figure caption, every table caption, every annex.
*   `cross_ref_df`: every “see Figure 2”, every “see Table 4”, every “see Annex B”.
*   `parsing_summary`: tells you if the PDF is born-digital, scanned, or mixed. Tells you if the OCR is good or bad.
*   `toc_df`：章节，即作者编写的结构。
*   `page_df` 和 `line_df`：正文。每一页，每一行。
*   `image_df`：每一页上的每一个图形。
*   `span_df`：粗体、斜体、颜色、字号。每一行的每一个文本跨度。
*   `object_registry`：每一个图注、表注、附录。
*   `cross_ref_df`：每一个“见图 2”、“见表 4”、“见附录 B”。
*   `parsing_summary`：告知 PDF 是原生数字格式、扫描件还是混合格式，并评估 OCR 的质量。

Retrieval reads these tables. Generation reads these tables. Highlighting reads these tables. You open the PDF once. After that, you only work with tables.
检索读取这些表格，生成读取这些表格，高亮显示也读取这些表格。你只需打开一次 PDF，之后的工作全部基于这些表格进行。

### 1. One table per entity
### 1. 每个实体对应一个表格

Everything we’ve extracted gets returned as a dictionary of tables plus a parsing summary, one table per entity of the document model. The `<unit>_df` naming convention makes granularity readable from the name itself.
我们提取的所有内容都以表格字典加解析摘要的形式返回，文档模型中的每个实体对应一个表格。`<unit>_df` 的命名约定使得从名称本身就能读出数据的粒度。

### 1.1. toc_df: table of contents
### 1.1. toc_df：目录

TOCs are everywhere in enterprise documents. Contracts, reports, policies, employee manuals, regulatory filings: almost all of them ship with a declared section structure, and that structure is the cheapest semantic signal you can hand a retriever.
目录在企业文档中无处不在。合同、报告、政策、员工手册、监管文件：几乎所有文档都带有明确的章节结构，而这种结构是你能提供给检索器最廉价的语义信号。

The catch: it isn’t always native. Sometimes it’s only typographic (bold headings, numbered sections, indented subheadings) and has to be reconstructed from `line_df` + `span_df`.
问题在于：它并不总是原生的。有时它仅仅是排版上的（粗体标题、编号章节、缩进子标题），必须通过 `line_df` + `span_df` 进行重构。
