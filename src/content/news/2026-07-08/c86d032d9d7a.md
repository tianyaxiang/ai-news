# A Production RAG Pipeline for PDFs: Relational Parsing, TOC Retrieval, Typed Answers
# 面向生产环境的 PDF RAG 流水线：关系型解析、目录检索与类型化答案

Enterprise Document Intelligence [Vol.1 #9A] – Same paper, same question as Article 1. One upgraded contract per brick: document parsing, question parsing, retrieval, generation.
企业文档智能 [第1卷 #9A] —— 与第1篇文章使用相同的论文和问题。每个模块（Brick）都升级了契约：文档解析、问题解析、检索与生成。

This article opens Part III of Enterprise Document Intelligence, a series that builds an enterprise RAG system from four bricks: document parsing, question parsing, retrieval, and generation. It is the first of two parts on the upgraded pipeline: this part upgrades each brick, one contract at a time, on the same paper and the same question as Article 1 (minimal RAG).
本文开启了“企业文档智能”系列的第三部分。该系列旨在通过四个模块构建企业级 RAG 系统：文档解析、问题解析、检索和生成。这是关于升级版流水线的两部分内容中的第一部分：本篇将逐一升级每个模块的契约，并使用与第1篇（最小化 RAG）相同的论文和问题进行演示。

A hundred lines of Python wire four functions together: parse the PDF, parse the question, retrieve a few pages, ask a model. That pipeline returns the right answer on a clean question against a paper with a built-in table of contents. On a real corpus it breaks the first time the user types “positonal encodig” with two typos, the first time the document is a 200-page contract with no PDF outline, the first time the question asks for every exclusion instead of one, the first time downstream code wants a typed object instead of a string.
一百行 Python 代码将四个功能串联起来：解析 PDF、解析问题、检索页面、调用模型。该流水线在处理带有内置目录的论文及规范问题时能返回正确答案。但在真实语料库中，一旦用户输入带有两个拼写错误的“positonal encodig”，或者文档是长达 200 页且没有 PDF 大纲的合同，又或者问题要求列出所有排除项而非仅一项，亦或是下游代码需要类型化对象而非字符串时，该流水线就会崩溃。

Four bricks need an upgrade each before the pipeline ships. Document parsing now returns more than a flat line_df: a relational set including a TOC, page-level metadata, and a typed parsing_summary carrying the document’s type, language, and a one-paragraph summary of what it is about.
在流水线投入生产前，四个模块都需要升级。文档解析现在返回的不再仅仅是一个扁平的 `line_df`，而是一个关系型数据集，包含目录（TOC）、页面级元数据，以及一个类型化的 `parsing_summary`，其中涵盖了文档类型、语言以及一段关于文档内容的摘要。

Question parsing turns the noisy user input into a structured brief, with keywords corrected against the corpus’s own vocabulary and an inferred answer shape (single value, listing, table).
问题解析将嘈杂的用户输入转化为结构化的简报，利用语料库自身的词汇表校正关键词，并推断出答案的格式（单值、列表或表格）。

Retrieval reads the TOC the way an expert would: hand the whole TOC to a small LLM that picks sections by semantic relevance, then merge with the keyword pages.
检索过程模仿专家的阅读方式：将整个目录交给一个小模型，由其根据语义相关性挑选章节，然后与关键词匹配的页面进行合并。

Generation returns a typed answer with one citable span per item, plus four context-quality indicators the pipeline reads to decide whether to ship the answer or run another pass.
生成模块返回一个类型化的答案，每个条目都附带可引用的片段，并包含四个上下文质量指标，流水线据此决定是直接输出答案还是进行二次处理。

### 1. Where the baseline RAG breaks
### 1. 基准 RAG 在何处失效

Pick a clean question on a clean PDF, run it through the simplest RAG pipeline: parse the PDF, extract keywords from the question, retrieve a few pages, ask an LLM. On the Attention paper with the question “What are the options for positional encoding?”, that pipeline returns “sinusoidal positional encoding and learned positional embeddings” with one contiguous span of pages cited. It worked, on a clean question, on a clean paper.
选取一个规范的 PDF 和一个规范的问题，运行最简单的 RAG 流水线：解析 PDF、提取关键词、检索页面、询问 LLM。对于《Attention Is All You Need》论文，当问题为“位置编码有哪些选项？”时，该流水线返回“正弦位置编码和学习到的位置嵌入”，并引用了连续的页面。在规范的问题和论文下，它运行良好。

Drop the same pipeline into an enterprise setting and four weak points appear quickly, one per brick:
将同样的流水线放入企业环境，四个弱点会迅速显现，每个模块各占一个：

*   **Document parsing:** the document is flattened. Article 1 (minimal RAG) parsed the PDF into one flat list of lines, enough to count keywords but nothing more. The pages, the sections, the tables, all the structure a downstream brick could scope on, thrown away at the first step.
    *   **文档解析：** 文档被扁平化了。第1篇（最小化 RAG）将 PDF 解析为单一的行列表，仅足以统计关键词。页面、章节、表格等下游模块本可利用的结构信息，在第一步就被丢弃了。

*   **Question parsing:** the question users type is rarely clean. “What are the optoins for posiitional encoding?” has two typos. The baseline keyword extractor never saw the word positional, it saw posiitional, so retrieval misses the very pages the answer lives on.
    *   **问题解析：** 用户输入的问题很少是规范的。“What are the optoins for posiitional encoding?” 有两个拼写错误。基准关键词提取器从未见过“positional”，它看到的是“posiitional”，因此检索过程错过了答案所在的页面。

*   **Retrieval:** the baseline never looks at structure. The Attention paper carries a clean built-in TOC, named sections with page numbers, three levels deep. The most precise retrieval signal in the whole document, ignored.
    *   **检索：** 基准方案从不关注结构。《Attention》论文带有清晰的内置目录，包含三级深度的命名章节和页码。这是整篇文档中最精确的检索信号，却被忽略了。

*   **Generation:** the answer comes back as a raw string. A list question (options) asks for one item per option, each with its own evidence. Free-form prose forces the caller to re-parse the answer to find the items; a typed schema with a per-item evidence span removes that step.
    *   **生成：** 答案以原始字符串形式返回。列表类问题（如“选项”）要求每个选项对应一个条目，且每个条目需有各自的证据。自由格式的文本迫使调用者重新解析答案以提取条目；而带有条目级证据片段的类型化模式则省去了这一步。

### 2. Four bricks, upgraded
### 2. 四大模块的升级

The shape that survives the upgrade is the same four bricks Article 1 (minimal RAG) introduced. What changes is the contract per brick: what each one consumes, what it produces, and how the next one consumes it.
升级后保留的架构依然是第1篇（最小化 RAG）中引入的那四个模块。改变的是每个模块的契约：它们各自消耗什么、产生什么，以及下一个模块如何使用这些产出。

Each of the four bricks is upgraded in its own articles, worth reading for the full contract:
这四个模块的升级过程在各自的文章中都有详细说明，阅读这些文章有助于理解完整的契约：

*   **Document parsing:** Article 5A (what to read in a PDF) and Article 5B (the relational data model).
    *   **文档解析：** 第 5A 篇（PDF 阅读内容）和第 5B 篇（关系数据模型）。
*   **Question parsing:** Article 6A (the thesis), Article 6B (extraction), and Article 6C (dispatch).
    *   **问题解析：** 第 6A 篇（论点）、第 6B 篇（提取）和第 6C 篇（分发）。
*   **Retrieval:** Article 7A (retrieval as filtering), Article 7B (anchor detection), and Article 7C (the LLM arbiter).
    *   **检索：** 第 7A 篇（作为过滤的检索）、第 7B 篇（锚点检测）和第 7C 篇（LLM 仲裁者）。
*   **Generation:** the answer contract, prompt assembly, and validation (Articles 8A to 8C, links to come).
    *   **生成：** 答案契约、提示词组装和验证（第 8A 至 8C 篇，链接待补充）。
