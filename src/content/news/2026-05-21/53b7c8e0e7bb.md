### Why run coding agents
### 为什么要运行编程智能体

First of all, I need to cover why you should run coding agents on your computer. If you’re working with programming, it should be pretty self-explanatory. Using coding agents to program instead of manually programming is just way more efficient. It can’t even compare to writing the code yourself or even to tab completions. Having agents write all the code for you is now very much possible, given how powerful the latest LLMs have become, and it’s simply far more efficient at implementing code than humans can ever be.
首先，我需要说明为什么要让编程智能体在你的电脑上运行。如果你从事编程工作，这应该是不言自明的。使用编程智能体来编程，而不是手动编写代码，效率要高得多。这甚至无法与你自己写代码或使用自动补全相提并论。鉴于最新的大语言模型（LLM）已经变得如此强大，让智能体为你编写所有代码现在完全可行，而且在实现代码方面，它们比人类高效得多。

However, typically, some dangers of running coding agents are pointed out, usually mentioning the two points below: It’s scary not to look and verify the code yourself or perform a human review. The agents can perform unsafe actions, and you need to make sure they don’t do anything they shouldn’t do. In this article, I will cover why I strongly disagree with these two points and how you can ensure you run coding agents safely on your computer in your environment.
然而，通常人们会指出运行编程智能体的一些危险，主要涉及以下两点：不亲自查看和验证代码或进行人工审查令人担忧；智能体可能会执行不安全的操作，你需要确保它们不会做不该做的事。在本文中，我将阐述为什么我强烈不同意这两点，以及你如何确保在自己的电脑环境中安全地运行编程智能体。

### Why you don’t need to manually review all code
### 为什么你不需要手动审查所有代码

First of all, I want to answer the first question about how many people think that all code should be manually reviewed. I strongly disagree with this argument because coding agents have become so powerful now that they write better code, or at least equally good code, than a lot of humans. Yes, the code might not be perfect, adhering to every formatting rule or best practices. However, the code that coding agents produce is typically very functional, and the agents are extremely good at discovering bugs. I’d argue that coding agents in many cases can produce better code than humans because they’re able to take in much more of the context around the repository and thus avoid a lot of bugs.
首先，我想回答第一个问题，即关于许多人认为所有代码都应经过人工审查的观点。我强烈反对这一论点，因为编程智能体现在已经变得非常强大，它们编写的代码比许多人类编写的更好，或者至少同样出色。是的，代码可能并不完美，未必完全符合每一条格式规则或最佳实践。然而，编程智能体生成的代码通常功能性很强，而且智能体在发现 Bug 方面非常出色。我认为在许多情况下，编程智能体能产出比人类更好的代码，因为它们能够获取代码库中更多的上下文信息，从而避免许多 Bug。

If you have a decently organized code repository with a lot of details in your agents.md files and other markdown files, and you let other coding agents perform code reviews on the code you produce, I don’t think you need to manually review your code. Of course, there are cases where you are touching very sensitive code that you know can lead to bugs. In these cases, you should naturally perform a human review, but for most of the code you produce, I don’t think a human review is necessary anymore.
如果你有一个组织良好的代码库，并在 `agents.md` 文件和其他 Markdown 文件中提供了大量细节，并且让其他编程智能体对你生成的代码进行审查，我认为你就不需要手动审查代码了。当然，在某些情况下，如果你触及了非常敏感的代码，且知道这可能导致 Bug，那么自然应该进行人工审查。但对于你编写的大多数代码，我认为人工审查已不再必要。

### Ensuring agents don’t perform unsafe actions
### 确保智能体不执行不安全的操作

The second point mentioned above was that agents can perform unsafe actions, and you need to make sure they don’t do something they shouldn’t do. It’s true that if you give your coding agent a lot of permissions, they can obviously perform unsafe actions. For example, if you give them wide AWS permissions, they can, of course, update your infrastructure. However, in my experience, I have two counterarguments:
上面提到的第二点是，智能体可能会执行不安全的操作，你需要确保它们不会做不该做的事。确实，如果你赋予编程智能体大量权限，它们显然可以执行不安全的操作。例如，如果你赋予它们广泛的 AWS 权限，它们当然可以更新你的基础设施。然而，根据我的经验，我有两个反驳观点：

The coding agents very, very rarely actually make these mistakes. I find that Claude Code and Codex almost always inform me before performing an irreversible decision, or at least a non-easily reversible decision. They don’t simply make serious mistakes that are very hard to reverse.
编程智能体极少真正犯下这些错误。我发现 Claude Code 和 Codex 在执行不可逆（或至少是不易逆转）的决策之前，几乎总是会通知我。它们不会轻易犯下难以挽回的严重错误。

If a coding agent is able to perform a destructive action, such as deleting a production database or equivalent, I’d argue the problem is not in the coding agent, but in the way you structure your code. An AI or a human should not be able to fully delete a production table, obviously. If so, you’ve first of all given them way too wide permissions. Technically, a human could make that mistake as well. And secondly, you’ve not structured your code well enough. For example, if a table is deleted, you should make sure you have a backup. I don’t think the argument that agents perform unsafe actions is really true. The coding agents basically don’t make these irreversible mistakes, and if such a destructive irreversible mistake is possible, such as deleting a production database, then you need to update your code infrastructure to make sure that it’s not possible.
如果编程智能体能够执行破坏性操作（例如删除生产数据库或类似操作），我认为问题不在于智能体，而在于你构建代码的方式。显然，无论是 AI 还是人类，都不应该能够轻易完全删除生产表。如果能做到，首先说明你赋予了它们过大的权限（从技术上讲，人类也可能犯同样的错误）；其次，说明你的代码结构不够完善。例如，如果表被删除了，你应该确保有备份。我不认为“智能体执行不安全操作”这一论点是成立的。编程智能体基本上不会犯这种不可逆的错误；如果确实存在这种破坏性的可能性，那么你需要更新你的代码基础设施，以确保这种情况无法发生。

### How many permissions to give your agents
### 应该赋予智能体多少权限

Now, let’s cover how many permissions you should be providing your coding agents. Whenever I run my agents, I run Claude with `--dangerously-skip-permissions` and Codex in YOLO mode. This means I ask it to basically never ask me for permission when performing an action. The only exception I have to this is when running the `rm` command, for example, deleting recursively like below: `rm -rf`. When running this command, the agents have to ask me for permission because I know it’s a destructive action on my computer that is not reversible (i.e., I cannot recover files that are deleted with this command).
现在，让我们谈谈你应该赋予编程智能体多少权限。每当我运行智能体时，我都会使用 `--dangerously-skip-permissions` 参数运行 Claude，并让 Codex 在 YOLO 模式下运行。这意味着我要求它们在执行操作时基本不需要征求我的许可。唯一的例外是运行 `rm` 命令时，例如递归删除：`rm -rf`。当运行此命令时，智能体必须征求我的许可，因为我知道这是电脑上一种不可逆的破坏性操作（即我无法恢复被此命令删除的文件）。
