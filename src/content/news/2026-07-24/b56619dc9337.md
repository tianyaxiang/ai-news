# Most RAG Hallucinations Are Extraction Errors: Seven Patterns for a Typed Generation Contract

**Large Language Model Most RAG Hallucinations Are Extraction Errors: Seven Patterns for a Typed Generation Contract Enterprise Document Intelligence [Vol.1 #8ter] – Naming the RAG error correctly matters: model reads the context, so a wrong answer is an extraction error, not a hallucination. Seven typed-contract patterns keep the generation brick honest, with a decomposition rule for small models Kezhan Shi Jul 23, 2026 11 min read**

大型语言模型中，大多数 RAG（检索增强生成）幻觉其实是提取错误：七种类型化生成契约模式。企业文档智能 [第1卷 #8ter] —— 正确命名 RAG 错误至关重要：模型读取了上下文，因此错误的答案是提取错误，而非幻觉。七种类型化契约模式确保了生成模块的可靠性，并为小型模型提供了分解规则。Kezhan Shi，2026年7月23日，阅读时长11分钟。

**Not every question wants the same shape of answer. One wants a single number with a citation, another a list with one item per row, a third a yes or no with a caveat. Force them all through one generation call and each shape degrades the others. The fix is a small set of generation patterns, one per answer shape.**

并非每个问题都需要相同形式的答案。有的需要带引用的单一数字，有的需要每行一项的列表，有的则需要带限定条件的“是”或“否”。如果强行通过单一的生成调用来处理所有这些需求，每种形式的质量都会受到影响。解决方案是采用一套小型生成模式，每种答案形式对应一种模式。

**This article is a companion to Enterprise Document Intelligence, the series whose philosophy is laid out in Amplify the Expert. It zooms in on brick 4 (generation) of the four-brick architecture, and pushes back on the vocabulary teams reach for when a RAG answer is wrong. Hallucination, in the strict sense, is a fabrication from the model’s parametric memory: the LLM invents a fact without any grounding in the input.**

本文是《企业文档智能》系列的配套文章，该系列的理念在《Amplify the Expert》一文中已有阐述。本文聚焦于四模块架构中的第4块（生成），并反思了团队在 RAG 答案出错时所使用的术语。从严格意义上讲，幻觉是指模型从其参数化记忆中捏造事实：即大语言模型在没有任何输入依据的情况下编造事实。

**In RAG the model reads the context; when the answer is wrong, the cause is upstream, in the extraction chain (parsing, question, retrieval, or the generation contract itself). Calling every failure a hallucination closes the debate on where to act. Naming the real cause opens it back up.**

在 RAG 中，模型会读取上下文；当答案错误时，原因在于上游的提取链（解析、提问、检索或生成契约本身）。将所有失败都称为“幻觉”会阻碍对改进方向的探讨。而指出真正的原因则能重新开启讨论。

**The mainstream story has generation as send retrieved chunks + question to the LLM, get an answer string back. We reject this whole frame. The answer is not a string. It is a typed Pydantic object with citations, fidelity flags, and self-assessment fields. The LLM is a function, not an oracle. The schema is the contract. The validator runs before the user sees anything.**

主流观点认为生成过程就是：将检索到的片段 + 问题发送给大语言模型，然后得到一个字符串答案。我们拒绝这种框架。答案不是字符串，而是一个类型化的 Pydantic 对象，包含引用、保真度标记和自我评估字段。大语言模型是一个函数，而不是神谕。模式即契约。验证器会在用户看到任何内容之前运行。

**Pattern 1 – The LLM is a function, not an oracle. The answer schema is a contract: structured retrieval in, a typed Pydantic object out, with citations, fidelity flags, and pipeline-feedback fields, never “the answer as a string”. Everything the generation brick does is in service of filling that contract, field by field.**

模式 1 —— 大语言模型是函数，而非神谕。答案模式即契约：输入结构化检索内容，输出类型化的 Pydantic 对象，包含引用、保真度标记和流水线反馈字段，绝非“字符串形式的答案”。生成模块所做的一切都是为了逐字段填充该契约。

**Take “what is the premium amount?”. The naive baseline returns a sentence: "The premium is $124 per month, per the contract.". To use that downstream you have to parse the number back out of the prose, and nothing tells you where it came from or how sure the model was.**

以“保费金额是多少？”为例。简单的基准方法会返回一个句子：“根据合同，保费为每月 124 美元。”要在下游使用该结果，你必须从文本中重新解析出数字，而且没有任何信息告诉你这个数字的来源，或者模型的置信度如何。

**The series returns a row instead: Amount(value=124.0, currency="USD", unit="month", evidence=[Span(line_start=42, line_end=42, quote="premium of $124 / month")], answer_found=True, confidence=0.95). The value is already typed, the evidence points at the exact line, and the self-assessment fields travel with it. The caller reads structured values, not text it has to re-read.**

本系列则返回一行数据：Amount(value=124.0, currency="USD", unit="month", evidence=[Span(line_start=42, line_end=42, quote="premium of $124 / month")], answer_found=True, confidence=0.95)。该值已类型化，证据指向确切行，且自我评估字段随之携带。调用者读取的是结构化数值，而非需要重新阅读的文本。

**Pattern 2 – Extract typed values, never compute. Ask for Amount(value, currency, unit) and let Python compare with a visible exchange rate; computation inside the model erases the audit trail and silently does conversions no human can replay.**

模式 2 —— 提取类型化数值，绝不进行计算。请求 Amount(value, currency, unit)，并让 Python 使用可见的汇率进行比较；模型内部的计算会抹除审计追踪，并悄悄进行人类无法复现的转换。

**Pattern 3 – Completeness from structure, not self-rating. The LLM inside the retrieval scope cannot see whether the next page continues a list; retrieval pulls one extra page and the pipeline checks for a section boundary, so completeness is deterministic and grounded in structure.**

模式 3 —— 完整性源于结构，而非自我评估。检索范围内的大语言模型无法判断下一页是否延续了列表；检索时多拉取一页，由流水线检查是否存在章节边界，因此完整性是确定性的，且基于结构。
