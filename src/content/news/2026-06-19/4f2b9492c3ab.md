### 1. The fields the parser decides with the document profile
### 1. 解析器结合文档概况所决定的字段

Take a one-page CV and the question “what is the name?”. Question parsing on its own returns keywords=["name"] and retrieval looks for the literal word name in the file. A CV never says name. Nothing matches, the answer comes back empty.
以一份单页简历和问题“名字是什么？”为例。仅进行问题解析会返回 keywords=["name"]，检索过程会在文件中查找字面意义上的“name”一词。简历中通常不会出现“name”这个词。结果将无法匹配，返回空答案。

A human would not answer that question with nothing else to go on: they would glance at the document first, see a resume, and read name as a request for the candidate’s name. The parser needs the same starting point. As soon as it sees that the document is a resume and that the candidate’s name sits at the top of page 1, the keyword name resolves to a person, not a literal token to grep for.
人类在没有其他信息的情况下不会这样回答问题：他们会先扫视文档，识别出这是一份简历，并将“名字”理解为对候选人姓名的请求。解析器也需要同样的起点。一旦它识别出文档是简历，且候选人姓名位于第 1 页顶部，关键词“name”就会被解析为“人名”，而不是需要搜索的字面标记。

Two more column families get filled by the dispatcher right after question parsing returns, using the parsed question PLUS the document’s profile. In the shipped code, that profile is the semantic zone of parsing_summary, the doc-level dict produced by the parser. It carries doc_type (resume, contract, invoice, …), typical_fields (the fields questions about this kind of document usually ask about), and a short LLM-written summary that fits at the head of a system prompt. The dispatcher reads these three fields and uses them to set chunk strategy and answer context. They land on the same question_df row, so retrieval and generation see one record.
在问题解析返回后，分发器会立即利用已解析的问题加上文档概况，填充另外两个列族。在交付的代码中，该概况是 parsing_summary 的语义区，即解析器生成的文档级字典。它包含 doc_type（简历、合同、发票等）、typical_fields（此类文档通常涉及的字段）以及一段简短的、由 LLM 编写的、适合放在系统提示词开头的摘要。分发器读取这三个字段，并利用它们设置分块策略和答案上下文。它们最终落在同一个 question_df 行中，因此检索和生成模块只需查看一条记录。

### 1.1 Dispatch: how much context, which chunk strategy, which model
### 1.1 分发：多少上下文、何种分块策略、使用哪个模型

Once the parser has the literal info, three more decisions follow: how much surrounding text to read and return, whether to combine the top-k chunks in one LLM call or feed them in sequence, and which model to call. All three are defaults the project can override per concept, per answer type, or per question. The cascade is the same every time: concept-level override > shape/type default > project fallback.
一旦解析器获取了字面信息，接下来还需要做出三个决策：读取并返回多少周围文本、是将 top-k 个分块合并在一次 LLM 调用中还是按顺序输入、以及调用哪个模型。这三者均为默认设置，项目可以针对特定概念、答案类型或问题进行覆盖。其优先级顺序始终为：概念级覆盖 > 形状/类型默认值 > 项目回退值。

How much context to read and return: Three fields on StructuralHints carry this:
关于读取并返回多少上下文：StructuralHints 中的三个字段负责此项：

*   **detection_context**: the granularity of the regex confirmation zone (Article 6_b, answer shape and answer type). "line" for amounts and dates, "paragraph" for narrative.
    **detection_context**：正则表达式确认区域的粒度（第 6_b 篇，答案形状和答案类型）。金额和日期使用“行”，叙述性内容使用“段落”。
*   **answer_context**: how much surrounding text the generator receives. "line" for a single value, "paragraph" for an explanation, "page" for a recap, "section" for a topic, "chapter" or "document" for a broad summary.
    **answer_context**：生成器接收的周围文本量。单个值使用“行”，解释使用“段落”，回顾使用“页面”，主题使用“章节”，广泛摘要使用“章”或“文档”。
*   **needs_summary**: True when the answer spans more than fits in a verbatim quote.
    **needs_summary**：当答案长度超过逐字引用范围时为 True。

Defaults come from the same two satellite tables we already met: answer_shapes_df for shape-level defaults (the default_answer_context column), concepts_df for concept-level overrides.
默认值来自我们之前提到的两个辅助表：answer_shapes_df 用于形状级默认值（default_answer_context 列），concepts_df 用于概念级覆盖。

“What is the annual premium?” is (single, amount) with no specific concept; it gets answer_context = "line" from answer_shapes_df. “What are the exclusions of this contract?” matches the exclusions concept; it gets answer_context = "chapter" from concepts_df, which overrides the listing-shape default of "section".
“年度保费是多少？”属于（单个，金额）类型，没有特定概念；它从 answer_shapes_df 获取 answer_context = "line"。而“本合同的除外责任有哪些？”匹配“除外责任”概念；它从 concepts_df 获取 answer_context = "chapter"，这覆盖了列表形状默认的“section”。

Why shape AND length matter beyond retrieval. The same hints feed the generation brick’s combined-vs-sequential dispatch. When the answer is a single fact in one chunk (an amount, a date, an IBAN, a yes/no), generation calls the LLM sequentially, chunk by chunk in retrieval rank order, and stops as soon as answer_found=True and complete_answer_found=True. That saves ~⅔ of the input tokens at k=3 when the answer is in the top-1 chunk.
为什么形状和长度在检索之外同样重要？这些提示信息同样会驱动生成模块的“合并 vs 顺序”分发逻辑。当答案是单个分块中的事实（金额、日期、IBAN、是/否）时，生成模块会按检索排名顺序逐个分块调用 LLM，并在 answer_found=True 且 complete_answer_found=True 时立即停止。当答案位于 top-1 分块时，这在 k=3 的情况下可节省约 2/3 的输入 Token。

When the answer is synthesised across passages (a list of exclusions scattered across pages, a definition plus its footnote, a comparison), generation combines all k chunks into one call. The decision is made once, here, by the parser; retrieval and generation just execute. At enterprise scale (millions of documents × top-k chunks per question), the per-question saving compounds into the bulk of the LLM bill.
当答案需要跨段落合成时（如分散在各页的除外责任列表、定义及其脚注、比较分析），生成模块会将所有 k 个分块合并为一次调用。该决策由解析器在此处一次性做出；检索和生成模块只需执行即可。在企业规模下（数百万文档 × 每个问题 top-k 个分块），每个问题的节省量将累积成 LLM 账单中的大头。

The strategy itself lives at the top level of ParsedQuestion.chunk_strategy, and the default value comes from the satellite tables, with this resolution order:
该策略本身位于 ParsedQuestion.chunk_strategy 的顶层，默认值来自辅助表，解析顺序如下：

```python
def resolve_chunk_strategy(
    answer_shape: str,
    matched_concept: str | None,
    answer_shapes_df: pd.DataFrame,
    concepts_df: pd.DataFrame,
) -> Literal["combined", "sequential"]:
    """Concept-level override > answer-shape default > hard default."""
    if matched_concept is not None:
        row = concepts_df[concepts_df["concept"] == matched_concept]
        if not row.empty and pd.notna(row.iloc[0].get("default_chunk_strategy")):
            return row.iloc[0]["default_chunk_strategy"]
    
    row = answer_shapes_df[answer_shapes_df["shape"] == answer_shape]
    if not row.empty:
        return row.iloc[0]["default_chunk_strategy"]
    
    return "combined"
```
