Article 1 (minimal RAG) showed the basic pattern: take the user’s question, send it to an LLM, and get an answer back. The series pushed it a step further with prompt engineering, asking the LLM to return a JSON object with the answer plus the source passages it used. With a well-parsed document and structured output, finding where the answer comes from is no longer the hard part. That’s the starting point. What’s left is the question itself, and that’s where most of the work in a real RAG pipeline ends up living.
第 1 篇（极简 RAG）展示了基本模式：获取用户问题，发送给大模型（LLM），并获得回答。本系列通过提示词工程（Prompt Engineering）更进一步，要求 LLM 返回一个包含答案及其所用源段落的 JSON 对象。有了解析良好的文档和结构化输出，找到答案来源已不再是难点。这只是起点。剩下的核心问题在于“问题本身”，而这正是真实 RAG 流水线中大部分工作所在。

Three levels of “working the question” show up in enterprise projects, and they tend to arrive in this order.
在企业项目中，对“问题处理”通常有三个层级，且往往按以下顺序出现：

1. **Natural questions from users.** A user types whatever comes to mind: “What’s the cap on liability?”, “Tell me about the exclusions”, “How does this compare to last year’s policy?” The system has to make sense of a string it didn’t author, in vocabulary that isn’t necessarily the document’s. This is the case most demos show, and the one with the most variance on the input side.
1. **用户的自然提问。** 用户输入任何想到的内容：“责任上限是多少？”、“告诉我关于除外责任的信息”、“这与去年的保单相比如何？”系统必须理解一段非其所写的字符串，且词汇未必与文档一致。这是大多数演示中展示的情况，也是输入端差异性最大的场景。

2. **Templates the developer writes ahead of time.** Pretty quickly, the team notices that the same questions come back over and over. For every contract, someone wants to know who the client is, who the insurer is, the annual premium, the guarantees, the exclusions, the renewal date. Rather than wait for users to type those one by one, the developer writes them once, jointly with the business team that owns the documents, and runs them across the whole corpus. The output is a JSON record per document. In practice this is what most companies build first; free-form chat tends to come later, on top of this base.
2. **开发者预先编写的模板。** 团队很快会发现，同样的问题会被反复问及。对于每一份合同，人们都想知道客户是谁、保险人是谁、年保费、担保条款、除外责任、续保日期等。与其等待用户逐一输入，开发者不如与拥有文档的业务团队合作，预先编写好这些问题，并在整个语料库上运行。输出结果是每个文档对应一条 JSON 记录。在实践中，这是大多数公司首先构建的内容；自由形式的聊天通常是在此基础上后续添加的。

3. **Helping users formulate.** A pattern that shows up once a system is in front of real users: business teams know what they’re after, but have trouble phrasing it tightly. “What’s the cap?” leaves a lot open. The system can step in and ask back: “do you mean liability, indemnification, deductible, or premium?” Same machinery as the other two levels (a typed object with named fields), but now the empty fields drive a small dialogue with the user.
3. **辅助用户表述。** 当系统面对真实用户时会出现这种模式：业务团队知道他们想要什么，但难以精确表述。“上限是多少？”留下了太多歧义。系统可以介入并反问：“您是指责任上限、赔偿额、免赔额还是保费？”这与前两个层级使用相同的机制（带有命名字段的类型化对象），但现在空缺的字段会驱动系统与用户进行简短的对话。

The three levels share the same machinery: turn the question into a structured object, then route the parts to retrieval and to generation. What changes is who fills the structure (the user, the developer, or both in dialogue) and what the system does when a field is empty (proceed with defaults, fail loudly, or ask back).
这三个层级共享相同的机制：将问题转化为结构化对象，然后将各部分路由至检索和生成环节。变化的是谁来填充该结构（用户、开发者，或在对话中双方共同填充），以及当字段为空时系统如何处理（使用默认值继续、报错提示，或进行反问）。

A user types one string into the chat box. “What is the maximum coverage amount? Don’t confuse it with the deductible, they’re often listed together.” There’s a lot in that string. A topic (maximum coverage amount). An expected shape (an amount, a number). A negative cue (don’t confuse with deductible). A hint about how the document presents the answer (often listed together).
用户在聊天框中输入一个字符串：“最高承保金额是多少？别把它和免赔额搞混了，它们经常列在一起。”这段字符串包含了很多信息：一个主题（最高承保金额）、预期的格式（金额、数字）、负面提示（不要与免赔额混淆）、关于文档如何呈现答案的线索（经常列在一起）。

If you hand the whole string to retrieval, none of those parts land where they should: the embedding pulls deductible-bearing lines closer, the format hint never reaches generation, the disambiguation gets either embedded with the rest or stripped at preprocessing. The result is a confident wrong answer (the deductible’s number, not the coverage’s), and a retriever that no observability dashboard can debug.
如果你把整个字符串直接交给检索环节，这些部分都无法发挥应有的作用：嵌入模型会将包含“免赔额”的行拉得更近，格式提示无法传达给生成环节，消歧信息要么被混入嵌入向量中，要么在预处理时被剔除。结果就是得到一个自信但错误的答案（给出了免赔额而非承保金额），且检索器在任何可观测性仪表板中都无法调试。

The fix is to parse the question first. Turn the noisy user string into a typed, structured brief that downstream bricks can act on. Then split the brief in two: the retrieval brick reads what it can act on (topic, rewrites, scope), the generation brick reads what it can act on (the original wording, the format constraint, the disambiguation). Neither brick gets confused by the other’s signal. Both stay focused on what they do best.
解决方法是先解析问题。将嘈杂的用户字符串转化为下游模块可以处理的类型化、结构化摘要。然后将摘要一分为二：检索模块读取它能处理的部分（主题、重写、范围），生成模块读取它能处理的部分（原始措辞、格式约束、消歧信息）。这样，两个模块都不会被对方的信号干扰，都能专注于各自最擅长的工作。

**1. Question parsing mirrors document parsing**
**1. 问题解析与文档解析相呼应**

**1.1 Same approach: a relational set of tables**
**1.1 相同的方法：关系型表集合**

Document parsing produces a relational set: line_df with one row per text line, page_df with one row per page, toc_df with one row per TOC entry, plus a few satellites. Question parsing has the same goal: turn the unstructured input into structured form before the next steps run on it. The artifact is the same kind of thing: a relational set.
文档解析产生一个关系型集合：line_df（每行文本一行）、page_df（每页一行）、toc_df（每个目录条目一行），以及一些附属表。问题解析的目标相同：在后续步骤运行之前，将非结构化输入转化为结构化形式。其产物也是同类事物：一个关系型集合。

The shape diverges in one obvious way. A document fills line_df with hundreds or thousands of rows. A question is a single string, so it fills one row in a question_df table. The columns of that row are what the parser computes: the spell-corrected text, the extracted keywords, the type of answer expected, the pages or sections the user mentioned, the decomposition pattern, the activations the dispatcher should turn on. Adding a parsing capability is adding a column.
其形态在一个明显的方面有所不同。文档会向 line_df 填充成百上千行数据。而问题是一个单一字符串，因此它在 question_df 表中只填充一行。该行的列是解析器计算得出的：拼写纠正后的文本、提取的关键词、预期的答案类型、用户提到的页面或章节、分解模式、分发器应开启的激活项。增加一项解析能力，就是增加一列。

The other half of the relational picture is satellite tables the question row links into. They’re as open-ended as the columns themselves: a project starts with the ones it needs, and adds others as new cases push for them. The most common one is the project’s expert keyword dictionary.
关系型图景的另一半是问题行所链接的附属表。它们和列一样具有开放性：项目从所需的表开始，并随着新案例的需求而增加其他表。最常见的是项目的专家关键词词典。
