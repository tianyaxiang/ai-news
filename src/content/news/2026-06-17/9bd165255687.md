# 大模型回退机制破坏了智能体流水线——我构建了缺失的恢复层

Large Language Model LLM Fallbacks Break Agent Pipelines — I Built the Missing Recovery Layer Standard model swapping forces your production AI to accept corrupted data. Here is the zero-dependency, state-aware recovery layer I built to preserve schema integrity during failovers.
大语言模型（LLM）的回退机制正在破坏智能体流水线——我构建了缺失的恢复层。标准的模型切换方式迫使你的生产级 AI 接受损坏的数据。以下是我构建的零依赖、具备状态感知能力的恢复层，旨在确保故障转移期间的模式完整性。

### The Moment Everything Broke
### 一切崩溃的时刻

I was building a three-agent pipeline for EmiTechLogic: a Planner, an Executor, and a Validator running sequentially, each feeding structured JSON output to the next. The pipeline worked fine in testing. Small loads, clean responses, no surprises. Then I ran it under realistic conditions.
我当时正在为 EmiTechLogic 构建一个三智能体流水线：规划者（Planner）、执行者（Executor）和验证者（Validator），它们按顺序运行，每个智能体都将结构化的 JSON 输出传递给下一个。流水线在测试中运行良好，负载小、响应干净、没有意外。但当我将其置于真实条件下运行时，问题出现了。

Step one, the Planner, finished. Step two, the Executor, hit a 429 rate limit midway through. My basic retry loop caught the error, swapped to a fallback model, and kept running. The pipeline reported 100% completion. No exceptions thrown. No error logs. But when I checked the downstream output, the confidence key was missing. The result field was just a string: “incomplete – schema mismatch during swap.” The Validator received structurally broken input and had no way to know it. The pipeline finished on paper, but the data was useless.
第一步，规划者完成了。第二步，执行者在中途遇到了 429 速率限制。我基础的重试循环捕获了错误，切换到备用模型并继续运行。流水线报告 100% 完成，没有抛出异常，也没有错误日志。但当我检查下游输出时，发现 `confidence` 键丢失了。结果字段仅仅是一个字符串：“incomplete – schema mismatch during swap”（不完整——切换期间模式不匹配）。验证者接收到了结构损坏的输入，却无法察觉。流水线在纸面上完成了，但数据毫无用处。

That is the failure mode this article is about. Not the 429 itself, and not the retry loop. The issue is what happens when you hand a fallback model an unchanged payload formatted for a different engine entirely.
这就是本文要讨论的故障模式。不是 429 错误本身，也不是重试循环。问题在于，当你将一个为完全不同的引擎格式化的负载，原封不动地交给备用模型时，会发生什么。

### Why This Failure Is So Hard to See
### 为什么这种故障如此难以察觉

Standard monitoring dashboards hide this bug because they only track process completion. They check if the API returned a 200 and if the thread exited cleanly. If the script finishes, the dashboard turns green. For multi-agent systems, uptime is the wrong metric. The metric that matters is schema integrity. A pipeline that silently completes with corrupted fields is often worse than a hard crash. A crash forces an immediate fix, while silent data corruption slips directly into your database unnoticed.
标准的监控仪表盘掩盖了这个 Bug，因为它们只追踪进程完成情况。它们检查 API 是否返回 200，以及线程是否正常退出。如果脚本运行结束，仪表盘就会显示绿色。对于多智能体系统，运行时间（uptime）是一个错误的指标。真正重要的指标是模式完整性。一个静默完成但字段损坏的流水线，往往比直接崩溃更糟糕。崩溃会迫使你立即修复，而静默的数据损坏会直接溜进你的数据库而不被察觉。

These agents are tightly coupled. The Executor expects the Planner’s exact JSON keys, and the Validator expects the Executor’s keys. When a model swap breaks the structure at step two, that step doesn’t throw an error. It just passes the malformed data down the line, ruining the final output somewhere downstream where you aren’t looking. Rate limiting isn’t a basic network infrastructure issue. It is a data integrity problem.
这些智能体是紧密耦合的。执行者期望得到规划者精确的 JSON 键，验证者期望得到执行者的键。当模型切换在第二步破坏了结构时，该步骤不会抛出错误，它只是将格式错误的数据向下传递，在下游某个你没注意到的地方毁掉了最终输出。速率限制不是一个基础的网络基础设施问题，而是一个数据完整性问题。

### The Anatomy of a Silent Pipeline Failure
### 静默流水线故障的剖析

The failure mechanism is quiet but destructive. API contracts are completely inconsistent across different model tiers. A premium model enforces strict JSON mode and uses a dedicated system prompt array. A cheaper fallback tier might not support an isolated system field at all, forcing you to merge instructions directly into the user text. It also rarely guarantees structured JSON outputs.
这种故障机制虽然安静，但具有破坏性。不同模型层级之间的 API 契约完全不一致。高级模型强制执行严格的 JSON 模式并使用专门的系统提示词数组。而更便宜的回退层级可能根本不支持独立的系统字段，迫使你将指令直接合并到用户文本中。它也很少能保证输出结构化的 JSON。

When a basic router catches a 429 and swaps the model ID, it forwards the original request payload unchanged. The fallback model gets a configuration it can’t parse. The network request succeeds because the API technically returned text. No exception is thrown. The pipeline keeps moving, but the data structure is already ruined. The next agent just gets raw text or missing keys instead of valid JSON.
当基础路由器捕获到 429 并切换模型 ID 时，它会原封不动地转发原始请求负载。备用模型收到了它无法解析的配置。网络请求成功了，因为 API 在技术上确实返回了文本。没有抛出异常，流水线继续运行，但数据结构已经损坏。下一个智能体得到的只是原始文本或缺失的键，而不是有效的 JSON。

### Building a Recovery Layer That Actually Understands Context
### 构建一个真正理解上下文的恢复层

I split the logic into four parts. Each one has a single job and nothing else. The first realization: not all failures are the same. A basic router treats every API error as a trigger to swap or retry. That logic fails instantly on context overflows or billing issues. You have to separate the root causes.
我将逻辑拆分为四个部分，每个部分只负责一项工作。第一个认识是：并非所有故障都一样。基础路由器将每个 API 错误都视为切换或重试的触发器。这种逻辑在遇到上下文溢出或计费问题时会立即失效。你必须区分根本原因。

A 429 means the model is temporarily throttled, so you swap and retry elsewhere. A context overflow means the prompt itself is too big, so a retry is just a waste of tokens because the payload needs to be trimmed first. A billing quota drop means the entire provider is dead for the session, so burning retries against it is pointless. The detector handles this by parsing the raw error string against specific pattern lists. Instead of a generic crash, it returns a typed `ThrottleEvent` containing a clean reason code and a backoff window tied to the specific error.
429 意味着模型被暂时限流，所以你应该切换并重试。上下文溢出意味着提示词本身太大，重试只是浪费 Token，因为必须先修剪负载。计费配额耗尽意味着整个提供商在本次会话中已不可用，因此对其进行重试毫无意义。检测器通过将原始错误字符串与特定模式列表进行匹配来处理此问题。它不会导致通用崩溃，而是返回一个类型化的 `ThrottleEvent`，其中包含清晰的原因代码和与特定错误关联的退避窗口。

The detector tracks provider windows using `time.monotonic()` for cooldown decay. It keeps track of remaining backoff times and monitors the request rate over a rolling 60-second window. Every routing attempt calls `is_throttled()` first. If a provider is in backoff, the router skips it entirely.
检测器使用 `time.monotonic()` 来追踪提供商的冷却衰减。它记录剩余的退避时间，并监控 60 秒滚动窗口内的请求速率。每次路由尝试都会先调用 `is_throttled()`。如果提供商处于退避状态，路由器会直接跳过它。
