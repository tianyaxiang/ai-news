# RAG Was Always a Temporary Workaround. What is Next?
# RAG 始终只是权宜之计，未来何去何从？

**Why the future of AI memory relies on persistent neural state, not vector databases.**
**为什么 AI 记忆的未来在于持久化的神经状态，而非向量数据库。**

### 1. The Great Translation Absurdity
### 1. 伟大的翻译荒谬论

Look closely at your favourite RAG system. Very closely. Somewhere inside that meticulously architected pipeline, a language model is producing rich, high-dimensional hidden states. Those states get compressed down into a string of characters. That string gets re-encoded, by a different neural network, into a different high-dimensional space. That vector gets stored. Later, another vector gets compared against it. The winning strings get pulled out, glued together, and handed to a third model that laboriously rebuilds a hidden state from those characters.
仔细观察你最常用的 RAG 系统。再仔细一点。在那精心构建的流水线深处，语言模型正在生成丰富的高维隐藏状态。这些状态被压缩成一串字符。随后，这串字符被另一个神经网络重新编码，进入另一个高维空间。该向量被存储起来。稍后，另一个向量会与它进行比对。胜出的字符串被提取出来、拼接在一起，并交给第三个模型，由它费力地从这些字符中重建隐藏状态。

We labelled this “memory.” A more honest name would be: a very elaborate, high-latency game of telephone in which every player is a neural network. Two amber boxes doing the thinking. Five grey boxes doing the paperwork the two amber boxes shouldn’t have needed in the first place.
我们将此称为“记忆”。一个更诚实的名称应该是：一场极其复杂、高延迟的“传声筒”游戏，其中每个参与者都是一个神经网络。两个“琥珀色盒子”负责思考，而五个“灰色盒子”则在处理那些本不该存在的文书工作。

The full chain, drawn plainly: Hidden State → Generate Text → Embed Text → Store Vector → Retrieve Vector → Append Text → Recompute Hidden State.
完整的链条清晰可见：隐藏状态 → 生成文本 → 嵌入文本 → 存储向量 → 检索向量 → 追加文本 → 重新计算隐藏状态。

Read that arrow chain twice. Two of the seven stages are neural-native. The other five exist only because we cannot yet persist the neural state itself, so we built an entire industry to reconstruct it from text every single time we needed it back. Vector databases, embedding models, rerankers, chunking heuristics, retrieval evaluators — an entire ecosystem to route around a missing feature.
把这个箭头链读两遍。七个阶段中只有两个是神经原生的。其余五个之所以存在，仅仅是因为我们还无法持久化神经状态本身，因此我们建立了一个庞大的产业，只为在每次需要时从文本中重建它。向量数据库、嵌入模型、重排序器、分块启发式算法、检索评估器——整个生态系统都是为了绕过一个缺失的功能而构建的。

### 2. The Context Window Illusion
### 2. 上下文窗口的幻觉

I can safely assume that by now the standard objection is warming up in the reader’s head. “None of this matters. Just use a two-million-token context window and dump everything in.” Fair objection. Also, no. Bigger context windows solve capacity. They do not solve portability. They do not solve persistence.
我可以肯定，读者的脑海中现在已经浮现出标准的反对意见：“这些都不重要。只要用两百万 token 的上下文窗口，把所有东西塞进去就行了。”这是一个合理的反对意见，但答案是否定的。更大的上下文窗口解决了容量问题，但它无法解决可移植性问题，也无法解决持久化问题。

Consider one autonomous agent handing off a task to another. Or an edge device (e.g., a drone, a phone, a robot, a network node) moving between compute clusters. Or a multi-agent pipeline in which the router, the tool caller, the safety filter, and the finaliser all live in different processes on different machines. In every one of those settings, the unit of transfer between two computers cannot practically be a two-million-token prompt.
试想一个自主智能体将任务移交给另一个智能体，或者一个边缘设备（如无人机、手机、机器人、网络节点）在计算集群之间移动。又或者是一个多智能体流水线，其中路由、工具调用、安全过滤和最终处理程序分别运行在不同机器的不同进程中。在所有这些场景下，两台计算机之间的传输单元实际上不可能是一个两百万 token 的提示词。

The bandwidth cost is punishing. The re-tokenisation is wasted work. And the receiver still has to re-read the whole transcript — a full prefill pass over every token — to reconstruct any semblance of the sender’s reasoning state. Even on modern hardware, that is not a free operation. It is exactly the original problem, dressed in a bigger context window. A larger context is a better book. It is not a way to teleport your last thought.
带宽成本高得惊人。重新分词是徒劳的功耗。接收方仍然必须重新阅读整个记录——对每个 token 进行完整的预填充（prefill）处理——才能重建发送方推理状态的任何表象。即使在现代硬件上，这也不是免费的操作。这本质上还是原来的问题，只是披上了更大的上下文窗口外衣。更大的上下文只是一本更好的书，它并不是传送你上一个思维的手段。

### 3. The Systems Engineer’s Reality (Latency Budgets)
### 3. 系统工程师的现实（延迟预算）

Prompt engineering stops at “does the model give the right answer?” Systems engineering starts at “…and at what millisecond?”
提示词工程止步于“模型是否给出了正确答案？”，而系统工程师则始于“……以及在多少毫秒内完成？”

Below is a rough, illustrative latency budget for a single RAG call. It is the kind of back-of-the-envelope you scribble the first time your latency service-level agreement (SLA) stops being generous:
以下是一个单次 RAG 调用的大致延迟预算示意图。这是当你第一次发现延迟服务等级协议（SLA）不再宽裕时，会在草稿纸上随手写下的那种计算：

| Step | Illustrative Latency (ms) |
| :--- | :--- |
| Token generation (upstream) | 15 |
| Embedding | 12 |
| Network I/O | 8 |
| Vector search | 25 |
| Reranking | 10 |
| Prompt reconstruction | 15 |
| Decoding | 50 |
| **Total** | **≈ 135 ms** |

Those are blocking, sequential operations. You cannot start decoding until the prompt has been reconstructed. You cannot reconstruct the prompt until the network hop and the vector search have returned. Every millisecond in that column has to wait for the previous one to finish.
这些都是阻塞式的顺序操作。在提示词重建完成前，你无法开始解码；在网络跳转和向量搜索返回结果前，你无法重建提示词。该列表中的每一毫秒都必须等待前一阶段完成。

In a chatbot, 135 milliseconds is invisible. Nobody complains. In a continuous robotics control loop, a haptic feedback system, a self-driving stack, or a wireless base-station handover between radio access nodes, 135 milliseconds is not within the budget. It is the budget, spent entirely on plumbing, before the model has said anything useful.
在聊天机器人中，135 毫秒是不可见的，没人会抱怨。但在连续的机器人控制循环、触觉反馈系统、自动驾驶堆栈或无线基站切换中，135 毫秒超出了预算。这甚至就是全部预算，且在模型说出任何有用的内容之前，这些时间就已经全部花在了“管道工程”上。

### 4. The Evolutionary Arc of Memory
### 4. 记忆的进化弧线

This is not the first time the field has hollowed out a translation layer. Roughly, it is the fifth. Every stage on this chart was once someone’s endgame. None of them stayed the endgame.
这并不是该领域第一次掏空一个翻译层。粗略算来，这已经是第五次了。图表中的每一个阶段都曾是某些人的终极目标，但没有一个阶段能保持这种地位。

Raw Files → Relational DBs → Search Indices → Text Embeddings → Vector Search → Latent Persistence
原始文件 → 关系型数据库 → 搜索索引 → 文本嵌入 → 向量搜索 → 潜在状态持久化

Every stage in that chain solved the retrieval problem at a higher level of abstraction than the one before, and every stage eventually stopped being the primary interface. Relational databases did not disappear — they quietly became the storage layer sitting under everything else. Search indices did not disappear — they became a feature inside larger platforms. Text embeddings did not disappear — they enabled the vector search era. Each layer keeps living. It just stops being the place where new applications get built.
链条中的每个阶段都以比前一个阶段更高的抽象级别解决了检索问题，且每个阶段最终都不再是主要接口。关系型数据库没有消失，它们悄然成为了支撑一切的存储层；搜索索引没有消失，它们成为了大型平台内部的一个功能；文本嵌入没有消失，它们开启了向量搜索时代。每一层都在继续存在，只是不再是构建新应用的核心阵地。
