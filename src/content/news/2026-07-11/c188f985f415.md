# When LangGraph Succeeds but Silently Goes Wrong
# 当 LangGraph 运行成功却悄然出错时

The most dangerous failure is a clean run. The most common mistake teams make when moving a LangGraph pipeline to production is assuming that a successful exit means a correct result. It does not. A graph can traverse every node, satisfy every edge condition, and write a final output -- all without triggering any error or alert -- while quietly taking the wrong path through the logic it was built to execute.
最危险的故障是“顺利运行”。当团队将 LangGraph 流水线投入生产环境时，最常见的错误就是认为程序成功退出就意味着结果正确。事实并非如此。一个图（Graph）可以遍历所有节点、满足所有边条件并输出最终结果——全程不触发任何错误或警报——但却在执行逻辑时悄然走上了错误的路径。

This is not a theoretical concern. Production AI systems fail in this mode routinely, and it is the failure mode most monitoring setups are not designed to catch. The standard observability stack answers one question: did the pipeline run? It does not answer the harder question: did the pipeline run correctly? Those are different questions, and closing the gap between them requires a different layer of instrumentation.
这并非理论上的担忧。生产环境中的 AI 系统经常以这种方式失效，而大多数监控设置根本无法捕捉到这种故障模式。标准的观测技术栈只能回答一个问题：流水线运行了吗？它无法回答更难的问题：流水线运行得正确吗？这是两个不同的问题，要弥合两者之间的差距，需要引入不同层级的监测手段。

### The "completes but drifts" failure mode
### “完成但漂移”的故障模式

A traditional software failure is visible. An exception propagates, a process crashes, an alert fires. The failure is discrete, locatable, and traceable. The failure mode that makes agentic pipelines genuinely difficult to operate looks nothing like this. In a LangGraph pipeline, the failure is often a routing decision that was never wrong enough to fail a check but was systematically off in a way that compounded over time.
传统的软件故障是可见的。异常会传播、进程会崩溃、警报会触发。这种故障是离散的、可定位且可追踪的。而让智能体（Agentic）流水线变得极其难以维护的故障模式则完全不同。在 LangGraph 流水线中，故障往往表现为一种路由决策：它从未错到足以触发检查失败，却在系统性地出现偏差，并随着时间推移不断累积。

Classification confidence drifts slightly as input distributions shift. Retrieval quality degrades when an upstream source changes its schema. A model's behavior shifts with a temperature change or a prompt update. None of these trigger a graph exception. Each run still finishes, still writes an output, still shows green. The system reports success. What it cannot report is that the graph has been taking the jurisdiction-misclassification path for the last three weeks.
随着输入分布的变化，分类置信度会轻微漂移。当上游数据源更改架构时，检索质量会下降。模型的行为会随着温度参数（temperature）的调整或提示词（prompt）的更新而改变。这些都不会触发图异常。每次运行依然完成、依然写入输出、依然显示绿色。系统报告成功。但它无法报告的是，该图在过去三周里一直走在“管辖区误分类”的路径上。

The 19-node financial pipeline we built for a client is a useful example. It processes transactions across seven data sources, classifying each by type, applicable tax jurisdiction, and accounting rules. The graph runs unattended against live data. On any given run, the graph exits cleanly. The question the original monitoring setup could not answer was whether the classifier was applying the right jurisdiction context to ambiguous transactions -- not whether it was calculating correctly given that context, but whether the upstream classification decision that set the context was right in the first place.
我们为客户构建的 19 节点金融流水线就是一个很好的例子。它处理来自七个数据源的交易，按类型、适用税收管辖区和会计准则对每笔交易进行分类。该图在无人值守的情况下处理实时数据。在任何一次运行中，图都能顺利退出。最初的监控设置无法回答的问题是：分类器是否为模糊交易应用了正确的管辖区上下文——不是问它在给定上下文下计算是否正确，而是问最初设定该上下文的上游分类决策本身是否正确。

That distinction matters. You can have a perfectly correct formula running on the wrong inputs. The graph will complete. The output will look plausible. The error is invisible unless you have instrumented the decision that set the inputs.
这种区别至关重要。你可能拥有一个完美的公式，却在错误的输入上运行。图会完成，输出看起来也很合理。除非你对设定输入的决策过程进行了监测，否则这种错误是不可见的。

### Routing decisions as data, not control flow
### 将路由决策视为数据，而非控制流

The reframe that changes how you approach this problem: a conditional edge in a LangGraph graph is not just control flow. It is a decision -- one made by a probabilistic model, under conditions that will change over time, for reasons that should be inspectable after the fact.
改变你处理此问题方式的重构思路是：LangGraph 图中的条件边（conditional edge）不仅仅是控制流。它是一个决策——由概率模型在随时间变化的条件下做出，且其背后的原因应当在事后可供审查。

A reader named Mateo Ruiz made this point sharply in a comment on the first post in this series. His observation was that state snapshots at critical checkpoints, edge-traversal metrics, and routing-decision logging are underdiscussed relative to their importance -- and that many production incidents are not crashes at all. The graph executes successfully but gradually starts taking unexpected paths because classification confidence, retrieval quality, or model behavior drifts over time.
一位名叫 Mateo Ruiz 的读者在本系列第一篇文章的评论中尖锐地指出了这一点。他观察到，关键检查点的状态快照、边遍历指标和路由决策日志的重要性被低估了——许多生产事故根本不是崩溃。图执行成功，但由于分类置信度、检索质量或模型行为随时间漂移，它逐渐开始走上意料之外的路径。

That framing is exactly right, and it points to a concrete design principle: persist the routing decision as data. At each conditional edge, log not just which path was taken, but the inputs that drove the decision and the model's confidence in it. The difference is between a system that tells you "it went to node B" and one that tells you "it went to node B because the classifier returned jurisdiction=California with a mid-range confidence score, given these three input features."
这种框架非常准确，它指向了一个具体的原则：将路由决策作为数据持久化。在每个条件边上，不仅要记录走了哪条路径，还要记录驱动该决策的输入以及模型的置信度。这两种系统的区别在于：前者只告诉你“它去了节点 B”，而后者告诉你“它去了节点 B，因为在给定这三个输入特征的情况下，分类器返回了 jurisdiction=California，且置信度为中等水平”。

The first answer tells you what happened. The second tells you why -- and why is what you need when you're trying to understand whether the decision was right. When routing decisions are stored as structured data, the question "why did it take this path three weeks into runtime?" becomes a query, not an archaeology project. You can ask which inputs correlate with low-confidence decisions, which nodes are seeing the highest variance in their routing distributions, and whether there are patterns in the cases that later needed correction.
第一个答案告诉你发生了什么。第二个答案告诉你为什么——而当你试图判断决策是否正确时，你需要的就是“为什么”。当路由决策以结构化数据存储时，“为什么运行三周后它走了这条路径？”这个问题就变成了一个查询，而不是一项考古工程。你可以查询哪些输入与低置信度决策相关，哪些节点的路由分布方差最大，以及那些后续需要修正的案例是否存在某种模式。

### What to instrument
### 需要监测的内容

The practical instrumentation work falls into four areas:
实际的监测工作分为四个方面：

1. **State snapshots at critical checkpoints** give you a timestamped record of the shared context flowing through the graph at the moments that matter most: before and after high-stakes nodes, at the entry point of any human review gate, and wherever the graph can take branches with meaningfully different downstream consequences. The snapshot should capture the full state object, not just the delta.
1. **关键检查点的状态快照**：为你提供在最重要时刻流经图的共享上下文的时间戳记录，例如在高风险节点前后、人工审核入口处，以及任何可能导致下游后果显著不同的分支点。快照应捕获完整的状态对象，而不仅仅是增量。

2. **Edge-traversal metrics** tell you which paths through the graph are actually being taken versus which ones you designed for. A conditional edge that routes 98 percent of traffic one direction and 2 percent the other is a very different system in production than one that splits 60/40. Tracking traversal counts and distributions over time surfaces routing drift before it becomes a pattern you are explaining after the fact.
2. **边遍历指标**：告诉你图中的哪些路径实际上被走了，而哪些是你预先设计的。在生产环境中，98% 的流量走一边、2% 走另一边的条件边，与 60/40 分流的系统有着本质区别。跟踪遍历计数和分布，可以在路由漂移成为事后解释的“模式”之前将其发现。

3. **Routing-decision logging**, as described above, captures the inputs and confidence for each conditional branch. This is where the decision lives: the prompt that was sent, the model's response, the parsed output that drove the routing logic. Without it, you cannot tell the difference between a correct low-confidence decision and an incorrect high-confidence one.
3. **路由决策日志**：如上所述，捕获每个条件分支的输入和置信度。这是决策的核心所在：发送的提示词、模型的响应、驱动路由逻辑的解析输出。没有它，你就无法区分正确的低置信度决策和错误的高置信度决策。
