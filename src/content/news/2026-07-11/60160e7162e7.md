### When PySpark starts to feel slow
### 当 PySpark 开始变慢时

When you first learn PySpark, it’s natural to focus on the code. You write things like:
初学 PySpark 时，自然会专注于代码本身。你会写出类似这样的代码：

```python
df.filter(df.city == "London")
df.groupBy("customer_id").sum("amount")
df.join(customers, on="customer_id")
```

But as you move into more advanced PySpark, you need to start thinking about what PySpark has to do behind the scenes. Some operations are fairly cheap. Filtering rows is a good example. Spark can look at each row, make a yes/no decision, and move on.
但随着你进入 PySpark 的进阶阶段，你需要开始思考 PySpark 在后台做了什么。有些操作成本很低，过滤行（Filtering rows）就是一个很好的例子。Spark 可以查看每一行，做出“是/否”的判断，然后继续处理。

```python
df_london = df.filter(df.city == "London")
```

Other operations are more expensive because PySpark has to move data around. Consider this. Like the filter, it’s just one line of code, but PySpark has to make sure all rows for the same customer end up together.
其他操作则昂贵得多，因为 PySpark 必须在各处移动数据。考虑一下这个例子：虽然它和过滤一样只有一行代码，但 PySpark 必须确保同一个客户的所有行都被汇集到一起。

```python
df_totals = df.groupBy("customer_id").sum("amount")
```

That’s a much more complicated operation and may need data to move between different parts of the job. That movement is often where the cost is. So the intermediate level habit is simple: Before reaching for tuning settings, ask a simpler question: is this line of code forcing Spark to move data around?
这是一个复杂得多的操作，可能需要数据在任务的不同部分之间移动。这种移动往往就是性能损耗的来源。因此，进阶水平的习惯很简单：在尝试调整配置设置之前，先问一个简单的问题：这行代码是否迫使 Spark 进行数据移动？

This doesn’t mean joins, aggregations, or data sorting are bad. On the contrary, they’re essential, but they are the type of operations you should treat with a little more care.
这并不是说连接（Join）、聚合（Aggregation）或数据排序（Sorting）不好。相反，它们是必不可少的，但你应当对这类操作多加留意。

### How PySpark divides the work
### PySpark 如何分配工作

A PySpark DataFrame isn’t treated as a single contiguous block of data. PySpark splits it into partitions. You can think of the partitions as chunks of data that PySpark can process separately and in parallel. That is one of the main reasons PySpark is performant and useful for processing large-scale data.
PySpark DataFrame 并不是作为一个连续的数据块来处理的。PySpark 将其拆分为多个分区（Partitions）。你可以将分区视为 PySpark 可以独立且并行处理的数据块。这也是 PySpark 在处理大规模数据时高效且实用的主要原因之一。

You can check how many partitions a DataFrame has like this:
你可以通过以下方式查看 DataFrame 包含多少个分区：

```python
df.rdd.getNumPartitions()
```

As output, you’ll get a simple integer number back, for example: 8. That means PySpark has split the data into eight chunks, and this matters because partitioning affects how much parallel work PySpark can do.
输出将是一个简单的整数，例如：8。这意味着 PySpark 已将数据拆分为八个块。这一点很重要，因为分区数量直接影响 PySpark 的并行处理能力。

If you have too few partitions, PySpark may not be able to use all available CPU cores. If you have too many partitions, PySpark may spend too much time managing lots of tiny tasks and data sets. Neither extreme is ideal.
如果分区太少，PySpark 可能无法利用所有可用的 CPU 核心；如果分区太多，PySpark 可能会花费过多时间来管理大量微小的任务和数据集。这两种极端情况都不理想。

If either of these situations arises, PySpark offers two useful functions to help. You can change the number of partitions with `repartition()`:
如果出现上述任何一种情况，PySpark 提供了两个有用的函数来解决。你可以使用 `repartition()` 来更改分区数量：

```python
df_more = df.repartition(16)
```

And you can reduce the number of partitions with `coalesce()`:
你也可以使用 `coalesce()` 来减少分区数量：

```python
df_fewer = df.coalesce(4)
```

`repartition()` can increase or decrease the number of partitions, but it causes PySpark to redistribute the data, which will usually have a performance hit. `coalesce()` is used to reduce the number of partitions and needs less data movement.
`repartition()` 可以增加或减少分区数量，但它会导致 PySpark 重新分发数据，这通常会带来性能损耗。`coalesce()` 用于减少分区数量，且需要的数据移动较少。

A practical rule is: Use `repartition()` when you need to spread data out more evenly. Use `coalesce()` when you simply want fewer output files.
一个实用的原则是：当你需要更均匀地分布数据时，使用 `repartition()`；当你仅仅想要减少输出文件数量时，使用 `coalesce()`。

For example, say you have a dataset that naturally produces 40 small-ish Parquet files when it’s written out to disk. You might do this instead:
例如，假设你有一个数据集，在写入磁盘时会自然产生 40 个较小的 Parquet 文件。你可以改为这样做：

```python
df.coalesce(4).write.mode("overwrite").parquet("output/sales")
```

That tells PySpark to write the result with four partitions, which typically produces four output files. You don’t need to obsess over exact partition counts at this stage. The important point is that partitions control how PySpark divides its work. Once you understand these ideas, PySpark becomes much easier to reason about. Jobs that slow down stop feeling like random events and you start to see the likely causes: a shuffle here, a wide join there, too many tiny files somewhere else.
这告诉 PySpark 将结果写入四个分区，通常会产生四个输出文件。在这个阶段，你不需要纠结于精确的分区数量。重点在于，分区控制了 PySpark 如何分配工作。一旦理解了这些概念，PySpark 就变得更容易理解了。变慢的任务不再显得像随机事件，你开始能看出可能的原因：这里的洗牌、那里的宽连接（Wide Join），或者别处太多的微小文件。

### When data has to move
### 当数据必须移动时

Some PySpark operations can happen inside each partition. Others require data from different partitions to be brought together. That movement is called a shuffle. A shuffle happens when PySpark has to redistribute data across partitions. This is one of the first Spark ideas that really changes how you write PySpark.
有些 PySpark 操作可以在每个分区内部完成，而另一些操作则需要将来自不同分区的数据汇集在一起。这种移动被称为“洗牌”（Shuffle）。当 PySpark 必须跨分区重新分发数据时，就会发生洗牌。这是改变你编写 PySpark 代码方式的最核心概念之一。

Shuffles aren’t rare, and they aren’t automatically wrong, but they are usually computationally expensive and should be avoided if possible. Common PySpark operations that may cause shuffles include: `groupBy()`, `join()`, `distinct()`, `orderBy()`, `repartition()`.
洗牌并不罕见，也不一定就是错误的，但它们通常计算成本很高，如果可能的话应该尽量避免。常见的可能导致洗牌的 PySpark 操作包括：`groupBy()`、`join()`、`distinct()`、`orderBy()`、`repartition()`。

Take this example:
以这个例子为例：

```python
df.groupBy("customer_id").sum("amount")
```

Spark can’t calculate the final total for each customer unless all rows for that customer are aggregated. That normally means moving data between partitions. Sorting is another common example:
除非将该客户的所有行聚合在一起，否则 Spark 无法计算每个客户的总额。这通常意味着要在分区之间移动数据。排序是另一个常见的例子：

```python
df.orderBy("transaction_date")
```

Sorting a whole dataset requires PySpark to compare and arrange data across the entire dataset, not just within each.
对整个数据集进行排序需要 PySpark 在整个数据集范围内比较和排列数据，而不仅仅是在每个分区内部。
