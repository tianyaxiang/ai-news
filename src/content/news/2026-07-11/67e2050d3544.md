# The Background Task That Froze: A Serverless CPU-Throttling Mystery
# 冻结的后台任务：无服务器 CPU 节流之谜

💡 Lesson learned building an agent on Cloud Run: don't schedule work to run after you send the response. On Cloud Run's default request-based CPU mode it freezes mid-run — do the latency-critical work before you respond.
💡 在 Cloud Run 上构建代理时学到的教训：不要在发送响应后安排任务运行。在 Cloud Run 默认的基于请求的 CPU 模式下，任务会在运行中途冻结——请务必在响应前完成对延迟敏感的工作。

I had a webhook receiver that acknowledged events in well under a second. Fast, clean, exactly what you want from a webhook handler. The problem was what happened after the ack: the job that receiving the webhook was supposed to launch showed up about three minutes later. Sometimes it showed up right away. Sometimes it never showed up at all — until something completely unrelated happened to hit the service, at which point the "stuck" job would suddenly, retroactively, complete. That last detail is what made it a genuine mystery for a while. A job that finishes the instant an unrelated request lands is not behaving like a slow job. It's behaving like something that was frozen, waiting for permission to keep running.
我有一个 Webhook 接收器，它能在不到一秒的时间内确认事件。快速、简洁，这正是你对 Webhook 处理程序所期望的。问题在于确认（ack）之后发生的事情：接收 Webhook 本应启动的任务大约在三分钟后才出现。有时它会立即出现，有时则根本不会出现——直到一些完全不相关的请求访问了该服务，此时那个“卡住”的任务会突然追溯性地完成。最后一个细节让它一度成为一个真正的谜团。一个在不相关请求到达时瞬间完成的任务，其表现不像是一个慢任务，而更像是被冻结了，正在等待继续运行的许可。

The setup: The receiver did the obvious, well-worn thing: validate the payload, dedupe it, and hand the actual work off so the HTTP response could go out immediately.
设置：接收器执行了显而易见且常用的操作：验证负载、去重，并将实际工作移交出去，以便能立即发出 HTTP 响应。

```python
@app.post("/webhook")
async def webhook(req):
    ...validate...
    background_tasks.add_task(dispatch_job, payload)
    return JSONResponse(202, ...)
```

FastAPI/Starlette's BackgroundTasks is the textbook way to do "ack now, work later." It runs your callback after the response has been sent, without making the caller wait for it. On a normal server, that's a fire-and-forget convenience with no real downside — the process keeps running, the callback runs whenever the event loop gets to it, done.
FastAPI/Starlette 的 BackgroundTasks 是实现“先确认，后处理”的教科书式方法。它在响应发送后运行回调，而无需调用者等待。在普通服务器上，这是一种“即发即忘”的便利功能，没有任何实际负面影响——进程持续运行，回调在事件循环调度到它时执行，一切就绪。

This wasn't running on a normal server. It was running on Cloud Run, with the platform's default CPU allocation mode, which is request-based: the instance only gets meaningful CPU while a request is actually in flight. The moment the response goes out, CPU gets throttled down to something close to zero — and it stays there until the next request arrives. Which means the code above doesn't say "run this job right after responding." On that platform, it says "run this job whenever the next request happens to show up, if one ever does."
但这并不是在普通服务器上运行。它运行在 Cloud Run 上，使用的是平台默认的基于请求的 CPU 分配模式：实例仅在请求实际处理期间获得有效的 CPU 资源。一旦响应发出，CPU 就会被节流到接近于零——并保持在该状态，直到下一个请求到达。这意味着上述代码并不是在说“响应后立即运行此任务”。在该平台上，它的意思是“只要下一个请求出现（如果真的有的话），就运行此任务”。

The symptom, and why it's invisible: Here's the part that made this hard to see in logs: the background task doesn't fail. Nothing is killed, nothing throws, nothing gets logged as an error. The log line right before the network call in dispatch_job prints just fine — it runs during the tail end of the request's own CPU window. Then the very next line, the one after the await that actually calls out to launch the job, just... doesn't print. Not for a while. Not until CPU comes back. The event loop isn't crashing. It's simply not being scheduled.
症状及其隐蔽性：这就是导致日志难以排查的原因：后台任务并没有失败。没有进程被杀掉，没有抛出异常，也没有记录任何错误。`dispatch_job` 中网络调用前的日志行打印正常——它是在请求自身的 CPU 时间窗口末尾运行的。然后紧接着的下一行，即调用启动任务的 `await` 之后的代码，就是……不打印。很长一段时间内都不会。直到 CPU 恢复。事件循环并没有崩溃，它只是没有被调度。

request in ──► handler runs (full CPU) └─ schedules background task response out ◄── 202 sent └─ CPU throttled to ~0 ← background task freezes HERE, usually mid-network-call ...minutes pass... next request in ──► instance gets CPU again ──► frozen task resumes, completes
请求进入 ──► 处理程序运行（全 CPU） └─ 调度后台任务 响应发出 ◄── 202 已发送 └─ CPU 节流至 ~0 ← 后台任务在此处冻结，通常在网络调用中途 ...几分钟过去... 下一个请求进入 ──► 实例再次获得 CPU ──► 冻结的任务恢复并完成

If the instance happens to scale to zero before any other traffic arrives, the frozen task never gets to finish. The work isn't delayed at that point — it's just gone. This isn't unique to BackgroundTasks either. It's the same failure shape for any post-response work: a bare `asyncio.create_task(...)` fired right before returning, or a "post-ack" queue that classifies and dispatches inside the same process after replying to the caller. Anything scheduled to run after the response, on a platform that only grants CPU during a request, inherits this problem.
如果实例在其他流量到达之前恰好缩容到零，那么冻结的任务将永远无法完成。此时工作不再是延迟，而是直接丢失了。这不仅限于 BackgroundTasks。任何响应后的工作都会出现同样的故障模式：在返回前触发的裸 `asyncio.create_task(...)`，或者在回复调用者后在同一进程内进行分类和分发的“确认后”队列。在仅在请求期间授予 CPU 的平台上，任何安排在响应后运行的任务都会继承这个问题。

And it's genuinely hard to catch by normal means, because:
这确实很难通过常规手段捕捉，因为：
* It's fast on busy days and slow-or-lost on quiet ones — the exact opposite of what you'd expect from an overload problem.
* 在繁忙时它很快，在安静时却变慢或丢失——这与你对过载问题的预期恰恰相反。
* Retry-happy callers (chat platforms, webhook senders that retry on timeout) generate the very traffic that unfreezes the task, which converts "broken" into "occasionally a few minutes slow" — indistinguishable from ordinary network flakiness unless you're looking closely.
* 频繁重试的调用者（聊天平台、超时重试的 Webhook 发送方）产生的流量恰好解冻了任务，这使得“损坏”变成了“偶尔慢几分钟”——除非仔细观察，否则无法与普通的网络波动区分开来。
* It works perfectly locally and under load testing, because both of those keep the instance saturated with requests, which is precisely the condition under which this bug can't occur.
* 它在本地和负载测试中运行完美，因为这两者都会让实例保持请求饱和，而这恰恰是该 Bug 不会发生的情况。

Confirming it without a debugger: The cheapest way to confirm CPU starvation as the cause, rather than guess at it, is to inject traffic. If a "stuck" background task completes the instant you curl the service's health endpoint a couple of times, you've proven it — decisively, and without touching a debugger or adding instrumentation. I'd recommend this as step one before reaching for anything heavier.
无需调试器进行确认：确认 CPU 饥饿是根本原因（而不是猜测）的最廉价方法是注入流量。如果一个“卡住”的后台任务在你几次 `curl` 服务健康检查端点时瞬间完成，你就证明了这一点——结论明确，且无需触碰调试器或添加监控工具。我建议在采取更复杂的手段之前，先执行这一步。

The fix: move it onto the request path: The real fix isn't a smarter background task. It's not scheduling the latency-critical part as a background task at all. In this case, the RPC that actually launches the job — as opposed to waiting for the job to finish — costs on the order of one second. That's well within most webhook ack budgets (5 seconds or more), so it can just be awaited before responding:
修复方案：将其移至请求路径中：真正的修复方法不是更智能的后台任务，而是根本不要将对延迟敏感的部分安排为后台任务。在这种情况下，实际启动任务的 RPC（而不是等待任务完成）耗时约为一秒。这完全在大多数 Webhook 确认预算（5 秒或更长）之内，因此可以在响应前直接 `await` 它：

```python
@app.post("/webhook")
async def webhook(req):
    ...validate, dedupe...
    recent[key] = now() # dedup marker
    try:
        await launch_job(payload) # ~1s: starts the job, doesn't wait for it
    except Exception:
        recent.pop(key, None) # let the sender's retry re-dispatch
        raise # → 5xx → sender retries
    return JSONResponse(202, ...)
```

Two details here matter as much as the await itself:
这里有两个细节与 `await` 本身同样重要：
1. On failure, roll back the dedup marker before re-raising. If you don't, your own dedup logic swallows the sender's retry — the sender re-delivers the exact same event, your dedup marker says "already seen," and the job never gets a second chance to launch.
1. 失败时，在重新抛出异常前回滚去重标记。如果不这样做，你自己的去重逻辑会吞掉发送方的重试——发送方重新发送完全相同的事件，你的去重标记显示“已见过”，任务就永远没有第二次启动的机会了。
2. Let the failure surface as a 5xx. That's what tells the webhook source to actually retry. A 202 that lies about success is worse than an honest error.
2. 让失败以 5xx 状态码呈现。这才是告诉 Webhook 源真正需要重试的信号。一个谎称成功的 202 响应比诚实的错误更糟糕。

I kept a background task only for the one case where the handler itself would legitimately run for minutes — a local-dev mode that executes the worker in-process instead of dispatching it elsewhere. That's a fine use of "run this later"; the RPC-launching case is not.
我只在一种情况下保留了后台任务，即处理程序本身确实需要运行几分钟的情况——例如在本地开发模式下，直接在进程内执行工作任务而不是将其分发到别处。那是“稍后运行”的合理用法；而 RPC 启动任务的情况则不然。
