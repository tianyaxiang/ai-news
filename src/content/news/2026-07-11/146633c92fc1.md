# I Built My Second ETL Pipeline. This Time, I Started Thinking Like a Data Engineer
# 我构建了第二个 ETL 流水线。这一次，我开始像数据工程师一样思考

**Building a production-ready RSS pipeline with Python, Docker, PostgreSQL, and Kestra**
**使用 Python、Docker、PostgreSQL 和 Kestra 构建生产级 RSS 流水线**

A few months ago, I decided I wanted to transition from data analyst to data engineer. Like many people starting out, I was overwhelmed by the sheer number of things I thought I needed to learn. Data warehouses, orchestration tools, distributed processing, streaming systems, cloud platforms, infrastructure. The list seemed endless.
几个月前，我决定从数据分析师转型为数据工程师。和许多初学者一样，我被自己认为需要学习的庞大知识体系淹没了：数据仓库、编排工具、分布式处理、流式系统、云平台、基础设施……这份清单似乎无穷无尽。

Instead of trying to learn everything at once, I took a different approach. I created a 12-month self-study roadmap built around one simple idea: learn by building. Rather than jumping from one tutorial to another, I would build a series of small projects. Each project would introduce a few new concepts while reinforcing what I had learned before. The goal wasn’t to build the most sophisticated applications possible. It was to develop the habit of thinking like an engineer by solving one problem at a time.
我没有试图一次性学会所有东西，而是采取了另一种方法。我制定了一个为期 12 个月的自学路线图，核心理念很简单：在实践中学习。我不再盲目地从一个教程跳到另一个教程，而是通过构建一系列小项目来学习。每个项目都会引入一些新概念，同时巩固之前学到的知识。我的目标不是构建最复杂的应用程序，而是通过逐一解决问题，培养像工程师一样思考的习惯。

The first project in that journey was a GitHub ETL pipeline. It began as a simple Python script that fetched repository data and exported it to a CSV file. As I learned more, I gradually improved it. I replaced CSV files with SQLite, made the pipeline idempotent to prevent duplicate data, and eventually automated it with GitHub Actions.
这段旅程的第一个项目是一个 GitHub ETL 流水线。它最初只是一个简单的 Python 脚本，用于抓取仓库数据并导出为 CSV 文件。随着学习的深入，我逐渐改进了它：用 SQLite 替换了 CSV 文件，通过幂等性设计防止数据重复，并最终使用 GitHub Actions 实现了自动化。

By the time I finished, I realized something that hadn’t been obvious when I started. Building the ETL logic was actually the easy part. The harder questions appeared once I stopped thinking about a script that runs once and started thinking about a system that needs to run over and over again without me watching it. How should it be scheduled? What happens if it fails halfway through? Where should retry logic live? How do you package the application so it runs the same way everywhere? Those questions taught me far more about data engineering than parsing JSON or writing SQL ever did.
完成时，我意识到了一件起初并不明显的事：构建 ETL 逻辑其实是最简单的部分。真正的难题在于，当我不再将其视为一个“运行一次的脚本”，而是将其视为一个“需要无人值守、反复运行的系统”时，挑战才真正开始。如何调度它？中途失败了怎么办？重试逻辑应该放在哪里？如何打包应用程序以确保它在任何地方都能以相同方式运行？这些问题教会我的数据工程知识，远比解析 JSON 或编写 SQL 多得多。

That first project left me with another challenge. GitHub Actions worked well for automating a small ETL pipeline, but I wanted to understand what changes when you use a workflow orchestrator built specifically for data engineering. I wanted to learn how engineers separate orchestration from execution, how containerized workloads fit into that picture, and what a production-minded pipeline actually looks like, even on a small scale. So for my second project, I decided to build an automated RSS ingestion pipeline.
第一个项目给我留下了另一个挑战。GitHub Actions 在自动化小型 ETL 流水线方面表现不错，但我更想了解：当使用专门为数据工程构建的工作流编排器时，会有什么不同？我想学习工程师如何将编排与执行分离，容器化工作负载如何融入其中，以及一个具备生产思维的流水线（即使规模很小）到底是什么样子的。因此，在第二个项目中，我决定构建一个自动化的 RSS 摄取流水线。

On the surface, it’s a fairly simple application. It fetches articles from an RSS feed, parses them into structured objects, and stores them in PostgreSQL. But the goal was never to build an RSS reader. The goal was to explore the engineering decisions that transform a Python script into a reliable data pipeline. In this article, I’ll walk through those decisions, the mistakes I made along the way, and the lessons I learned building my first pipeline with Kestra.
从表面上看，这是一个相当简单的应用程序：从 RSS 源抓取文章，解析为结构化对象，并存储到 PostgreSQL 中。但我的目标从来不是构建一个 RSS 阅读器，而是探索如何通过工程决策将一个 Python 脚本转化为可靠的数据流水线。在本文中，我将梳理这些决策过程、我犯过的错误，以及我使用 Kestra 构建第一个流水线时学到的经验。

### Why Build Another ETL Pipeline?
### 为什么要再建一个 ETL 流水线？

After finishing my first ETL project, I considered moving on to something completely different. Maybe a data warehouse project. Maybe Apache Spark. Maybe an API with a more complex transformation layer. Instead, I built… another ETL pipeline. At first, that probably sounds like a step backwards. After all, I’d already built an extraction pipeline, made it idempotent, and scheduled it with GitHub Actions. Why repeat the same exercise?
完成第一个 ETL 项目后，我曾考虑转向完全不同的领域，比如数据仓库项目、Apache Spark，或者带有更复杂转换层的 API。但我最终还是选择了……再建一个 ETL 流水线。起初，这听起来可能是在走回头路。毕竟，我已经构建过一个提取流水线，实现了幂等性，并用 GitHub Actions 进行了调度。为什么要重复同样的练习？

Because I wasn’t trying to learn a new dataset. I was trying to learn a new way of thinking. One lesson from my first project stuck with me. Writing the ETL logic wasn’t the difficult part. The difficult part was everything surrounding it. How should the pipeline be executed? How should it recover from failures? How do you package it so it runs consistently on any machine? Where does scheduling belong? And perhaps the biggest question of all: Where should the responsibilities of the application end, and where should the responsibilities of the orchestration layer begin?
因为我不是在学习处理新的数据集，而是在学习一种新的思维方式。第一个项目教会我一个深刻的道理：编写 ETL 逻辑本身并不难，难的是它周围的一切。流水线该如何执行？如何从故障中恢复？如何打包以确保在任何机器上都能稳定运行？调度应该放在哪里？以及最重要的问题：应用程序的职责在哪里结束，编排层的职责又从哪里开始？

Those questions don’t have much to do with RSS feeds or GitHub repositories. They’re engineering questions, and I realized I could explore them with almost any data source. That’s why I chose an RSS feed. Not because RSS is particularly exciting, but because it’s intentionally simple. The extraction logic only takes a few lines of Python. That meant I could spend less time worrying about business logic and more time thinking about architecture.
这些问题与 RSS 源或 GitHub 仓库关系不大。它们是工程问题，我意识到可以用几乎任何数据源来探索它们。这就是我选择 RSS 源的原因——不是因为 RSS 有多令人兴奋，而是因为它足够简单。提取逻辑只需要几行 Python 代码，这意味着我可以少花时间纠结业务逻辑，而将更多精力投入到架构思考中。

For the same reason, I decided to move away from GitHub Actions for this project. GitHub Actions was a great introduction to scheduling. It showed me how to automate a workflow and gave me my first taste of running an ETL pipeline without manual intervention. But GitHub Actions isn’t designed specifically for orchestrating data workflows. I wanted to understand what changes when you use a tool that’s built with data pipelines in mind. That’s what led me to Kestra.
出于同样的原因，我决定在这个项目中放弃 GitHub Actions。GitHub Actions 是了解调度的绝佳入门工具，它向我展示了如何自动化工作流，并让我第一次体验了无需人工干预运行 ETL 流水线的感觉。但 GitHub Actions 并非专门为编排数据工作流而设计。我想了解使用专门为数据流水线构建的工具会有什么不同，这便是我选择 Kestra 的原因。

Rather than asking, “How do I run this Python script every hour?”, I found myself asking a different set of questions. How should retries be configured? How are workflow executions tracked? How should environment variables be passed into a container? What does a failed execution look like? How do you separate the application from the infrastructure that runs it? Those questions were exactly what I wanted to explore. By choosing a simple ETL pipeline, I could focus on the engineering decisions instead of getting distracted by complicated business logic.
我不再问“如何每小时运行这个 Python 脚本？”，而是开始思考一系列新问题：如何配置重试？如何跟踪工作流执行情况？如何将环境变量传递给容器？执行失败是什么样子的？如何将应用程序与运行它的基础设施分离？这些正是我想要探索的问题。通过选择一个简单的 ETL 流水线，我可以专注于工程决策，而不被复杂的业务逻辑所干扰。

Looking back, I think that was the right decision. This project isn’t interesting because it processes RSS feeds. It’s interesting because building it forced me to think about reliability, repeatability, and orchestration in a way my first project never did.
回过头来看，我认为这是一个正确的决定。这个项目之所以有趣，并不是因为它处理的是 RSS 源，而是因为它迫使我以一种前所未有的方式去思考可靠性、可重复性和编排问题。
