# RAG 生成的循环工程：逐个迭代 top-k

The user asks “what is the effective date of this policy?”. Retrieval returns five candidate line-windows and passes them all to the LLM in one prompt. The LLM reads all five to extract the same date the first candidate already had. Chunks two to five were signatures, footnotes, and a paragraph about historical dates. Paid for nothing.
用户询问“该政策的生效日期是什么？”。检索系统返回五个候选行窗口，并将它们全部放入一个提示词中传给大模型（LLM）。LLM 读取了全部五个窗口，却提取出了第一个候选窗口中已经包含的日期。第二到第五个片段全是签名、脚注和一段关于历史日期的段落。这完全是浪费成本。

Send the top-1 first, ask the LLM if that is enough, and stop when it says yes: sequential feeding cuts the token cost by 80% on this class of question. The rest of the article catalogues where sequential wins, where a single-call over all K is the right default, and how the question parser dispatches between the two.

### 1. Two regimes for feeding the top-K to generation
### 1. 将 top-K 馈送至生成模块的两种机制

Retrieval hands generation an ordered top-K. There are two ways to feed it in, and they cost very differently.
检索系统会将排序后的 top-K 结果交给生成模块。馈送方式有两种，且成本差异巨大。

#### 1.1 The fixed top-K pipeline and what it wastes
#### 1.1 固定 top-K 流水线及其浪费之处

The default pattern across most RAG tutorials looks like this: `top_k = retrieval(question, k=5); answer = generation(question, top_k)`. It runs in two steps for every question. Token cost is roughly `generation_cost(question + 5 chunks of context)`. Latency is roughly `retrieval + one LLM call`. And the LLM happily processes the 5 chunks even when the top-1 was already sufficient and chunks 2..5 added zero information.
大多数 RAG 教程中的默认模式如下：`top_k = retrieval(question, k=5); answer = generation(question, top_k)`。每个问题都分两步运行。Token 成本大致为 `generation_cost(问题 + 5 个上下文片段)`。延迟大致为 `检索时间 + 一次 LLM 调用时间`。即使 top-1 已经足够，且第 2 到第 5 个片段没有提供任何额外信息，LLM 依然会处理这 5 个片段。

#### 1.2 Sequential: top-1 first, sufficiency predicate, escalate if needed
#### 1.2 顺序处理：优先 top-1，充分性判定，必要时升级

The sequential regime treats the K candidates as an ordered list and asks the generation brick to validate sufficiency at each step:
顺序机制将 K 个候选内容视为有序列表，并要求生成模块在每一步验证充分性：
```python
for i, candidate in enumerate(top_k):
    answer = generation(question, [candidate])
    if answer.answer_found and answer.complete_answer_found:
        break
```
The sufficiency signal lives inside the typed contract Article 8A introduced. The `AnswerWithEvidence` schema exposes `answer_found` (was the question’s information present in this candidate?) and `complete_answer_found` (was the entire answer present, not a fragment?). The loop reads those fields, not a custom heuristic.
充分性信号存在于第 8A 篇引入的类型化契约中。`AnswerWithEvidence` 模式暴露了 `answer_found`（问题信息是否出现在该候选内容中？）和 `complete_answer_found`（是否包含完整答案，而非片段？）。循环读取这些字段，而非依赖自定义的启发式规则。

#### 1.3 Batch: send all K at once, let the LLM arbitrate
#### 1.3 批处理：一次性发送所有 K 个，由 LLM 进行仲裁

Batch mode keeps the default behaviour: one LLM call sees all K candidates and produces one typed answer. Its case is built on three question types where sequential breaks down:
批处理模式保持默认行为：一次 LLM 调用查看所有 K 个候选内容并生成一个类型化答案。其适用场景基于顺序处理失效的三种问题类型：

*   **Listing questions:** “list all exclusions in this contract”. Sequential would stop at top-1 and miss the other four.
*   **列表类问题：** “列出合同中的所有除外条款”。顺序处理会在 top-1 处停止，从而遗漏其余四个。
*   **Comparison questions:** “is the premium higher than the previous year’s?”. The answer requires both candidates in the same call.
*   **比较类问题：** “保费是否高于去年？”。答案需要将两个候选内容放在同一次调用中进行对比。
*   **Tight-score retrieval:** when the top-K candidates’ relevance scores are within 5% of each other, retrieval cannot reliably promote top-1 to the top.
*   **高分重叠检索：** 当 top-K 候选内容的相关性得分相差在 5% 以内时，检索系统无法可靠地确定 top-1 就是最优解。
