### Key Takeaways
### 核心要点

What this post is: a step-by-step tour of a from-scratch CUDA inference runtime for Qwen2.5-Coder-7B-Instruct on NVIDIA H100 (`sm_90`) — every hot path annotated for why, not just what.
本文内容：这是一次针对 NVIDIA H100 (`sm_90`) 架构上 Qwen2.5-Coder-7B-Instruct 模型的从零构建 CUDA 推理引擎的逐步指南——每一个热点路径不仅标注了“是什么”，还解释了“为什么”。

What the runtime does today: greedy graph-tier token match passes on H100 at ≤4096 context, steady-state decode lands around ~16.7 ms/token (~60 tok/s), and first-token latency is ~128 ms for a 512-token prompt.
当前引擎的功能：在 H100 上支持 ≤4096 上下文长度的贪婪搜索（Greedy Search）图层匹配，稳态解码速度约为 16.7 毫秒/token（约 60 tok/s），512 token 提示词的首字延迟约为 128 毫秒。

The one number that made everything else make sense: eager decode was ~119 ms/token. Wrapping decode in a captured CUDA graph collapsed that to ~17 ms/token. A 7× win — same kernels, just less driver.
让一切变得合理的关键数据：即时（Eager）解码约为 119 毫秒/token。将解码过程封装在捕获的 CUDA Graph 中后，这一数值降至约 17 毫秒/token。性能提升了 7 倍——内核完全相同，只是减少了驱动程序的开销。

Three bugs did most of the teaching: `__syncthreads()` inside a warp-specialized branch is undefined behaviour (and corrupts softmax at KV page tails), CUDA graphs are not optional, and `prmt.b32` beat `__dp4a` on Hopper for gate/up/down GEMV shapes — the INT8-activation path was tried and removed.
三个 Bug 带来的教训：在 Warp 专业化分支中使用 `__syncthreads()` 是未定义行为（会导致 KV 页面末尾的 Softmax 计算损坏）；CUDA Graph 不是可选项，而是必须项；在 Hopper 架构上，对于 gate/up/down GEMV 形状，`prmt.b32` 的表现优于 `__dp4a`——INT8 激活路径已被尝试并移除。

llama.cpp is the reference implementation, not the rival: at commit b3040 with Q4_K_M on the same GPU class, the same pp512/tg128 protocol posts ~43 ms TTFT and ~4.95 ms/token decode. It is the yardstick to check whether your own build is behaving reasonably.
llama.cpp 是参考实现，而非竞争对手：在同一 GPU 等级下，使用 commit b3040 和 Q4_K_M 量化，相同的 pp512/tg128 协议下，其 TTFT（首字延迟）约为 43 毫秒，解码速度约为 4.95 毫秒/token。它是衡量你自己的构建是否合理的标尺。

TL;DR up front, so you can leave with the point: if you have ever wanted to actually build an LLM inference runtime yourself — pack your own weights, own every barrier, capture your own CUDA graphs — this is what that journey looks like on an NVIDIA H100 (Hopper, sm_90) for Qwen2.5-Coder-7B.
开篇先说结论：如果你曾想过亲手构建一个 LLM 推理引擎——打包自己的权重、掌控每一个同步屏障、捕获自己的 CUDA Graph——这就是在 NVIDIA H100 (Hopper, sm_90) 上为 Qwen2.5-Coder-7B 构建引擎的完整历程。

You will emerge with strong opinions about warp specialization, TMA, `__syncthreads()`, and the fact that a CUDA graph is not a bonus, it is a necessity. This article is a tour of a small LLM runtime called `annotated-llm-runtime` — roughly two dozen CUDA/C++ source files, every hot path commented for why, not just what — and the three war (figurative) stories that produced most of the annotations.
你将对 Warp 专业化、TMA、`__syncthreads()` 以及“CUDA Graph 不是加分项，而是必需品”这一事实产生深刻的见解。本文将带你浏览一个名为 `annotated-llm-runtime` 的小型 LLM 推理引擎——它包含约二十几个 CUDA/C++ 源文件，每个热点路径都不仅有代码，还有关于“为什么”的详细注释，以及产生这些注释的三场“战争”（比喻）故事。

### 1. Why build your own LLM runtime at all?
### 1. 为什么要从零构建自己的 LLM 推理引擎？

Most production LLM inference already routes through a very well-tested path: GGUF → llama.cpp → done. It works, it is fast, it has an army of contributors, and it is the right tool for shipping. Let me be clear right now: nothing in this post is trying to challenge that, and I recommend you keep using it in production, as I do.
大多数生产环境的 LLM 推理都走了一条经过充分验证的路径：GGUF → llama.cpp → 完成。它工作稳定、速度快、拥有庞大的贡献者群体，是产品交付的正确工具。我必须明确一点：本文无意挑战这一点，我也建议你在生产环境中继续使用它，正如我所做的那样。

Ownership of the decode stack is where the value is. If you ever need to change the quantization format, add a custom sampler, plug in a novel attention variant, or ship on a new architecture, you cannot really do it from inside a black box. You need to be able to open every kernel and know what will break — and what has already broken and been fixed.
掌握解码栈的控制权才是价值所在。如果你需要更改量化格式、添加自定义采样器、插入一种新颖的注意力机制变体，或者在新的架构上部署，你无法在黑盒内部完成这些操作。你需要能够打开每一个内核，了解什么会崩溃——以及什么已经崩溃并被修复了。

I think, at some point of time, we have already or we will ask ourselves the following question: Can I write decode myself, understand every launch, own every barrier, and still get the correct tokens out? The short answer is yes.
我想，在某个时刻，我们都已经或将会问自己这样一个问题：我能自己写解码器吗？理解每一次内核启动，掌控每一个同步屏障，并最终输出正确的 token 吗？简短的回答是：可以。

If you have ever been curious about how a real decoder is actually glued together — packed weight files, INT4 dequant, paged KV, warp specialization, CUDA graphs, TMA — but you have been too intimidated to read all fifty-thousand lines of GGML to find out, this repo (and this post) is the same journey compressed into roughly two dozen CUDA/C++ source files with dense comments.
如果你曾好奇一个真正的解码器是如何拼凑在一起的——打包权重文件、INT4 反量化、分页 KV Cache、Warp 专业化、CUDA Graph、TMA——但又因为 GGML 五万行代码而望而却步，那么这个仓库（以及这篇文章）就是将这段旅程压缩进约二十几个带有详尽注释的 CUDA/C++ 源文件中。

Every hot path in the code carries the why. Every barrier says which threads arrive on it and which threads must not. Every kernel says which shape it was tuned for and which shape it silently regressed on. It is the version of the codebase I wish had existed when I started.
代码中的每一个热点路径都解释了“为什么”。每一个同步屏障都说明了哪些线程会到达这里，哪些线程绝对不能到达。每一个内核都说明了它针对哪种形状进行了调优，以及在哪些形状上出现了静默回归。这就是我刚开始时梦寐以求的代码库版本。

Some of the annotations exist because I had to talk myself through fixing the code. Others exist because I had to talk myself out of a bad idea I had already implemented. This is the second kind of comment, in the wild:
有些注释的存在是因为我必须通过自言自语来理清修复代码的思路。另一些注释则是因为我必须说服自己放弃已经实现的一个糟糕想法。以下是第二种注释的真实案例：

`// prmt.b32 path exercises Hopper integer permute pipes — baseline before fused scale broadcast.`
`// ValueShuffle lane map is frozen to match offline packer — runtime cannot reinterpret nibbles.`
`// prmt.b32 路径利用了 Hopper 的整数置换管道——这是融合缩放广播之前的基准。`
`// ValueShuffle 通道映射已冻结以匹配离线打包器——运行时无法重新解释半字节（nibbles）。`

That comment is the epitaph of a wasted afternoon.
那条注释就是一个浪费掉的下午的墓志铭。

### 2. Where the stack currently sits
### 2. 当前引擎的性能水平

Before we walk through the internals, the honest current state of the runtime. This is measured on the same H100 class, at the benchmark_e2e protocol locked in the config (batch=1, pp512/tg128, --warmup 3 --prompt-tokens 512 --gen-tokens 128):
在深入了解内部结构之前，先诚实地展示一下该引擎的当前状态。这是在同一 H100 等级上，按照配置中锁定的 benchmark_e2e 协议（batch=1, pp512/tg128, --warmup 3 --prompt-tokens 512 --gen-tokens 128）测得的数据：

| Metric | This Solution | llama.cpp Q4_K_M |
| :--- | :--- | :--- |
| TTFT (512 token prompt) | 128ms | 43ms |
| Decode ITL (steady-state) | 16.7 ms/token | 4.95 ms/token |
| Decode throughput | 60 tok/s | 200 tok/s |

If you are rebuilding this stack yourself, the llama.cpp column is a reference implementation for the same protocol — pinned at commit b3040 with the standard offload flags (-ngl 99 -c 8192 -b 512 -cb) and the official Qwen/Qwen2.5-Coder-7B-Instruct-GGUF weights. Same GPU class, same model, same greedy decode, different amount of tuning behind them. It is the yardstick to check that your own build is behaving reasonably, not a rival to chase this early.
如果你正在自己重建这个栈，llama.cpp 列是同一协议的参考实现——锁定在 commit b3040，使用标准卸载标志 (-ngl 99 -c 8192 -b 512 -cb) 和官方 Qwen/Qwen2.5-Coder-7B-Instruct-GGUF 权重。相同的 GPU 等级、相同的模型、相同的贪婪解码，只是背后的调优程度不同。它是用来检查你的构建是否合理的标尺，而不是现阶段要追赶的竞争对手。

For the CUDA-flavored readers: the single biggest lever inside this stack was CUDA graph decode. Eager per-token decode on the same kernels came in around 119 ms/token. Wrapping steady-state decode in a captured graph and replaying it dropped that to ~17 ms/token. Same kernels, just submitted differently. Whole story on that in Section 6.
对于 CUDA 开发者：该栈中最大的性能杠杆是 CUDA Graph 解码。在相同内核上进行即时（Eager）逐 token 解码约为 119 毫秒/token。将稳态解码封装在捕获的图中并重放，将其降至约 17 毫秒/token。内核相同，只是提交方式不同。关于此的完整故事在第 6 节。
