# Building Trustworthy Production RAG Systems Through Continuous Evaluation
# 通过持续评估构建值得信赖的生产级 RAG 系统

A practical guide to building an evaluation workflow that catches retrieval failures, hallucinations, and performance drift before they reach users.
这是一份实用的指南，旨在构建一套评估工作流，在检索失败、幻觉和性能漂移影响用户之前将其捕获。

If you’ve built a RAG system then you’ve already been through this situation when it looks like your RAG system is working fine but it is still wrong. The retrieval fetches some chunks, passes it to the generative model, and it writes a fluent answer. Nothing throws an error, and yet the answer might be built on the wrong document, or missing half the information it needed, or grounded in something that’s technically true but three versions out of date.
如果你构建过 RAG 系统，一定经历过这种情况：系统看起来运行正常，但结果却是错的。检索模块抓取了一些片段，传给生成模型，模型写出了流畅的回答。没有任何报错，但答案可能基于错误的文档，或者丢失了一半必要信息，又或者基于技术上正确但已过时三个版本的旧信息。

The only way to actually know and mitigate this is to evaluate it properly on a regular basis. This article walks through how to build that evaluation pipeline yourself, starting with the parts that feel too basic to write about, through automated evaluation with RAGAS, past the point where RAGAS stops being enough, and into what it takes to run this as an actual process rather than a notebook you open before a stakeholder meeting.
要真正发现并缓解这些问题，唯一的办法就是定期进行适当的评估。本文将带你逐步构建评估流水线，从看似基础的环节开始，涵盖使用 RAGAS 进行自动化评估，探讨 RAGAS 的局限性，并深入讲解如何将其转化为一种常态化流程，而不是仅仅在利益相关者会议前打开的笔记本文件。

### Building the Golden Dataset
### 构建黄金数据集 (Golden Dataset)

Before touching any evaluation library, you need a set of questions with known correct answers. This is called a golden dataset, and skipping it is the most common mistake teams make while working on a RAG system. Jumping straight to running RAGAS on random queries with nothing to compare against tells you almost nothing about whether the system is actually right or wrong.
在接触任何评估库之前，你需要准备一组已知正确答案的问题。这被称为“黄金数据集”，跳过这一步是团队在开发 RAG 系统时最常犯的错误。直接对随机查询运行 RAGAS 而没有对比基准，几乎无法告诉你系统到底是正确还是错误。

A good golden dataset entry has three parts: the question, a correct answer written by someone who knows the domain, and which document contains the answer. The third field is what lets you tell a retrieval failure (wrong chunk fetched) apart from a generation failure (right chunk, wrong answer written from it). These are different bugs with different fixes, and a dataset without this field will leave you debugging blind.
一个优秀的黄金数据集条目包含三个部分：问题、由领域专家编写的正确答案，以及包含答案的文档来源。第三个字段能让你区分检索失败（抓取了错误的片段）和生成失败（片段正确但回答错误）。这是两种不同的 Bug，需要不同的修复方法，如果数据集缺少这个字段，你将陷入盲目调试的困境。

Twenty to thirty questions is enough to get real signal at the start. The category field matters more than the count of entries, and this is usually where teams underinvest the most. A dataset made only of easy factual lookups will pass everything and tell you nothing, because easy factual lookups are the one failure mode RAG systems rarely have.
起步阶段，20 到 30 个问题足以获得有效的信号。类别字段比条目数量更重要，而这通常是团队投入最少的地方。如果数据集只包含简单的客观事实查询，系统会全部通过，但你依然一无所获，因为简单的客观事实查询是 RAG 系统极少出现的故障模式。

### Doing a simple manual pass check before automating
### 在自动化之前进行简单的手动检查

Run the golden set through your pipeline and read the answers next to the ground truth before setting up any scoring tool. This step gets skipped constantly because it feels too basic to be real engineering work but it tells you whether your dataset itself is sound, and gives you a feel for what “wrong” looks like in your specific system before you trust a metric to catch it for you.
在设置任何评分工具之前，先将黄金数据集运行一遍你的流水线，并将生成的答案与标准答案进行对比阅读。这一步经常被跳过，因为它看起来太基础，不像真正的工程工作，但它能告诉你数据集本身是否可靠，并让你在依赖指标之前，先直观感受你的系统中“错误”是什么样子的。

### Automate scoring with RAGAS
### 使用 RAGAS 自动化评分

Once the basics hold up, RAGAS gives you a fast, repeatable way to score the pipeline on four dimensions without reading every answer by hand.
一旦基础工作就绪，RAGAS 提供了一种快速、可重复的方法，可以在四个维度上对流水线进行评分，而无需手动阅读每一个答案。
