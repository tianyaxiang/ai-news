# Most RAG Hallucinations Are Retrieval Failures: How the Retrieval Brick Decides What the Model Can Invent

**大多数 RAG 幻觉都是检索失败：检索模块如何决定模型能“编造”什么**

Enterprise Document Intelligence [Vol.1 #7quinquies] – Hallucination is usually garbage-in. Fix retrieval, and the model has nothing left to make up.
企业文档智能 [第1卷 #7quinquies] —— 幻觉通常源于“垃圾进，垃圾出”。修复检索，模型就无从编造了。

Kezhan Shi | Jul 15, 2026 | 13 min read
Kezhan Shi | 2026年7月15日 | 13分钟阅读

When a RAG system returns a confident wrong answer, the instinct is to reach for the generation knobs: a stricter prompt, a bigger model, a temperature of zero. That treats a retrieval failure as a generation failure. The model was handed the wrong pages and did its job on them. Retrieval decides what the model can invent, because it decides what the model can see.
当 RAG 系统给出一个自信但错误的答案时，人们的本能反应是去调整生成参数：更严格的提示词、更大的模型、或者将温度设为零。这实际上是将检索失败误当作了生成失败。模型只是拿到了错误的页面，并基于这些页面完成了它的工作。检索决定了模型能“编造”什么，因为它决定了模型能看到什么。

This is a manifesto piece in Enterprise Document Intelligence (the series philosophy). It zooms in on brick 3, retrieval, with one claim: most of what teams log as LLM hallucination in RAG is not the model inventing facts. It is the model answering, faithfully, from context retrieval should never have put in front of it.
这是《企业文档智能》系列的一篇宣言（该系列的哲学核心）。本文聚焦于第三块积木——检索，并提出了一个观点：团队在 RAG 中记录的大多数 LLM 幻觉，并非模型在编造事实。而是模型忠实地根据检索系统不该提供给它的上下文进行了回答。

### 1. A measurement: the answer ranked last of 55
### 1. 一项测量：答案在 55 个页面中排名垫底

We ran the textbook naive pipeline on the NIST Cybersecurity Framework v1.1, a real 55-page enterprise standard: embed every page with a sentence-transformer, embed the question, rank pages by cosine similarity, keep the top-k. The question was “what backup practices keep data available after a ransomware attack?”, and the framework answers it plainly in subcategory PR.IP-4: “Backups of information are conducted, maintained, and tested.”
我们在 NIST 网络安全框架 v1.1（一份真实的 55 页企业标准）上运行了教科书式的朴素流水线：使用句子转换器（sentence-transformer）对每一页进行嵌入，对问题进行嵌入，按余弦相似度对页面排序，并保留前 k 个结果。问题是“哪些备份实践能在勒索软件攻击后保持数据可用？”，框架在子类别 PR.IP-4 中明确回答了这个问题：“对信息进行备份、维护和测试。”

The entire pipeline is a dozen lines, and it runs on a laptop with no API key.
整个流水线只有十几行代码，可以在没有 API 密钥的笔记本电脑上运行。

```python
import numpy as np
from sentence_transformers import SentenceTransformer
from docintel.parsing.pdf import parse_pdf

model = SentenceTransformer("all-MiniLM-L6-v2")
lines = parse_pdf("nist_csf.pdf")["line_df"]
pages = lines.groupby("page_num")["text"].apply(" ".join)

question = "what backup practices keep data available after a ransomware attack?"
emb = model.encode([question, *pages], normalize_embeddings=True)
scores = emb[1:] @ emb[0] # cosine of every page against the question
order = np.argsort(-scores) # most similar page first

# the answer lives in subcategory PR.IP-4, on page 41
rank = list(pages.index[order]).index(41) + 1
print(rank) # 55 (out of 55 pages)
```

That last line prints 55. The answer is not merely outside the top-k window; it is the single least similar page in the whole document, with the lowest cosine score of all. Here is the part that should sting. The question contains the word "backup," and exactly one page in the entire framework also contains it: page 41, the answer. Cosine ranked that page dead last anyway. The one word that mattered was averaged away by the rest of the question.
最后一行代码输出 55。答案不仅不在前 k 个窗口内，而且是整个文档中相似度最低的页面，其余弦得分是所有页面中最低的。最令人扎心的是：问题中包含“backup”（备份）一词，而整个框架中恰好只有一个页面包含该词：第 41 页，即答案所在页。然而，余弦相似度算法还是将该页排在了最后。那个最关键的词被问题中其余部分的信息“平均”掉了。

The page that ranked first was the Data Security category (PR.DS), which is about protecting data and is the wrong control; the rest of the top five were framework prose. Feed generation those five and it never sees the backup subcategory. It is handed data-security policy and asked how to survive ransomware, so it answers from what it was given, or fills the gap from its training, and that confident, source-free answer is what a reviewer later logs as a hallucination. The model did not invent from nothing. It answered the wrong pages.
排名第一的是“数据安全”（PR.DS）类别，它讨论的是保护数据，属于错误的控制措施；前五名中的其余部分则是框架的序言。将这五页内容喂给生成模型，它永远看不到备份子类别。它被给予了数据安全策略，却被问及如何应对勒索软件，因此它基于所给内容进行回答，或者从其训练数据中填补空白。这种自信但缺乏依据的回答，就是评审人员随后记录的“幻觉”。模型并非凭空捏造，它只是回答了错误的页面。

### 2. The three retrieval conditions that cause hallucination
### 2. 导致幻觉的三种检索情况

Hallucination in a grounded pipeline is almost always one of three retrieval conditions, and none of them is the model’s fault. They are worth separating because they fail differently and they leave different traces in a log.
在基于事实的流水线中，幻觉几乎总是由以下三种检索情况之一引起的，且没有一种是模型的错。区分它们很有必要，因为它们的失败方式不同，在日志中留下的痕迹也不同。

#### 2.1 The answer was not retrieved
#### 2.1 答案未被检索到

The relevant control sat at rank 55 and the top-k window ended at five. The model is asked a question whose answer is nowhere in its context, so it fills the gap from its parameters. Handed the Data Security pages and the framework’s front matter and asked how to survive ransomware, it returns something like “maintain strong access controls, encrypt data at rest, and segment the network”: fluent, security-flavoured, and supported by nothing it was given. This is the classic confident invention, and it is a recall failure upstream.
相关的控制措施排在第 55 位，而前 k 个窗口只取前 5 位。模型被问到一个在上下文中找不到答案的问题，于是它从参数中填补了空白。当被给予数据安全页面和框架前言，并被问及如何应对勒索软件时，它回答诸如“保持强大的访问控制、加密静态数据、进行网络分段”之类的内容：这些回答流畅、具有安全专业感，但没有任何所给材料的支持。这是典型的自信编造，本质上是上游的召回失败。

#### 2.2 The wrong passage was retrieved
#### 2.2 检索到了错误的段落

A passage that sounds right but says something else wins the ranking. On this run the Data Security category (PR.DS) ranked first: PR.DS-1 (data-at-rest is protected) and PR.DS-2 (data-in-transit is protected) read, to cosine, as the answer to keep data available, because both are about protecting data. They are real controls, and they are the wrong ones. Neither is the backup subcategory. This is the failure that survives review. The model reads PR.DS, answers from it, and often...
一段听起来正确但内容无关的段落赢得了排名。在这次运行中，“数据安全”（PR.DS）类别排在第一：PR.DS-1（静态数据受保护）和 PR.DS-2（传输中数据受保护）在余弦算法看来，就是“保持数据可用”的答案，因为两者都与保护数据有关。它们是真实的控制措施，但却是错误的。它们都不是备份子类别。这种失败往往能通过评审。模型阅读了 PR.DS，并据此回答，通常……
