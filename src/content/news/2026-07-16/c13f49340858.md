# Don’t Let Claude Grade Its Own Homework
# 别让 Claude 给自己的作业打分

**Agentic AI: Cross-provider PR review with Codex in GitHub Actions, and why a second opinion from a different lab beats any self-review**
**智能体 AI：通过 GitHub Actions 使用 Codex 进行跨供应商 PR 代码审查，以及为什么来自不同实验室的“第二意见”优于任何自我审查**

A while ago, I had a handful of parallel Claude agents write me up a strategy folder for a side project. The pull request looked great: ten documents, all neatly cross-linked, and it read well. Then the automated reviewer posted its findings, and Claude, to its credit, did not object: Claude conceding, mid-session: “The automated review caught a real bug: the doc writers used guessed filenames for cross-links. Valid finding, I’ll fix it.”
前段时间，我让几个并行的 Claude 智能体为我的一个副项目编写了一个策略文件夹。提交的合并请求（PR）看起来很棒：十份文档，全部整齐地交叉链接，读起来也很通顺。随后，自动化审查工具发布了审查结果，Claude 倒也坦诚，没有提出异议。Claude 在会话中承认道：“自动化审查发现了一个真正的错误：文档编写者在交叉链接中使用了猜测的文件名。这是一个有效的发现，我会修复它。”

The docs it wrote linked to files it invented itself. The full audit came back: 30 of the 95 relative links pointed at files that did not exist. ./market.md instead of market-analysis.md, ./monetization.md instead of business-model.md, some files were even made up entirely! Skimming that PR as a human, I would have approved it, since every one of them looked plausible. Luckily, my automated reviewer caught it because it resolved every link instead of judging whether the names “vibed” right.
它编写的文档链接到了它自己虚构的文件中。完整的审计结果显示：95 个相对链接中有 30 个指向了不存在的文件。例如用 `./market.md` 代替了 `market-analysis.md`，用 `./monetization.md` 代替了 `business-model.md`，有些文件甚至完全是凭空捏造的！作为人类，如果我只是粗略浏览那个 PR，我肯定会批准它，因为每一个链接看起来都煞有介事。幸运的是，我的自动化审查工具捕捉到了这些错误，因为它会解析每一个链接，而不是仅仅判断这些名字看起来是否“顺眼”。

To be fair, Claude does not have any bad intent behind this. A language model simply generates the most plausible continuation, and sometimes the most plausible continuation is fiction delivered with complete confidence. From the reading side, though, that distinction brings little comfort. Confident and wrong reads exactly like confident and right.
平心而论，Claude 并没有恶意。语言模型只是在生成最合理的续写内容，而有时最合理的续写内容就是一本正经地胡说八道。然而，从阅读者的角度来看，这种区别并不能带来多少安慰。自信的错误读起来和自信的正确一模一样。

This post is about the system I have in place to help me review Claude’s mistakes and hallucinations: a Codex agent running in GitHub Actions that reviews every pull request. It walks you through my setup, why cross-provider review beats both self-review and a human-reads-everything review, how to enforce memory and accountability in the review loop, the cost and reliability learnings I gathered along the way, and the actual code I use to make this work.
这篇文章介绍了我建立的一套系统，旨在帮助我审查 Claude 的错误和幻觉：一个在 GitHub Actions 中运行的 Codex 智能体，它会审查每一个合并请求。本文将带你了解我的设置，解释为什么跨供应商审查优于自我审查和人工全量审查，探讨如何在审查循环中强化记忆和问责制，分享我在过程中积累的成本与可靠性经验，并提供我实现这一功能的实际代码。

### Agent throughput broke the human review loop
### 智能体吞吐量打破了人工审查循环

Hallucinations by themselves are a manageable problem, since a very careful review or functional assessment would catch them. However, what broke over the past year is the volume that needs reviewing. A single engineer now runs multiple agent sessions in parallel, and each of them can produce a thousand-line diff in the time it takes to refill your coffee. Producing code has become the fast and easy part; reviewing it honestly is what eats the day.
幻觉本身是一个可控的问题，因为仔细的审查或功能评估就能发现它们。然而，过去一年里出现的问题是需要审查的工作量激增。现在，一名工程师可以同时运行多个智能体任务，每一个任务在倒杯咖啡的时间内就能产生上千行的代码差异（diff）。编写代码已经变得快速而简单；而认真审查代码才是真正耗费时间的工作。

That makes the human reviewer the bottleneck of the whole delivery pipeline, and the bottleneck is always the next thing you should automate. That bottleneck invites two familiar failure modes. Either you approve whatever the agents produce without truly reading it, and the broken links, phantom parameters, and subtly wrong edge cases ship to production. Or you still insist on reading every line yourself, and your fast and smart AI agents work even less than you do. The first is negligent, the second wasteful, and most teams oscillate between both depending on the deadline.
这使得人类审查员成为了整个交付流水线的瓶颈，而瓶颈永远是下一个应该自动化的环节。这个瓶颈会导致两种常见的失败模式：要么你对智能体生成的代码不加阅读直接批准，导致损坏的链接、虚构的参数和隐蔽的边缘情况被发布到生产环境；要么你坚持阅读每一行代码，结果你那快速且聪明的 AI 智能体反而比你工作得更少。前者是疏忽，后者是浪费，大多数团队会根据截止日期的紧迫程度在两者之间摇摆。

The conclusion I keep landing on: our review process has to scale with the speed we’re producing code. The human element stays, but the human reviewer’s focus should go to the parts that truly matter. The grind of checking whether links resolve, whether the new parameter exists, whether there’s a bug or vulnerability, and whether the code changes match what was initially requested, all of that has to run automatically at the same cadence as the coding agents, even before a person ever spends a second reviewing the code.
我得出的结论是：我们的审查流程必须与我们生产代码的速度同步扩展。人类的参与依然必要，但人类审查员的精力应该集中在真正重要的地方。检查链接是否有效、新参数是否存在、是否存在漏洞、代码变更是否符合最初需求等繁琐工作，都必须与编码智能体以相同的节奏自动完成，甚至在人类花时间审查代码之前就应完成。

Insight: hallucinations are a constant, throughput is what changed. Any process that requires a human to read every line caps your agents at human reading speed, so the very first review pass has to be automated.
洞察：幻觉是恒定的，改变的是吞吐量。任何要求人类阅读每一行的流程都会将智能体的效率限制在人类阅读速度上，因此第一轮审查必须实现自动化。

### Pick a reviewer that does not share the author’s blind spots
### 选择一个与作者没有共同盲点的审查者

The obvious first idea to prevent these mistakes is to have the agent review its own work… but it is also the weakest one. A hallucination that made it into the diff is, by definition, plausible to the model that produced it. Asking the same weights to find mistakes in the code, especially within the same already-biased coding session, usually reproduces the exact blind spots that let the mistakes through.
防止这些错误最直观的想法是让智能体审查自己的工作……但这恰恰是最薄弱的方法。既然幻觉已经进入了代码差异中，那么对于生成它的模型来说，它在定义上就是“合理”的。要求相同的权重去寻找代码中的错误，尤其是在同一个已经存在偏见的编码会话中，通常会重现导致错误产生的盲点。

This intuition has research behind it. Panickssery et al. showed at NeurIPS 2024 that LLM evaluators recognize and favor their own generations: the better a model is at identifying its own output, the more generously it scores it. Follow-up work ties this self-preference bias to familiarity: AI judges score text higher when it reads as predictable to them, and nothing reads as more predictable to a model than the text it just wrote itself.
这种直觉是有研究支持的。Panickssery 等人在 NeurIPS 2024 上展示了 LLM 评估者会识别并偏袒自己的生成内容：模型识别自身输出的能力越强，给出的评分就越慷慨。后续研究将这种“自我偏好偏差”与熟悉度联系起来：当 AI 评判者认为文本是可预测的时，给出的分数就越高，而对于模型来说，没有什么比它自己刚刚写出的文本更可预测了。

Different LLM providers train on different data, with different recipes and different feedback loops. No provider manages to eliminate mistakes, but the kind of mistakes a model makes traces back to how it was trained, and that differs per provider. That decorrelation is exactly what you want in your review agent.
不同的 LLM 供应商使用不同的数据、配方和反馈循环进行训练。没有哪家供应商能完全消除错误，但模型犯错的类型可以追溯到其训练方式，而这在不同供应商之间是不同的。这种去相关性正是你所需要的审查智能体特性。

In the year that I’ve used this setup, OpenAI’s Codex has consistently flagged issues in Claude-written PRs that Claude’s own review passed without blinking, from invented filenames to entire code blocks that had nothing to do with the initial request, or that introduced outright security violations.
在我使用这套系统的这一年里，OpenAI 的 Codex 一直能指出 Claude 编写的 PR 中存在的问题，而 Claude 自己的审查却对此视而不见——从虚构的文件名到与最初需求毫无关系的整个代码块，甚至是直接引入的安全漏洞。

One conclusion you cannot take from this is that Codex is the better engineer and you should hand it the keyboard instead of Claude. Codex misfires fairly regularly, too! Just in a different direction than Claude does. The power sits in working multimodel: combining providers within the same workflow, where the models fill each other’s gaps. Another incorrect conclusion is that Codex’s reviews are perfect and can be accepted blindly. Hallucinations happen in reviews too, so treat the review as advisory.
