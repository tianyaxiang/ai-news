The main reason Recursive Language Models feel inaccessible to a lot of the audience is that some of the ideas are actually quite counter-intuitive compared to existing methods (like ReAct, CodeAct, vanilla subagents, etc). The best way to understand RLMs is to first understand where those other methods fail, and realize the single missing piece in agentic harnesses. The idea of passing context around by reference, instead of replicating it.
递归语言模型让许多读者感到难以理解的主要原因在于，与现有方法（如 ReAct、CodeAct、普通子代理等）相比，其中一些概念实际上相当反直觉。理解 RLM 的最佳方式是首先了解其他方法的缺陷，并意识到代理框架中缺失的那关键一环：通过引用传递上下文，而不是复制上下文。

### 1. Of all the complicated experiments I ran…
### 1. 在我运行的所有复杂实验中……

… the most enlightening was this silly experiment where I asked an RLM to: “Generate 50 names of fruits and count the number of R in each, return as a dictionary.”
……最具有启发性的是这个简单的实验：我要求一个 RLM “生成 50 个水果名称，统计每个名称中字母 R 的数量，并以字典形式返回。”

And a more advanced variation (let’s call it Problem 2): “Generate a dictionary of different categories: fruits, countries, animals. For each category, generate 50 names of that and count the number of R in each, return as a nested dictionary.”
以及一个更高级的变体（我们称之为问题 2）：“生成一个包含不同类别（水果、国家、动物）的字典。对于每个类别，生成 50 个名称并统计其中字母 R 的数量，最后返回一个嵌套字典。”

For problem 1, the expected output is something like: `{"strawberry": 3, "berry": 2, ... "grape": 1}`
对于问题 1，预期的输出类似于：`{"strawberry": 3, "berry": 2, ... "grape": 1}`

And for problem 2, it is something like: `{ "fruits": {"strawberry": 3, "berry": 2, ... "grape": 1, ...}, "countries": {"united states of america": 1, "russia": 1, ...}, "animals": {"kangaroo": 1, "tiger": 1", ... "deer": 1, ...} }`
对于问题 2，预期的输出类似于：`{ "fruits": {"strawberry": 3, "berry": 2, ... "grape": 1, ...}, "countries": {"united states of america": 1, "russia": 1, ...}, "animals": {"kangaroo": 1, "tiger": 1", ... "deer": 1, ...} }`

I know it is a silly problem, but the way an RLM solves it is fundamentally different from other architectures like ReAct or CodeAct. And understanding how each method solves this toy problem is all you are going to need to appreciate the beauty of RLMs. Let’s begin!
我知道这是一个简单的问题，但 RLM 解决它的方式与 ReAct 或 CodeAct 等其他架构有着本质区别。理解每种方法如何解决这个小实验，足以让你领略 RLM 的精妙之处。让我们开始吧！

### 2. The Agentic Landscape
### 2. 代理技术全景

#### 2.1 Direct Generation
#### 2.1 直接生成

The first method is just direct generation. The LLM “thinks” about the user’s request and auto-regressively generates a dictionary. No harness, no scaffold, just direct next-token prediction in a loop.
第一种方法是直接生成。LLM “思考”用户的请求，并自回归地生成字典。没有框架，没有脚手架，只是在循环中进行直接的下一个 Token 预测。

Problems with this approach:
这种方法存在的问题：
* LLM has no way to verify if it is mathematically correct.
* LLM 无法验证其数学计算是否正确。
* LLM will likely be wrong because, fundamentally, alphabet counting is not a “next word prediction” problem. Chances of hallucination or errors are extremely high, even if the underlying LLM is intelligent.
* LLM 很可能会出错，因为从根本上讲，字母计数不是一个“下一个词预测”问题。即使底层 LLM 很智能，产生幻觉或错误的概率也极高。

#### 2.2 ReAct (Reasoning and Acting)
#### 2.2 ReAct（推理与行动）

ReAct is a reasoning-and-acting loop where the LLM thinks about the problem first (chain-of-thought) and then generates a tool call. Basically, in the system prompt, we pass a list of “function names”, and instructions about how to call them.
ReAct 是一个推理与行动循环，LLM 首先思考问题（思维链），然后生成工具调用。基本上，我们在系统提示词中传入“函数名称”列表以及如何调用它们的说明。

For example, you could give a simple tool to the LLM that is just: `def count_alphabets_in_word(word: string, alphabet: string) -> int`
例如，你可以给 LLM 提供一个简单的工具：`def count_alphabets_in_word(word: string, alphabet: string) -> int`

Using the above idea, the ReAct agent will be able to do the following:
使用上述思路，ReAct 代理将能够执行以下操作：
* Generate a list of fruit names.
* 生成水果名称列表。
* Use the tool to pass each fruit name and receive the output integer.
* 使用工具传入每个水果名称并接收输出的整数。
* From its output memory, reconstruct the dictionary of which fruit got what count and then return.
* 从输出内存中重构字典，记录哪个水果对应哪个计数，然后返回。

The stack trace of such a transaction would look like this:
此类交互的堆栈跟踪如下所示：

```text
# User
Generate a dictionary with 50 fruits and the number of 'r' in each

# Assistant
<think> 50 fruit names are: strawberry, berry, grape, ... </think>

# Assistant
count_alphabets_in_word("strawberry", "r")

# Tool_Out(executes our function)
3

# Assistant
count_alphabets_in_word("berry", "r")
## Tool call executed!

# Tool_Out(executes our function)
2
.
.
.
# Assistant
<think> I now have everything I need in my message history, let's construct that dictionary </think>
{ "strawberry": 3, "berry": 2, .... }
```

You see what the problems are, right? First, you have to define a function `count_alphabet_in_r` beforehand for this specific use-case. If you don’t define a function, the agent just falls back to the old way (i.e. straight generation of alphabet counts)!
你看出问题所在了吧？首先，你必须针对这个特定用例预先定义一个函数 `count_alphabet_in_r`。如果你不定义函数，代理就会退回到旧方法（即直接生成字母计数）！

This guarantees that the LLM has some hint about what the output is, but the LLM still has to generate the tokens one at a time from its message history. The LLM still has to remember the counts of each word, and reproduce it verbatim from memory. Transmission errors can still happen during this stage.
这保证了 LLM 对输出内容有一些提示，但 LLM 仍然必须从消息历史中逐个生成 Token。LLM 仍然必须记住每个单词的计数，并从内存中逐字重现。在此阶段仍可能发生传输错误。

The problem compounds when you extend it to the multi-category setting of Problem 2. LLM has to repeat a long trace of function calls and remember what happened at each turn, and generate the answers token by token.
当你将其扩展到问题 2 的多类别设置时，问题会变得更加复杂。LLM 必须重复一长串函数调用，记住每一步发生了什么，并逐个 Token 地生成答案。

As an dev, ReAct is great if you developing narrow applications, where you want the agent to have access to specific tools (web search, document search, calculator, terminal access, file edit, diff applier etc), but you will rarely develop a general agent and optimize for niche skills like these. Basically, for general agents, only those universal tools are good. You won’t write tools like `count_alphabet_in_r` unless you specifically know your users will want it. What if the LLM could create its own tools?
作为开发者，如果你在开发狭窄的应用场景，希望代理能够访问特定工具（网页搜索、文档搜索、计算器、终端访问、文件编辑、差异应用等），ReAct 非常棒。但你很少会为了这些小众技能去开发一个通用代理。基本上，对于通用代理，只有那些普适性工具才有用。除非你明确知道用户需要，否则你不会编写像 `count_alphabet_in_r` 这样的工具。如果 LLM 能自己创建工具呢？

#### 2.3 CodeAct
#### 2.3 CodeAct

CodeAct 允许 LLM 编写并执行代码。这意味着你（人类）不再需要编写精确的工具。你只需赋予 LLM 编写任何 Python 代码并在沙盒终端环境中执行的能力，读取结果并生成输出即可。过程大致如下：
