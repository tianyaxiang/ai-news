# Behind the Scenes of Distributed Training and Why Your GPU Wiring Matters as Much as Your Strategy
# 分布式训练幕后：为什么 GPU 的物理连接与训练策略同样重要

A measured look at distributed training, from DDP and FSDP to the ZeRO stages in between, and why the wiring between your GPUs matters as much as the strategy you choose.
本文深入探讨了分布式训练，涵盖了从 DDP、FSDP 到中间的 ZeRO 阶段，并解释了为什么 GPU 之间的物理连接与你选择的策略同样重要。

Training a model on one GPU is straightforward. You load the weights, load the data, and wait for it to finish. For most models, that’s fine, and the only real cost is time. When the wait gets too long, the usual fix is to add GPUs. Each one trains on a different slice of the data in parallel, and the work finishes faster. Nothing about the model state changes; you’ve just thrown more hands at it.
在单张 GPU 上训练模型非常简单。你加载权重，加载数据，然后等待训练完成。对于大多数模型来说，这没问题，唯一的实际成本就是时间。当等待时间过长时，通常的解决方法是增加 GPU。每张 GPU 并行处理数据的一个切片，工作完成得更快。模型状态本身没有任何改变；你只是增加了人手。

That covers the common case, where the problem is time. Large models add a second problem that more hands can’t solve: space. Over the past year, a major part of my work has involved training and fine-tuning large foundation models across different domains, including single-cell models and vision-and-language transformer models. Once you get past a few billion parameters, the model, its gradients, and its optimizer states no longer fit on a single GPU. You can’t just replicate the work across GPUs because a copy doesn’t fit on any of them. You have to split the model itself.
这涵盖了以时间为瓶颈的常见情况。大型模型带来了第二个仅靠增加人手无法解决的问题：空间（显存）。在过去的一年里，我的大部分工作涉及在不同领域训练和微调大型基础模型，包括单细胞模型和视觉-语言 Transformer 模型。一旦参数量超过几十亿，模型、梯度和优化器状态就无法再放入单张 GPU 中。你不能简单地在 GPU 之间复制工作，因为任何一张卡都装不下完整副本。你必须拆分模型本身。

These two problems map onto two strategies. DDP (Distributed Data Parallel) attacks time. Every GPU keeps a full copy of the model and trains on a different slice of the batch. FSDP (Fully Sharded Data Parallel) attacks space. The model is split into pieces so that no GPU has to hold the entire model.
这两个问题对应两种策略。DDP（分布式数据并行）旨在解决时间问题。每张 GPU 保留模型的完整副本，并在不同的批次切片上进行训练。FSDP（完全分片数据并行）旨在解决空间问题。模型被拆分成多个部分，因此没有任何一张 GPU 需要持有整个模型。

There aren’t really just two options, though. These are just the two ends of one dial. In between these sit the ZeRO stages from DeepSpeed (Microsoft’s training library), which let you slide from one end to the other, step by step, trading a little memory for a little communication at each stop rather than jumping straight from “everybody holds everything” to “nobody does.”
不过，实际上并不只有这两种选择。它们只是一个调节旋钮的两端。在这两者之间，存在着来自 DeepSpeed（微软的训练库）的 ZeRO 阶段，它们让你能够一步步地在两端之间滑动，在每个节点上用少量的通信换取少量的内存，而不是直接从“每个人持有所有东西”跳跃到“没人持有所有东西”。

Where to set that dial and what it actually controls is covered in the first part of the article. But there’s a second thing that decides how well any of these strategies work. Each one of them works by moving data between GPUs. And how fast that data moves depends on the physical wires connecting them. This was something I didn’t think about until the same code, on the same kind of GPUs, ran several times slower on one node than on another. So in the second half, we’ll discuss the hardware underneath it, and how the wiring should also be something you consider.
如何设置这个旋钮以及它实际控制的内容将在本文的第一部分介绍。但还有第二个因素决定了这些策略的运行效果。它们每一个都需要在 GPU 之间移动数据。而数据移动的速度取决于连接它们的物理线路。在发现同一段代码、在同类 GPU 上，在不同节点运行速度差异巨大之前，我从未考虑过这一点。因此，在后半部分，我们将讨论底层的硬件，以及为什么布线也是你需要考虑的因素。

### Part 1: The Software: What Each GPU Holds
### 第一部分：软件——每张 GPU 持有什么

Every distributed training strategy is really an answer to one question: what does each GPU keep its own copy of? For that to make sense, let’s look at an example. Take Mistral-7B and fine-tune it with Adam in BF16 mixed precision. The parameters take 14 GB: 7 billion of them, two bytes each. The gradients take another 14 GB. Then we have the optimizer states, which are the most expensive part. Adam keeps two running averages for every parameter, momentum and variance, both stored in FP32, which works out to roughly 58 GB. Add it up, and we reach 87 GB before the model has seen a single batch or a single activation has been computed.
每种分布式训练策略实际上都是在回答一个问题：每张 GPU 保留了什么副本？为了理解这一点，我们来看一个例子。以 Mistral-7B 为例，使用 BF16 混合精度和 Adam 优化器进行微调。参数占用 14 GB：70 亿个参数，每个占用 2 字节。梯度占用另外 14 GB。然后是优化器状态，这是最昂贵的部分。Adam 为每个参数保留两个运行平均值（动量和方差），均以 FP32 存储，总计约 58 GB。加起来，在模型看到单个批次或计算出单个激活值之前，我们就已经占用了 87 GB。

For an A100 GPU that has 80 GB of VRAM, the model doesn’t fit. So we distribute. And the question, as mentioned at the beginning of this section, is: what does each GPU hold?
对于拥有 80 GB 显存的 A100 GPU 来说，模型装不下。所以我们进行分布式处理。正如本节开头提到的，问题在于：每张 GPU 持有什么？

### Distributed Data Parallel (DDP): Everyone Holds Everything
### 分布式数据并行 (DDP)：每个人持有所有东西

The simple answer is that every GPU holds all of it. This is DDP, and it’s the one most people first encounter (which was the case for me as well). Each GPU keeps a complete copy of the model: all the parameters, all the gradients, all the optimizer states. What gets divided is the data. Each GPU takes a different slice of the batch, runs its own forward and backward pass, and computes gradients from its own slice.
简单的回答是，每张 GPU 都持有所有内容。这就是 DDP，也是大多数人最先接触到的（我也是如此）。每张 GPU 都保留模型的完整副本：所有参数、所有梯度、所有优化器状态。被划分的是数据。每张 GPU 获取批次的不同切片，运行各自的前向和反向传播，并根据自己的切片计算梯度。

The catch is that those gradients are all different because each GPU saw different data. If every GPU just stepped its optimizer now, the copies would drift apart, and you’d no longer have one model, you’d have four. So before stepping, the GPUs average their gradients. Every GPU sends its gradients out, receives everyone else’s, and they all end up with the same averaged gradient. They then take an identical optimizer step and the copies stay in sync. That averaging is the one piece of communication DDP needs, and it happens once per step. It is called all-reduce, and we’ll come back to it later when we start counting what each strategy costs the wire.
问题在于，这些梯度各不相同，因为每张 GPU 看到的数据不同。如果每张 GPU 现在直接执行优化器步骤，副本就会产生偏差，你将不再拥有一个模型，而是四个。因此，在执行步骤之前，GPU 会对梯度进行平均。每张 GPU 发出自己的梯度，接收其他所有人的梯度，最终它们都得到相同的平均梯度。然后它们执行相同的优化器步骤，副本保持同步。这种平均化是 DDP 所需的唯一通信，每步发生一次。它被称为 all-reduce，当我们开始计算每种策略对线路的消耗时，我们会回到这一点。

DDP is fast and simple precisely because it communicates so little. One all-reduce per step, and PyTorch overlaps even that with the backward pass, so on good hardware, you barely notice it. But its limitation is built into the design. Every GPU holds everything, which means adding GPUs does nothing for the memory problem. If 87 GB doesn’t fit on one A100, it doesn’t fit on four either, because each of the four is still holding the full 87 GB. Adding GPUs to DDP buys you throughput, but it doesn’t reduce the memory each GPU has to hold.
DDP 之所以快速且简单，正是因为它通信量极少。每步一次 all-reduce，而且 PyTorch 甚至将其与反向传播重叠，因此在好的硬件上，你几乎感觉不到它的存在。但它的局限性植根于设计之中。每张 GPU 持有所有内容，这意味着增加 GPU 对内存问题毫无帮助。如果 87 GB 装不进一张 A100，那也装不进四张，因为四张中的每一张仍然持有完整的 87 GB。在 DDP 中增加 GPU 可以提高吞吐量，但不会减少每张 GPU 必须持有的内存。

### Fully Sharded Data Parallel (FSDP): Nobody Holds Everything
### 完全分片数据并行 (FSDP)：没人持有所有东西

If the problem is that every GPU holds everything, the obvious fix is to make sure no GPU does. This is FSDP. Instead of each GPU keeping a full copy, the model is split into pieces, and each GPU owns just one piece of the parameters, one piece of the gradients, and one piece of the optimizer states.
如果问题在于每张 GPU 都持有所有内容，那么显而易见的解决方法是确保没有任何一张 GPU 持有所有内容。这就是 FSDP。每张 GPU 不再保留完整副本，而是将模型拆分成多个部分，每张 GPU 只拥有参数、梯度和优化器状态的一部分。
