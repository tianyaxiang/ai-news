我们以 NIST SP 800-53 Rev. 5（信息系统和组织的安全与隐私控制，美国政府工作，在美国属于公有领域）和 NIST 网络安全框架 2.0 为例进行演示。可运行的代码路径调用受 OpenAI 使用条款约束的 OpenAI 服务。

### 1. 问题：目录太长，无法直接交付
NIST SP 800-53 长达 492 页。它定义了美国联邦系统必须满足的安全控制措施，逐项控制，共分为二十个系列。用户提出了一个简单的问题：“账户管理控制要求什么？”答案位于第 46 到 50 页的 AC-2 控制项中，共五页。

朴素的 RAG 方法会对每一页进行嵌入（Embedding），并取与问题最相似的 top-k 个结果。在这里它会以一种特定的方式失败：单词“账户”、“管理”、“控制”和“访问”出现在数百页中，因为整份文档都是关于控制措施的。Top-k 返回的结果会将 AC-2 与 AC-3、AC-17、一些审计控制项以及术语表混在一起，生成模型必须猜测哪一个是用户想要的。你付出了双倍的代价：既嵌入了 492 页文档，答案却依然模糊不清。

人类会选择打开目录。这就是本文存在的必要性：目录本身就有 358 条目。你不会把 358 行目录全部扔给同事，就像你不会把 492 页文档全部扔给他们一样。专家会先扫描章节列表（11 个标题），选择“控制措施”，打开二十个系列，选择“访问控制”，再打开该部分找到具体的控制项，最终定位到 AC-2。先看顶层，再向下深入，一次只做一个小决定。

这就是本文构建的循环，也是该系列的核心论点：放大专家的能力。像专家一样行事。不要一次性把整个文档或整个目录扔给模型。

### 2. 目录：三个层级深度
这是文档实际的目录页，旁边是解析器从其原生大纲中读取的 `toc_df`。

*左侧：文档自身的目录页；右侧：解析器的 toc_df，第三章下的二十个控制系列，每个都是你可以跳转的页面范围 – 图片由作者提供*

打印出来的页面只停留在系列级别。解析器则更深入。第 5B 篇将 PDF 的原生大纲转换为 `toc_df`，每个标题一行，对于这份文档，它在三个层级上共有 358 行，从 11 个顶层章节一直到 316 个独立控制项（AC-1, AC-2 等），每个都有其页面范围。二十个控制系列位于中间。检索过程就是遍历这棵树。当文档没有原生大纲可遍历时，第 5septies 篇会从页面中重建一个；这里的大纲很清晰，所以我们直接使用它。

### 3. 循环：一次一个层级
检索过程自顶向下遍历树。它只向 LLM 提供当前层级，首先是 11 个章节标题，每个标题作为紧凑的一行：标题、页面范围，以及在标题模糊时提供简短的关键词统计。模型选择回答问题的分支。如果该分支很宽且有更细分的子项，检索过程会打开它，其子项成为下一层级，步骤重复。它会在叶子节点或足够小到可以通读的章节处停止。

*检索内部的有界循环：选择、打开、重复，直到到达叶子节点或小章节；LLM 在每个层级判断是否需要深入 – 图片由作者提供*

这种结构带来了两个好处。长目录永远不会完整地进入提示词：模型一次读取 11 行，然后是 20 行，再是 20 多行，绝不会一次性读取 358 行。而且，这种下降实际上是可选的。在短文档中，顶层没有子项可打开，因此循环运行一次，其行为与扁平路由完全相同。模型在每个层级都会判断是否有理由深入。

### 4. 路由步骤（代码实现）
该步骤是一个函数 `reason_on_toc`，每个层级调用一次。它将层级作为文本读取，而不是表格，因此 20 个条目的层级就是 20 行短文本，而不是一个随关键词增加列数的网格。

```python
# 自顶向下：一次只输入一个层级，绝不输入完整的 358 行目录。
# 每个条目是一行紧凑文本：标题 + 页面范围 (+ 关键词命中数)。
level = toc_df[toc_df.level == toc_df.level.min()] # 11 个章节标题

while True:
    pick = reason_on_toc(question, level, client=client) # 一次 LLM 调用
    section = level[level.id.isin(pick.section_ids)]
    kids = immediate_children(toc_df, section)
    
    if kids.empty or section.n_pages <= SMALL: # 到达叶子节点，或足够小
        break
    level = kids # 打开它，继续深入
```
