# How to Sync Bank Transactions with Open Banking APIs (2026 Guide)
# 如何通过开放银行 API 同步银行交易（2026 年指南）

Most open banking tutorials stop at "make a GET request." In production, you need consent flows, token refresh, pagination across thousands of transactions, deduplication, and rate-limit handling. This guide walks through the full sync pipeline — what actually breaks at 3am when your cron job fires.
大多数开放银行教程只讲到“发送一个 GET 请求”就结束了。但在生产环境中，你需要处理授权流程、令牌刷新、跨数千条交易的分页、去重以及速率限制。本指南将带你了解完整的同步流水线——即当你的定时任务在凌晨 3 点运行时，哪些环节真正容易出错。

### The sync lifecycle
### 同步生命周期

A real bank-data sync has four phases:
真实的银行数据同步包含四个阶段：

1. **Onboard:** User consents to read-only access → you get a consent token.
   **接入：** 用户同意只读访问权限 → 你获得一个授权令牌。
2. **Initial fetch:** Pull all historical transactions (can be thousands).
   **初始抓取：** 拉取所有历史交易（可能多达数千条）。
3. **Incremental sync:** Poll for new transactions on a schedule.
   **增量同步：** 按计划轮询新交易。
4. **Token refresh:** Consent expires (90 days default under PSD2) → re-authorize.
   **令牌刷新：** 授权过期（PSD2 标准下默认为 90 天）→ 重新授权。

### Step 1: The consent flow
### 第一步：授权流程

All PSD2 Account Information Service (AIS) providers use a similar OAuth-style redirect:
所有 PSD2 账户信息服务 (AIS) 提供商都使用类似的 OAuth 风格重定向：

```javascript
// 1. Redirect user to the bank's consent page
const authUrl = provider.getAuthUrl({
  redirect_uri: 'https://yourapp.com/callback',
  scope: 'accounts transactions',
  access_valid_for_days: 90
});
// User clicks → bank asks "Allow YourApp to read your transactions?"
// User approves → redirects back with a code
```

The consent token you receive is time-limited (PSD2 mandates max 180 days, most default to 90). Store the expiry date — you'll need it.
你收到的授权令牌是有时效的（PSD2 规定最长 180 天，大多数默认为 90 天）。请务必存储过期日期——你之后会用到它。

### Step 2: Fetch accounts
### 第二步：获取账户

```javascript
async function getAccounts(consentToken) {
  const res = await fetch(`${API_BASE}/accounts`, {
    headers: { 'Authorization': `Bearer ${consentToken}` }
  });
  const data = await res.json();
  return data.accounts; // [{ id, iban, name, currency, balance }]
}
```

### Step 3: Sync transactions (the hard part)
### 第三步：同步交易（难点）

This is where tutorials end and production begins:
这是教程结束、生产环境开始的地方：

```javascript
async function syncTransactions(accountId, consentToken, lastSyncCursor) {
  let allTransactions = [];
  let cursor = lastSyncCursor;
  let hasMore = true;

  while (hasMore) {
    const url = `${API_BASE}/accounts/${accountId}/transactions?cursor=${cursor}`;
    const res = await fetch(url, {
      headers: { 'Authorization': `Bearer ${consentToken}` }
    });

    if (res.status === 429) { // Rate limited — back off
      const retryAfter = res.headers.get('Retry-After') || 60;
      await sleep(retryAfter * 1000);
      continue;
    }

    const page = await res.json();
    allTransactions.push(...page.transactions);
    cursor = page.next_cursor;
    hasMore = page.has_more;
    
    // Respect rate limits between pages
    await sleep(200);
  }
  return { transactions: allTransactions, nextCursor: cursor };
}
```

**Deduplication:** Banks will return overlapping transactions between syncs. The cursor handles most of this, but edge cases exist:
**去重：** 银行在不同同步周期之间会返回重叠的交易。游标（Cursor）可以处理大部分情况，但仍存在边界情况：
1. A pending transaction settles (bookingDate changes, transactionId may change).
   待处理交易结算（入账日期 bookingDate 变更，transactionId 可能改变）。
2. A transaction is reversed and re-booked.
   交易被撤销并重新入账。
3. The bank back-dates a transaction.
   银行对交易进行追溯日期处理。

Store a composite key: `{bankTransactionId, bookingDate, amount}`. If the bank provides a `transactionId` or `entryReference`, prefer that. If not, hash the three fields above.
请存储一个复合键：`{bankTransactionId, bookingDate, amount}`。如果银行提供了 `transactionId` 或 `entryReference`，优先使用它们。如果没有，则对上述三个字段进行哈希处理。

### Step 4: Token refresh
### 第四步：令牌刷新

PSD2 consent expires. You cannot silently refresh it — the user must re-authorize. Design for this:
PSD2 授权会过期。你无法静默刷新它——用户必须重新授权。请为此做好设计：

```javascript
async function checkConsent(consentToken) {
  const res = await fetch(`${API_BASE}/accounts`, {
    headers: { 'Authorization': `Bearer ${consentToken}` }
  });
  if (res.status === 401 || res.status === 403) {
    // Consent expired or revoked
    notifyUser('Your bank connection needs reauthorization');
    return { valid: false, needsReauth: true };
  }
  return { valid: true };
}
```

Send the notification early (7 days before expiry). Users who see "reconnect your bank" at the moment of need will do it. Users who see it when they're not using your app will ignore it.
请提前发送通知（过期前 7 天）。当用户在需要使用时看到“重新连接银行”的提示，他们会去操作；如果他们在不使用应用时看到，他们会忽略它。

### Production checklist
### 生产环境检查清单

*   **Rate limits:** Most providers allow 4-15 requests/minute per consent. Batch where possible.
    **速率限制：** 大多数服务商允许每个授权每分钟 4-15 次请求。尽可能进行批量处理。
*   **Idempotency:** Use the bank's `transactionId` as your dedup key, not your own generated ID.
    **幂等性：** 使用银行的 `transactionId` 作为去重键，而不是你自己生成的 ID。
*   **Pagination:** Some banks return 100/page, others 500. Always respect the `has_more` flag.
    **分页：** 有些银行每页返回 100 条，有些返回 500 条。务必遵循 `has_more` 标志。
*   **Webhooks:** Some providers offer transaction webhooks (push instead of poll). Use them if available — they reduce API calls by 90%+.
    **Webhooks：** 一些服务商提供交易 Webhook（推送而非轮询）。如果可用，请务必使用——它们能减少 90% 以上的 API 调用。
*   **Error handling:** Banks go down. The PSD2 network is 6,000+ banks — some will be unavailable. Retry with exponential backoff, but don't block the entire sync on one failing account.
    **错误处理：** 银行系统会宕机。PSD2 网络涵盖 6,000 多家银行，总会有不可用的情况。请使用指数退避算法进行重试，但不要因为一个账户同步失败而阻塞整个同步任务。
*   **Data freshness:** Transactions can appear 1-3 days after the actual payment (pending → booked). Don't display "real-time" if you're polling daily.
    **数据新鲜度：** 交易可能在实际支付后 1-3 天才显示（从待处理变为已入账）。如果你是每日轮询，请不要显示“实时”字样。

### What breaks in production
### 生产环境中容易出现的问题

The three things that caused the most production incidents in my experience:
根据我的经验，以下三点导致了最多的生产事故：

1. **Consent expiry cascade:** Multiple users' consents expire on the same day (if you onboarded them in a batch). Your reauth notification queue overflows. Stagger onboarding or add jitter to expiry notifications.
   **授权过期雪崩：** 如果你是批量接入用户，他们的授权会在同一天过期。你的重新授权通知队列会溢出。请错开接入时间或在过期通知中加入随机延迟（Jitter）。
2. **Transaction ID instability:** Some banks change the `transactionId` when a pending transaction settles. Your dedup fails and you get duplicates. Always store both pending and booked versions, and reconcile by matching on amount + date + counterparty.
   **交易 ID 不稳定：** 一些银行在待处理交易结算时会更改 `transactionId`。这会导致去重失败并产生重复数据。请务必同时存储待处理和已入账版本，并通过匹配金额 + 日期 + 交易对手来进行对账。
3. **Rate limit headers:** Some providers return `X-RateLimit-Remaining` in headers, not the response body. If you ignore headers, you'll get 429s that seem random.
   **速率限制响应头：** 一些服务商在响应头（Headers）中返回 `X-RateLimit-Remaining`，而不是在响应体中。如果你忽略响应头，就会遇到看似随机的 429 错误。

The open banking APIs are powerful, but the production details — consent lifecycle, deduplication, rate limiting — are where most integrations stall.
开放银行 API 功能强大，但生产环境中的细节——授权生命周期、去重、速率限制——才是大多数集成项目停滞不前的地方。
